    Carl,

We're still not receiving your RSV records, and I found out why: Because
the timestamp in those records are screwed up, and not in the expected
format.

For example: https://gpn-husker.unl.edu:8443/rsv/

gram-authentication-probe
<https://gpn-husker.unl.edu:8443/rsv/gpn-husker.unl.edu/index.html#gram-authentication-probe.org.osg.globus.gram-authentication>;
org.osg.globus.gram-authentication
<https://gpn-husker.unl.edu:8443/rsv/gpn-husker.unl.edu/index.html#gram-authentication-probe.org.osg.globus.gram-authentication>;
1235742481 	OK

I remote-debugged, and found the following lines in your RSV perl module
line 1342-1347:

if ($type eq "Local") {  ## Double escaping required to pass "\ "
$system_date_cmd_out = `$h_tmp{'systemDateCmd'} -u +%s`;
}
else { $system_date_cmd_out = `$h_tmp{'systemDateCmd'} -u +%s`; }
chomp ($system_date_cmd_out);

This is not what we have in our release perl module, and it does not
correspond to the hack I recommended to you in this ticket either. What
I'd recommended was:

Search for sub-routine "sub Get_Timestamp { " in
$V_L/osg-rsv/bin/probes/OSG_RSV_Probe_Base.pm
Replace:
chomp ($system_date_cmd_out);|
with
$system_date_cmd_out =~ s/\s*$//s; ## Remove trailing spaces/newlines

You appear to have changed the parameters passed to the data command

    itself, which is what is causing the problem now. Please restore the
file to its original state, and only modify the line shown above, and
you should be all set.

For completeness, this is what I expected to see in your perl module:

sub Get_Timestamp {
my %h_tmp        = %{$_[0]};
my $system_date_cmd_out    = undef;

if ($type eq "LOCAL") {  ## Double escaping required to pass "\ "
$system_date_cmd_out = `$h_tmp{'systemDateCmd'} +%F\\ %T\\ %Z`;
}
else { $system_date_cmd_out = `$h_tmp{'systemDateCmd'} -u +%FT%TZ`; }
$system_date_cmd_out =~ s/\s*$//s; ## Remove trailing spaces/newlines
return \$system_date_cmd_out;
}
