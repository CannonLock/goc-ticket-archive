    Nov 10, 2011 03:04 PM UTC by Thomas Lee 
    Release.gpg is not a certificate or key; it is a GPG signature.  It is created by Scott's "install-latest-cache-files-new.sh" script, located in /home/steige on software1/software2/software-itb.  The script uses a "gpg -abs" command to create "Release.gpg," which is a detached ASCII-armored signature of the file "Release" in the same directory.

Here is the text of the "Release" file (located in /usr/local/software-cache/debian-repo/dists/stable) from software-itb:

Origin: OSG-ITB
Suite: stable
Date: Tue Nov  1 12:56:18 UTC 2011
MD5Sum:
8d18d0506599571f3bd887fb7aff0e30 447 main/binary-amd64/Packages
1d329a98087f3bca7b73d38d338a30a7 324 main/binary-amd64/Packages.gz
8d18d0506599571f3bd887fb7aff0e30 447 main/binary-i386/Packages
1d329a98087f3bca7b73d38d338a30a7 324 main/binary-i386/Packages.gz

As you can see, it has MD5 digests of each release file; those files might be different between ITB and production.  There is also a timestamp.  However, even if the files and timestamp are the same, the "Origin: OSG-ITB" line would vary between the ITB and production instances (where it says "Origin: OSG"), again causing the signature in Release.gpg to differ.

Therefore I assert that it is normal for Release.gpg to differ between ITB and production instances of the service -- in fact, it would be nigh impossible for it to be the same between software-itb and either of the production instances.
