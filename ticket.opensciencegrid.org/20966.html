    Hello,

I hope that you are right Augustine, and that you have found the
correction. But just in case, the process is still a problem, I will
share my findings, actions, and theories from looking at gratiaweb-itb.

I found that the:
1) Cron jobs for the static graphs in "gratia_data.cron" were running at
the same times as the one on my machine
2) Duplicate process began at the same time as the second cron job; the
one that runs a series of vo specific graphs
3) Number of static graphs was 305
4) Run times for graph generation was 15-26 seconds
5) Error messages for the stack trace, provided by Thomas Lee, indicated
that it initiated from a __bootstrap, threading failure, after running
through the apache (httpd) server

Corrective Action taken:
1) Calculated that the total estimated static graph generation should be
somewhere between 76-152 minutes
2) Corrective action was taken, in that I modified the number, and times
of those cron jobs. The goal was to offset the time of the cron job
initiation from all the other cron jobs in the system that likely
occurring at exactly the same time. This causes each query that is run
to encounter excessive competition for the same data by all the other
servers also running that cron job.

My current theory for duplicate process is that is is generated by the
httpd server through the cherrypy query handling module. It this is true
there will be little we can do control the generation of the process,
because it will occur as the system becomes overloaded and sluggish.

Until we hear further, here is hoping that the problem has already been
solved.


    Best Regards,
Brad Hurst
