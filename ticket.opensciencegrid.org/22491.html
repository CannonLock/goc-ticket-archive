    Mar 11, 2015 06:04 PM UTC 
    We are still working on a longer term solution.  As of now we have no automated way to support T3 requests for the creation of relevant meshes.  The perfSONAR developers are considering the creation of future discovery tools that would be network topology aware.  This could be used to automate meshes for end-sites that are otherwise "orphaned".   However it won't be included until at least perfSONAR v3.6 (~Jan 2016).

For the nearer term, two things are being worked upon for this type of use-case:

1) The creation of a standalone mesh-config GUI, installable by end-users and campuses to create and define their own meshes.  This has been delayed to mid-Summer though (we had hoped to have it by April originally).
2) Adding existing T3's to their own (temporary) mesh to provide some default testing.   This could be implemented in the next two weeks after we iron out a few details.

I will respond to this ticket again when the "T3" mesh is ready.

Sorry for the slowness on this.

Sorry for the delay in responding.  Just back from CERN yesterday.

Based on our discussions the way forward for Tier-3 sites should be as follows:

1) Tier-3 sites should remove any older mesh-configuration from their  /opt/perfsonar_ps/mesh_config/etc/agent_configuration.conf file
2) Tier-3 sites should use the new "auto-mesh" URL following the instructions in https://twiki.opensciencegrid.org/bin/view/Documentation/InstallUpdatePS
In this case the configuration line should be: "configuration_url https://myosg.grid.iu.edu/pfmesh/mine/hostname/psonar-owamp.brazos.tamu.edu";

As of right now we don't have an easy way to determine which tests a specific Tier-3 should participate in.   For cases where the host is not part of ANY mesh the auto-mesh will return valid JSON BUT the current mesh agent in the perfSONAR toolkit may treat this as an error and generate a warning email.   We are working with Aaron Brown (cc'ed here) on how best to fix this behavior.

This setup will allow the hosts using the auto-mesh to benefit from a future configuration that will identify suitable test points for Tier-3 sites (for example testing to the "nearest" backbone instance and/or campus border instance).

Trey, please let us know if you hit any issues in doing this or if you have any comments, suggestions or questions.  Thanks,
