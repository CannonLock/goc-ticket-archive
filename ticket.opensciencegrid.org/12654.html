    Hi,

I looked at a two of the failed jobs.  These ran on acas1025 and
acas1614.  The Ganglia memory plots for the past day for these
machines are attached: both hosts have had plenty of free RAM
at all times.

The OS is not killing these processes: they appear to be receiving
NULL from malloc(), indicating this system was unable allocate
more memory to the processes.  The logfile you referred to in
this ticket indicates that the virtual memory allocation for this
process reached ~3.9 GB:
Py:PerfMonSvc        INFO <vmem>:   (   3928.965 +/-      0.000 ) MB

Therefore, assuming this is 32-bit code, it appears the software
is hitting the 4 GB maximum VMEM size for a 32-bit process.  This
is not an OS issue, but simply a physical limitation of a 32-bit
process space.  The only way around it would be run 64-bit code.

Chris
