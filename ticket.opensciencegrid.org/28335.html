    Jun 6, 2016 01:59 PM UTC by Tim Cartwright 
    Matt, I am the OSG Software manager and as I happen to be on ticket duty this week, I looked through your ticket a bit. First of all, congratulations on figuring out a great many things over the past few months! I know that it is not always very easy, and it seems like you have nonetheless accomplished a lot.

Looking through the more recent parts of this ticket, I feel that you have asked some good, higher-level questions, but mostly have received detailed technical answers. Let me try to clarify a few things....

By design, the OSG provides little to no documentation for setting up specific batch systems like HTCondor, PBS, and SLURM. The OSG works with many such systems and each one is complex and has many configuration options. We simply lack the people-time to document how to set them all up. And really, there are so many site-specific requirements and approaches that we simply could not cover them all anyway without just referring back to the batch system’s own documentation.

Further, with OSG 3.3 we dropped the “full” client from our distribution. That set of packages was meant for a use case that is quite rare now: setting up a direct submission node into the OSG. These days, most end users are using things like OSG Connect or VO-specific submission points (“VO front-ends” in our jargon) that have different set-up needs. As it happens, it sounds like you are trying to set up a direct submission node, so you fall into this use case for which we now provide less direct packaging support. We are still happy to provide support this way, though, via direct interaction through tickets. And note that we still support the concept of a “Worker Node Client” (osg-wn-client), which is the OSG-specific software that is *added to* batch system software running on each worker node; but of course that is a different use case entirely.

As for your most recent troubles, you are running into more jargon… An HTCondor “schedd” (short for “SCHEDuler Daemon”) is the HTCondor service that maintains your queue of jobs that running or waiting to run. So even if you are submitting jobs locally that are intended to run remotely, there must be a local place to hold and track those jobs. That place is the local queue, and we call it a schedd. That’s why you need one.

I hope that helps! Generally speaking, if you are asking questions that do not get answered, especially higher-level or conceptual questions — shout louder! Both the GOC and Software teams should be comfortable handling such questions or escalating them to others who can answer them. So hold us accountable for getting you what you need!

— Tim

    Apr 28, 2016 05:02 PM UTC by Terrence Martin 
    Absolutely, condor is used for submission to OSG sites in the US and all
US CMS sites use it.

Your submit file should look something like.

universe = grid
output = myoutput.$(Cluster).$(Process)
error = myerror.$(Cluster).$(Process)
log = mylog.$(Cluster).$(Process)
executable = myTestScript.sh
arguments = 700
grid_resource = condor osg-gw-6.t2.ucsd.edu osg-gw-6.t2.ucsd.edu:9619
request_cpus = 1

That is just one example, but will get you started. You will want to
tune your job script to tidy up after itself, condor tends to want to
send everything back at the end, and that can be too much for the submit
side. In CMS the jobs send their data back via Gridftp to a storage site
and delete the rest before they leave for example.

Terrence
