    Jun 14, 2006 09:41 PM UTC by Footprints Remedy 
    Title=runaway globus-job-manager process.

Status=Service Desk

This problem is for the Fermilab LQCD Cluster. The cluster head-node NFS
mounts various RAID disks physically attached to the head-node, a public
dCache pnfs server and a local dCache pnfs server.

We noticed that the following program

/usr/local/vdt/globus/libexec/globus-job-manager

executes "df" to gather information about disk space on the head-node. In
the instance of any of the above mentioned NFS mount points failing, the
globus-job-manager hangs, making matter worse, consumes a lot of memory and
never recovers even after the NFS mount points have recovered.

In a recent instance, the public dCache system was down for a few hours on
Jun 7 AM causing the globus-job-manager to use up all the main memory (8G)
on the head-node as noted below:

The head-node had to be rebooted to recover the system. How can we avoid
this situation in the future?

we can submit a ticket to the OSG goc

I have spoken to the gridcat administrators and they confirm that the gridcat process is
running the site_verify.pl script four times a day against the various sites,
and this does contain a "df" command which would
cause the problem you are
seeing.

I am going to attempt to
forward this ticket to the
Grid Operations Center and
ask their people to
modify that script accordingly so it won't
do a blind df.
