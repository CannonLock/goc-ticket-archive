    Jan 11, 2017 03:51 PM UTC by Dave Dykstra 
    That would be ideal.  I would set it up that way if I could, but the problem is that for security reasons, it cannot be done on the new VM.  There is a secure per-repository key, and it is stored only in /etc.  It would not be secure to copy that to the persistent storage, so the new machine has to generate a new key and write out the signed certificate to the storage.  The recover_oasis_rollback command enables the old VM to overwrite that with a certificate signed by another new key.  (The old VM can't even use its old key even though it is in its /etc because of a chicken-and-egg problem: cvmfs won't mount the unrecognized certificate so it can't publish a new one.)

I also can't have the boot process do the equivalent of recover_oasis_rollback on every boot, because it requires the OSG masterkey to sign the new repository key.
