    Mar 25, 2015 08:35 PM UTC by Carl Edquist 
    Neha and I got to discuss this a bit today...

So, there is a dn in the usr table and then a subject_string in the certificate table.  It seems to work fine using to use the command line tools to do a DN rename like this:

voms-admin --vo VONAME --nousercert add-certificate OLD_DN OLD_CA NEW_DN NEW_CA
voms-admin --vo VONAME --nousercert remove-certificate OLD_DN OLD_CA

After this you can do voms-admin commands against NEW_DN and NEW_CA.

The confusing part is that there is no way to rename the dn in the usr table -- but this seems to be irrelevant.  As far as I can tell the dn in the usr table isn't used for anything -- the voms-admin commands work against the DN/CA pairs in the certificate table.  I can send an email to the voms developer (Andrea Ceccanti) to confirm this, but at least in my tests the voms-admin commands work fine regardless of the dn in the usr table.

Also, on the web interface, the DN displayed for the user 'more info' is from the certificate table, not the dn in the usr table.

So my recommendation is just to ignore the dn in the usr table -- pretend it doesn't exist!  :)

If you really want, you can probably update the dn in the usr table directly, but it's probably better to ignore it since it doesn't mean anything.  Even if you want to use it for doing sql queries, you should probably be querying the certificate table instead, even though it's more annoying to write...

So instead of:

select * from usr where dn like '%neha%';

you can do something like:

select * from usr where userid in (select usr_id from certificate where subject_string like '%neha%');

which apparently reflects the actual dn strings that voms cares about...

    I think you can use the "add-certificate" and "remove-certificate" subcommands with the voms-admin command line tool.

You can run:
voms-admin --help-command add-certificate
voms-admin --help-command remove-certificate

for usage, but it looks like you can specify either .pem files or the DN/CA strings.
