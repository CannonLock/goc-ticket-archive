I'm not sure there is a definitive solution to this, but wanted to document
it so everyone was aware and for discussion purposes.  I would hope that we
can find some resolution however.

This is little complex to describe... so here goes.

On 6/26/06, I traced a problem with a corrupt GUMS generated grid-mapfile
back to the VOMS server.

The grid-mapfile looked like this:
"/DC=gov/DC=fnal/O=Fermilab/OU=People/CN=Mark . Green /UID=mlgreen" cdf
"/DC=gov/DC=fnal/O=Fermilab/OU=People/CN=Mark Green
/USERID=mlgreen" cdf
"/DC=gov/DC=fnal/O=Fermilab/OU=People/CN=Mark Green /UID=mlgreen" cdf

Hopefully the above did not get line wrapped.... the 2nd entry is actually
split across 2 lines.  There is apaprently a newline character that is
causing it.  I found this on https://voms.cnaf.infn.it:8443/voms/cdf .

To verify I ran:
voms-admin -url https://voms.cnaf.infn.it:8443/voms/cdf list-members /cdf

and could see the newline split of the output.

Now, this is likely an anomoly in the CDF VOMS as these are Kerberos
certificates and my guess is they were doing a mass update (probably just a
SQL update of the database) to add "USERID" DN entries for each "UID" entry
they had.  This was done to get around a problem we have had with KCA
certificates and the versions of openssl that have evolved over

    time....enough on this.

When you do a retrieval using mysql from the GUMS database (containing this
entry from the CDF VOMS), you can see that the entry is split across 2
lines in the SQL output.
select * from USER where DN like '%Mark Green%';

As for the affect on authorization services on a CE node:
1. pre-ws using grid-mapfile
- No affect as the gatekeeper and gsiftp services read the
file with each invocation and just skip over the bad line(s).

2. web services using grid-mapfile
- On startup of the container (globus-ws start), the container
will fail to start if it detects a corrupt grid-mapfile.

- On startup with a good grid-mapfile, the container
apparently caches the grid-mapfile. Then when an authorization
is required, it checks the timestamp on the grid-mapfile
to see if it has been updated.

If it has been updated, it will attempt to cache the new file.
Upon detecting the invalid line(s), it throws an exception
in the container.log and back to the submitting user.  It aborts
the re-caching attempt and continues running with the old
cached grid-mapfile.

The submitting user who got this error will be authorized on
their next submittal (assuming they were in the grid-mapfile
to begin with).

3. pre-ws and web services with PRIMA/GUMS authorization services
- I have not as yet been able to test this, but hope to be able to.
My expectation is that the only certificate affected might be
the bad one with the newline (or whatever character it is).

Possible resolutions:
1. In globus-ws:
- It should ignore the individual grid-mapfile entries (as the
pre-ws services does).  This minimizes the number of users
affected.  It would also allow the individual submitter who
got unlucky enough to have been the 1st one submitting a job
at the time a refresh was necessary from having their job
rejected.
-  The log4j should be setup so these type messages generate
a mail message to an administrator.

2. In GUMS:
- I'm not sure if GUMS can detect if this type of data is passed
it from GUMS (or any other source), but if possible, it should
generate a warning when the refreshes are done from the VOMS
servers (or other source).  It should refresh all entries but the
ones in error.
- It may be more possible to detect the condition when it is
generating the grid-mapfile than on the refresh of it's USER table.

I should note here that the edg-mkgridmap process apparently
detects the condition and does not create an invalid grid-mapfile
entry.
