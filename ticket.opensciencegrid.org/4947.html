    Jun 4, 2008 03:17 PM UTC by Anthony Tiradani 
    Hello,

The "GlueCEInfoDefaultSE=UNAVAILABLE" message is due to a missed use
case in the configure-gip script.  The fix for this is already in the
ITB and waiting for the OSG 1.0 release.

I think that the fastest way to fix this for you is to edit
$VDT_LOCATION/monitoring/osg-attributes.conf:

* The line that reads OSG_DEFAULT_SE="UNAVAILABLE" needs to be changed
to OSG_DEFAULT_SE="osggate.clemson.edu"

Then re-run configure-gip:

source $VDT_LOCATION/setup.sh  (or csh if you are using that shell)
. $VDT_LOCATION/vdt/setip/configure-gip

This should fix the problem and your site should begin passing that test
the next time the test is run.

    Hello,

There are a couple of things that need to be addressed.  First, when I
tried to perform some simple validation tasks, I received the following
error:

"GRAM Job submission failed because the job manager failed to create an
internal script argument file (error code 22)"

This is usually because the user's home directory has not been created.
I tried coming in using an ops proxy, so there should not have been a
problem.  Please check to make sure that the ops VO is supported and
that the Unix user for that VO has been created with a home directory.

Second, the error shown on the
http://gip-validate.grid.iu.edu/production/index.html page can be
misleading.  The error actually is "GlueCEInfoDefaultSE=UNAVAILABLE".
Do you have a Storage Element for your site?  If not, then the
configure-osg.sh script will fill in "UNAVAILABLE".  I am trying to find
out if that is an error on the part of the configure scripts or if that
is a false positive on the part the the gip validate page.

I will get that information and get back to you on that.  If you fix the
first issue (the Unix user issue), I will run some simple test jobs (id,
date, etc) and perform some gip validation tests to make sure that
CLEMSON-IT is reporting correctly.

First see if your gip is reporting correctly. Run
% $VDT_LOCATION/gip/bin/gip_info
a see if the missing CEUniqueID is present.

If it is absent you could be having the same problem I encountered. For
my case I figured out the CE section might be dropped (for pbs batch
system) if there are any problem running any plugin or provider.

To confirm if you are seeing the same issue please do the following:
% export GIP_LOCATION=$VDT_LOCATION/gip
% $GIP_LOCATION/plugins/osg-info-dynamic-plugins
% $GIP_LOCATION/providers/osg-info-dynamic-providers

If there are any errors in either of the plugin or provider command,
that could be the problem.

If gip_info is reporting correctly then it could be either the CEMon not
transporting it correctly or BDII dropping it due to some value that

    violates the schema.
