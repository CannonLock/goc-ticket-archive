Another approach is to use a user proxy with a slightly longer lifetime. We allow proxy lifetimes up to 7 days. It is easy to configure this with grid-proxy-init. If the machine is a secure machine, we also allow automatic proxy renewals. If you could tell me what type of machine you will use for proxy renewal, I can tell you whether this is allowed or not. To give you an idea, what we mean by a secure machine:
1) the machine does not serve a large number of internet facing publicly available services. for example, an apache server has a much higher possibility to get compromised than a server that only allows users from a single domain.
2) the machine has all its security patches up to date and professionally managed.
3) the machine has network filters in place to restrict access
4) the console access to the machine is only via secure means, ssh with no anonymous access or no root username and password,e ct.
5) the privileged systems such as sudo are properly logged.
6) The administrator professionally observes the machine that he will recognize if the machine is compromised.

Please let me know which way you like to proceed.

    Jul 15, 2014 08:26 PM UTC by Vince Neal 
    Hi, Ken

As far as I am aware it is not technically a violation of the rules to use a method to auto-renew your proxy though I do not believe it is a best practice..  Another option is to specify the length of the proxy by using the argument -valid H:M. (grid-proxy-init -valid 196:00)
You can enter any value for H:M up to the expiration date and time of your certificate.  This is certainly not a best practice and many sites will reject a proxy with a months long expiration.  As you noted, a better solution would be to get a host certificate.

Please let me know if I can assist further.

Thanks,
