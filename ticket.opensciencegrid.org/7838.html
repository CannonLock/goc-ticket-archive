NFS-Lite.  For now, we are therefore resorting to a heuristic: if the
requested executable is in the user's home directory (which contains the
GASS Cache), then it is transferred to the worker node.  Otherwise, it
is assumed that the user wants to run the specified executable as it
exists on the worker node.

Open Science Grid FootPrints wrote:
> [Duplicate message snipped]
    Dec 14, 2009 05:11 PM UTC by GLOW-TECH 
    The reported error indicates (admittedly in a confusing way) that the
user's executable is dynamically linked with glibc 2.4.  Many of the
execute nodes here are Scientific Linux 4 machines, which means they
only have glibc 2.3.

As an immediate workaround, I would suggest compiling the program
against glibc 2.3.4 or earlier.  Using static linking might also work.

We will be upgrading all worker nodes to Scientific Linux 5 within the
next few weeks.  Then I would expect glibc 2.4 applications to be
supported (SL5 uses glibc 2.5).

--Dan
