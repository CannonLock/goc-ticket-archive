    Hi all,

Sure I'll be happy to address these. I think most of them fall into the
category of "sharp edges that could cut an unsuspecting user" rather
than outright bugs.

To summarize:

a) Does not notice duplicate userGroup names.
b) Stale entries in USER table.
c) Stale entries in MAPPING table.
d) Concern about uniqueness of DNs (and FQANs) in USER table.
e) Does not terminate on duplicate groupMapping
f) No update on config reload.

Answers:

a),e)
===================
Yes, these are 'sharp edges' that could get a new admin. I agree that
some day it would be nice to polish these down. I know Gabriele's
original development roadmap involved creating a config file creation
front-end, which would have a simplified input format and/or a Web-based
configuration editor. Either (or both) of these would naturally prevent
this sort of mistake. I think this functionality was kept out of the
core GUMS code in order to keep it small and fast (or as small and fast
as a J2EE Java app can be--which may not be that fast, and certainly
isn't small.)  :-)

b),c),d)
==================
These concerns seem to reflect a misunderstanding of how the Hibernate
ORM thing is working. The database isn't queried directly and the
developer doesn't create the queries--they get created automatically by
Hibernate. GUMS only interacts with Java objects that have been

    re-consitituted from the DB. Hibernate (and GUMS) doesn't even
necessarily need a database as a back-end.

So for b) and c), there could indeed be stale entries in the USER DB,
with GROUP_NAMEs not in the current config, but these would never be
reconstituted into objects, so GUMS would never see them if they are not
named in the current config. The ones that do get reconsituted into
objects get completely re-created every 6 hours. The MAPPING db is
simply additive. If you remove a pool account persistenceFactory, the
entries would be ignored by Hibernate.

For d) DNs and FQANs don't need to be unique within the whole database,
just unique within a UserGroup object (which maps to a single GROUP_NAME).

f)
====================
By 'reload and update' I assume you mean connect to all the VOMS
servers to get updated membership lists. I think this omission is
intentional--the idea being that most alterations will involve things
like which GroupMappings are valid for which HostGroups . These can be
changed without requiring a user list update.

We already accept the idea that who is in a VO can change and our system
doesn't pick it up for 6 hours. If a new groupMapping is created and
hasn't been updated from a VOMS server, it just doesn't have any users
in it yet.

I don't understand what you mean by an 'out of sync' condition. A GUMS
Configuration consists of basically a list of groupMappings and a list
of hostGroups. HostGroup entries refer to groupMappings. GroupMappings
are defined as lists of user DNs and FQANs which get updated from VOMS
servers. When a config is reloaded, all the groupMappings referred to by
a hostGroup must exist, or an error is thrown, i.e. the Config is
logically consistent.

During the course of a single *call* there is a caching mechanism such
that the state of GUMS at the beginning of the call is the one used
until the call is completed. This avoids synchronization issues with
Hibernate.

--------------------------

I admit I could be missing something, and there certainly could be
failure modes Gabriele didn't think of, but I'm pretty sure the basic
object mapping in Hibernate is correct, or we would have run into big
problems before now. I, for one, have renamed groupMappings and
re-arranged everything in the config file and deleted my GUMS db several
times since I took things over here at BNL. There were occasional
glitches in terms of users getting authorized, but it was always because
1) GUMS did exactly what I told it to, 2) I hadn't triggered an
updateGroups(), or 3) a VOMS server was down. I've never seen a failure
traceable to a logically incorrect reconstruction of a Java object from
the GUMS DB.

Cheers,

--john

    Jun 29, 2006 04:50 PM UTC by Footprints Remedy 
    Short Description=GUMS gums.config validation - OSG 0.4.1 / VDT 1.3.10b

Status=Service Desk

This is in OSG 0.4.1 / VDT 1.3.10b / GUMS 1.1.

The following identify some validations on the gums.config file
that I believe are missing from GUMS and can cause some
incosistent and/or incorrect account assignments and authorizations.

John Weigand

a) the name attribute of the userGroup element is used as the
GROUP_NAME in the USER table.  There is no check to insure
uniqueness, so the 2nd groupMapping encountered will overlay
the 1st in the database if the userGroup name is the same
and you don't even know it.

b) if you change the name of the userGroup name element from say cmsuser
to uscmsuser, the original  'cmssuer' USER table entry is not removed.
So you now have two sets of entries for everyone in that VO with the
same FQAN (Role not even a factor). My guess is that it is just
chance which one they pick up (based on the db query) when
authorization is requested.... and I don't know how gums responds
when it tries to find the account mapping for a non-existent GROUP_NAME.

c) The same is true on the removal of a groupMapping element.
It is never deleted from the database.  Although this may not
be an issue since GUMS will not attempt to process it when an
authorization is attempted... since there will be no entry in the
the hostGroups.... BUT... if the FQAN is the same for a similar group,
it will be a problem.

d) I haven't tried this one, but I believe the FQAN created from the
gums.config table should be unique across the gums.config file.

    The USER table entries should be unique on DN and FQAN.
If I have the same DN with multiple entriies for the same FQAN in
the USER table, it is just chance which one will be used.

I'm pretty sure that originally there were no indexes on these tables
and no transaction level updating in GUMS.
I think Gabrielle added some with 1.0 or 1.1.
I think he may have a missed a few that would eliminate these
conditions.

e) This one is not related to the translation from config to database,
but I believe causes a problem when GUMS attempts to process an
authorization request. The name attribute of the groupMapping element
should also be unique across the gums.config.
GUMS currently terminates the table update if you have a groupMapping
not in the hostGroups element.
HOWEVER, it does not when you have groupMapping elements of the same
name.  Now which, groupMapping userGroup entry GUMS uses .. I have no
clue.

f) When a gums.config file is modified, GUMS does detect that the file
has changed.  But it does not attempt to reload and update.
This is something the admin must perform manually and is likely to
be missed.  This creates a out of sync condition since authorizations
are immediatley using the new mapping against an outdated database.
