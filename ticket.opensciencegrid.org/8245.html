- Java 1.6.0u6 or lower

Hey Brian,

In setting up to do your test  I also changed our install script a bit and reinstalled using the change.   The change involves copying dCacheSetup to
poolSetup and to gridftpSetup.   Those particular files are then no longer soft-links to dCacheSetup.

I changed the java memory used in poolSetup to 2048m instead of 1024m.   I set the buffer variables in dCacheSetup and poolSetup back to
1048576 and only set  the large buffers 8388608 in gridftpSetup.

That was it…after reinstalling and running strace I get to enabling pool umfs13_4…and then there is NO problem!   I will redeploy using the new
setup on all our storage nodes.

FYI,  I had previously tried  increasing the java memory to 2048m when I used only ‘dCacheSetup’ but I still had the problem so I am guessing the
buffer setting was the culprit (or perhaps just the combined memory pressure).

However, I will be peeved if I have to run ‘strace’ in another window to make this work J

More soon,  once I verify things are working cluster-wide.

Thanks,

Shawn

Can you run strace against a pool Java process?  You'll need the "-f" flag, otherwise Java makes strace unhappy.

(replace 1234 with your PID, of course)  Look for the utime call and see what error is getting returned from the system.
