    Sep 29, 2006 11:13 PM UTC by leighg 
    From Alain Roy-

- progress on GUMS memory leak
>>Ted has identified the GUMS memory leak using jProfiler. It turn out
>>that both Tomcat and Axis cache a session for every authorization
>>request. The sessions were not configured to expire, therefore they
>>were never garbage collected. Ted has changed the parameters of the
>>axis sessions in the configuration and shows that the memory is
>>periodically garbage collected (See attached plot: blue area is
>>utilized memory, green is free memory: after a certain threshold of
>>memory growth, the garbage collection is triggered and memory usage
>>drops).
>>Ted still wants to spend some time investigating if this solution is
>>the best for the problem.
    Sep 26, 2006 04:13 PM UTC by Mariachi 
    Our new full-time Java developer and I are both looking into it. He is
re-working the build system (with Maven v.2) to make it easier to do
testing.

We have a new version of the privilege.jar and are looking to get an
updated version of the egee-trustmanager . We also now have instructions
from Tim Cartwright at VDT on running GUMS in Tomcat standalone (without
Apache). The plan is to check each of these, in turn, to at least
identify which component/configuration causes the leak.

Again, for the record, our formal recommendation for the VDT is to have
a weekly Tomcat restart until this is tracked down. For reference, I've
copied below an exchange on the LCG-ROLLOUT list with the same
recommendation. (Maarten Litmaath is one of the gLite developers.) So
we're hardly the only ones with this common problem.
