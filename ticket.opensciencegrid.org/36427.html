    Mar 14, 2018 03:33 PM UTC by Thomas Lee 
    nf_conntrack is Netfilter (aka iptables) connection tracking. As with all Operations Center machines, this machine's firewall settings block most incoming packets that aren't part of a locally-established outgoing connection, for security purposes. This invokes netfilter connection tracking, which must then keep track of all outgoing connections. There's a sysctl parameter, net.netfilter.nf_conntrack_max, which limits the maximum number of connections that can be tracked, and when this limit is reached, nf_conntrack can't track any more connections, meaning that the outgoing connection is made, but the incoming packets are then rejected, leading to a TIME_WAIT state as the process waits for the response. Most software will impose a timeout and give up if there appears to be no response to an outgoing connection; DNS resolvers usually have a fairly short timeout.

This VM has an astonishing number of outgoing network connections active.

# netstat -tpn | wc -l
2012

Many of them are in a TIME_WAIT state, as they are probably the outgoing connections waiting for responses that were blocked by the firewall due to the connection tracking table being full. Whatever process makes those connections, its timeout seems rather high. The question remains what is making all those connections; if many connections are being made to hosts that actually aren't responding (as opposed to just seeming not to respond due to the nf_conntrack table being full), and those connections are hanging on for a long time in a TIME_WAIT state, that could result in filling up the table.

Let's look at something else ...

# netstat -tpn | tail -n +3 | awk '{ print $5 }' | sort | wc -l
2418
# netstat -tpn | tail -n +3 | awk '{ print $5 }' | sort | uniq | wc -l
313

(Numbers will vary because connections are constantly being made and timing out.) This is telling us that there are 2418 connections being made to about 313 unique destinations. Something is trying over and over to connect to the same targets even though the previous attempt hasn't timed out yet.

I have doubled the size of the connection tracking table and decreased the generic and TCP timeouts. Let's see what effect this has. For now, I notice that there haven't been any nf_conntrack entries in /var/log/messages since the change.
