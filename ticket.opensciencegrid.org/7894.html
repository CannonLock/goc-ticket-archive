Date-Sent: Monday, December 28, 2009 4:25 PM -0800

Dear GOC,

Terrence is on vacation, so James and fkw are the backup sysadmins at UCSD.

We just fixed a set of 3 corrupted CRLs on our cluster. The cluster was unusable for
CMS until this was fixed.
I.e. this sort of thing stops CMS operations at a site dead in the tracks.

I believe that this experience is an indication of a bug in:

/code/osgcode/wn-client/fetch-crl/share/doc/fetch-crl-2.6.6/fetch-crl.cron

The symptom is that crl's are successfully updated, unless for some reason a
zero-size file is written instead of the proper
*.r0 file for a CRL. This zero-size file is then never again updated. The problem is
thus never fixed despite the fact that
our crontab says:

root@codefs /code/osgcode/wn-client/globus/TRUSTED_CA# crontab -l
18 0,3,6,9,12,15,18,22 * * *
/code/osgcode/wn-client/fetch-crl/share/doc/fetch-crl-2.6.6/fetch-crl.cron

We had to replace the broken CRLs by hand with ones that were not corrupted.
We figured out which ones were broken by successive execution of srmls. Thankfully,
srmls provides you with the
filename of the CRL it can't read as part of its error message.

I'll let you follow up on this with Terrence once he comes back.

    Just didn't want to forget this during the holiday season.

thanks, frank
