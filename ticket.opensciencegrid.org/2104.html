Catherine Ruby wrote:
>
>The goal of our work with SRM and storage elements was to provide an
>Operations-Dashboard-esque interface for storage elements running SRM,
>where small test modules (Site Functional Tests, or SFTs) could be run
>for each storage element and published on a Storage Elements Dashboard
>for an overview of an SE's functionality.  We were looking into the
>package srm-tester, which can run a full diagnostic on SRM functionality
>(operations ping/get/put/copy/get,put/get,copy/put,copy/all) in a
>modular manner, allowing a site functional test to consist of simply a
>call to the installed srm-tester for the given operation.
>
>There are two key components to launching SE Dashboard infrastructure
>similar to that of the Operations Dashboard:
>
>a) the establishment of a set of SFTs which can be run on an SE for

    >environment information in the database - ie, any path/port/env
>variables key to running srm-tester on the site should be stored in the
>database and usable for executing SFTs in an automated fashion by
>building the appropriate argument string to pass to the srm-tester
>
>b) the ability to discover this information in a programmatic manner,
>facilitating easily rolling in new SE's for testing as well as automated
>updates if an SE changes its configuration
>
>As far as the last time I heard regarding getting SE testing online,
>this is what I know:
>
>1) There is currently no means of programmatically discovering SE site
>information.  With the exception of contacting a site admin directly,
>there is no way of determining the set of 6 key attributes for running
>reliable SFTs - these are
>    - the fully-qualified host name
>    - the SRM version (as far as I know everyone is running version 1,
>version 2 is much more flexible and Fermilab had plans to roll over to
>it in the future as of the March 21 meeting)
>    - the service URL - the location of the SRM service on the machine
>which facilitates SRM transactions
>    - a policy URL
>    - a boolean OVERWRITE bit - i.e., whether or not the SE is
>configured to allow overwriting files, this will explain certain
>failures on machines when attempting to perform a put
>    - separate GET/PUT paths for the MIS VO (because an SE may allow
>read-only access for a file location)
>There was talk about adding this to the GRIS such that we can reliably
>discover this information and use automated scripts to support the SE
>Dashboard, I'm not sure whether or not progress was made on this since
>the March 21 meeting.
>
>2) I am having trouble executing srm-tester reliably myself at this
>time, I collaborated with Alex Sim, Vijaya L. Natarajan and Shaowen Wang
>on this and they were quite helpful in sending me informaiton, but it is
>unclear to me whether it is a configuration problem on my end, a
>permissions problem with the proxies I was using on the site, or
>configurations on the site which prohibit the operations I was
>attempting to perform.  I've executed the srm-tester successfully in the
>past (meaning that it ran without a local error but encountered errors
>on the sites I was testing), however the failure outputs are somewhat
>brief and it was unclear where the problems were occuring and whether
>the service was actually down or I was attempting some sort of illegal
>operation with respect to site policies.  Further documentation on the
>srm-tester (if available) would help me sort this out, but at the point
>where we left off I was considering attempting to initiate the SRM
>service directly and writing SFTs manually, if only to help me determine
>exactly where I was encountering failures.
>
>Thank you,
>Cathy
