    Hello,
recent SAM tests have probably showed that your site is not correctly configured to use the voms information
Here is some information provided by Maarten Litmaat (available in https://prod-grid-logger.cern.ch/elog/ATLAS+Computer+Operations+Logbook/18501 but provided in case you cannot read ATLAS elog):

We do apologize if you are already aware of the issue and already working on this problem, but please reply to this ticket with the current status. - ADCoS Shifter

------------------------------------------------------------------------------------------
The sites simply should fix their configurations to comply with
what is advertised in the ATLAS VO card on the CIC Portal:

https://cic.gridops.org/index.php?section=vo&page=homepage&subpage=

The best would be for them to fix their YAIM site-info.def files
with the correct definitions to support ATLAS:

----------------------------------------------------------------------
VO_ATLAS_VOMSES="\
'atlas lcg-voms.cern.ch 15001 \
/DC=ch/DC=cern/OU=computers/CN=lcg-voms.cern.ch atlas 24' \
'atlas voms.cern.ch 15001 \
/DC=ch/DC=cern/OU=computers/CN=voms.cern.ch atlas 24' \
'atlas vo.racf.bnl.gov 15003 \
/DC=org/DC=doegrids/OU=Services/CN=vo.racf.bnl.gov atlas 24' \
"
VO_ATLAS_VOMS_CA_DN="\
'/DC=ch/DC=cern/CN=CERN Trusted Certification Authority' \
'/DC=ch/DC=cern/CN=CERN Trusted Certification Authority' \
'/DC=org/DC=DOEGrids/OU=Certificate Authorities/CN=DOEGrids CA 1' \
"
----------------------------------------------------------------------

And reconfigure or create the missing LSC file(s) manually.
For the BNL server:

----------------------------------------------------------------------

    $ cat /etc/grid-security/vomsdir/atlas/vo.racf.bnl.gov.lsc
/DC=org/DC=doegrids/OU=Services/CN=vo.racf.bnl.gov
/DC=org/DC=DOEGrids/OU=Certificate Authorities/CN=DOEGrids CA 1
----------------------------------------------------------------------

A quick hack would be to upgrade the lcg-vomscerts rpm to the latest
version (not provided in the DPM repositories, because it should not
be needed on a correctly configured DPM):

http://eticssoft.web.cern.ch/eticssoft/repository/org.glite/lcg-vomscerts/6.1.0/noarch/lcg-vomscerts-6.1.0-1.noarch.rpm
