    Scott,
no, the values for BNL_ATLAS_SE disk and tape as they are shown below are correct. The problem arises whenever the values for BNL_ATLAS_SE_AWS_x are not equal to 0. What we've learned last week is, while the values are correctly aggregated across all items belonging to the same resource group, and correctly reported to REBUS, the value that one finds in the report sent out by the GOC shows only the value under BNL_ATLAS_SE.(disk).
I found this out ~2 weeks ago when comparing the values shown in REBUS with the ones shown in the GOC report. At that time the values in OIM were as follows
BNL_ATLAS_SE: 11000 TB (disk)
BNL_ATLAS_SE: 12000 TB (tape)
BNL_ATLAS_SE_AWS_East: 500 TB
BNL_ATLAS_SE_AWS_West: 500 TB

The values found in REBUS were
BNL_ATLAS_SE: 12000 TB (disk)
BNL_ATLAS_SE: 12000 TB (tape)

The values found in the GOC report were
BNL_ATLAS_SE: 11000 TB (disk)
BNL_ATLAS_SE: 12000 TB (tape)
which is not consistent when considering the AWS fractions.

So, in order to maintain consistency, and as we had no intentions to have the AWS part reflected in the installed capacity reporting we decided to set those to 0.

Michael
