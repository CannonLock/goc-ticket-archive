    Jul 20, 2009 07:26 PM UTC by Thomas Lee 
    The problem stems from the fact that the GRNOC doesn't have a DOEgrids cert, whereas OIM now requires one.  That leaves us with two options: make an exception in OIM so the GRNOC can probe the server without a DOEgrids cert, or get the GRNOC a DOEgrids cert.  Neither one is ideal.

Option 1: Make an exception in OIM.

Pros: Easily done; easily maintained.  The GRNOC is able to probe whether the server responds.

Cons: The GRNOC is unable to probe under the same precise conditions that users' browsers would be subjected to.

Option 2: Get the GRNOC a DOEgrids cert.

Pros: The GRNOC would be able to probe under the same precise conditions that users' browsers would be subjected to.

Cons: Slightly more difficult to accomplish, and seriously more difficult to maintain: there would be a DOEgrids cert that we would be responsible for keeping up to date, but that we could not directly test or access.

My analysis: If we follow Option 1 and put an exception in the Apache configuration that lets the GRNOC's Nagios probe the server without a certificate, we accomplish everything we truly need.  The only situation this would not be able to monitor would be a hypothetical condition where the server suddenly began to disallow users with valid certificates.  This could happen if, for example, the server certificate expired, but we have other mechanisms to prevent that.  Before the change to the server configuration that caused this whole problem, the Nagios probe's real purpose was simply to make sure the Apache server was up, not to test for expired server certificates, and if we need Nagios to test the certificate separately, I'm sure a separate probe already exists for that.
