    Nov 8, 2010 10:23 PM UTC by Terrence Martin 
    If by checkpoint you mean making a random write to a central file no. You
could write a small job related file via say lcg-cp, but small files are
undesirable and need to be managed to keep from getting out of hand. The idea
is that data storage is for job data meant to be analyzed or as a seed for
monte carlo not job state.

This really comes back to the core issue with submitting so many small jobs to
perform relatively small activities and then trying to stitch together state
between those submitted jobs.

You could also send work flow state to a central service managed by the VO.
That way scalability is on the VO, not the site where the site is just
providing CPU and storage for data to be analyzed as necessary. In contrast to
storage networking is fairly easy to come by.

Terrence

    It is mounted under /hadoop, however you should be aware that

a) You cannot write to this area from the worker nodes
b) This is not a POSIX file system, it offers data storage with basic file
semantics, soft links, hard links and other POSIX features are not available
c) Writing to this area is done via SRM (lcg-cp or srmcp)

Ligo has written to this area at UCSD in the past, as recently as Oct 20th
actually. Whoever those folks are probably have more insight into how you can
make use of the hadoop storage area at UCSD.

Terrence
