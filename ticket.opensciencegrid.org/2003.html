No, it has not.  While we now provide an interface for setting the MySQL root
    
password, we don't record or re-use it in any of our other scripts.  Normally mysql
    
would prompt for a password, but due to some internal systems we run it without a
    
tty.  This might be easy to fix, but it might not be.  We're past the feature freeze
    
for 1.3.7 and almost to the release itself.  I'll check it out, but I think it might
    
be too late to add this feature.  I'll let you know either way and update the release
    
notes accordingly (bug fix vs known problem).  
    

    
In the meantime they should be able to work around the problem by temporarily
    
unsetting the root password while taking the appropriate security precautions.  This
    
should be fixed in 1.3.8, which will be out before OSG 0.4.
    

    
--Nate

Apache HTTPD 2.0.54

VOMS 1.4.2
    
VOMS Admin Server 0.7.6
    

    
The configure_voms script is used to add a VO in VOMS. This works  
    
fine for the initial implementation and also works fine if you add  
    
additional VOs for that VOMS instance.... so long as you have not  
    
assigned a 'root' password for the MySql database.
    

    
If, after the initial configuration, you add a 'root' password, the  
    
script will fail when it attempts to create the database. Inside the  
    
configure_voms script, it is the execution of this step that causes  
    
the error...
    

    
[root@cmssrv05 setup]# configure_voms --server y --vo weigand --mail- 
    
from weigand.fnal.gov --smtp-host smtp.fnal.gov
    
Command failed: cat /home/vdt/osg-itb-016/voms/var/etc/edg-voms-admin/ 
    
weigand/install-voms-1-mysql-as-root.sql | /home/vdt/osg-itb-016/ 
    
mysql/bin/mysql -S /tmp/mysql.sock -u root
    

    
It does not tell you why, but in this case I know the root password  
    
has not
    
been supplied.
    

    
This then puts you in a state where you have a partial installation,  
    
i.e., a
    
subsequent exexution then fails in the edg-voms-admin-configure step...
    

    
[root@cmssrv05 setup]# configure_voms --server y --vo weigand --mail- 
    
from weigand.fnal.gov --smtp-host smtp.fnal.gov
    
Command failed: /home/vdt/osg-itb-016/voms/sbin/edg-voms-admin- 
    
configure install --port 15011 --vo weigand --mail-from  
    
weigand.fnal.gov --smtp-host smtp.fnal.gov --password ABBBBCCCCDDDD -- 
    
voms-group=daemon --tomcat-group=daemon --certdir=/home/vdt/osg- 
    
itb-016/globus/TRUSTED_CA
    

    
..again with no indication of the problem, but it is caused by the  
    
directory
    
structure for the VO(weigand) being partially created. This is an old  
    
VOMS
    
installation problem.
    

    
The files/directories that are getting updated on failure are:
    
./vdt-app-data
    
last_port
    
voms/weigand (VO_NAME)
    
voms-admin/weigand (VO_NAME)
    

    
The latter 2 directories need to be removed in order to undo this. The
    
last_port file can be reset, but it is not necessary.
    

    
Apache and Tomcat appear to untouched on this failure.
    

    
The solution should prompt for the root password and retain it as  
    
there are
    
4 mysql executions in this configure_voms script and if you blow any  
    
one of
    
them you will wind up in this partially installed state.
