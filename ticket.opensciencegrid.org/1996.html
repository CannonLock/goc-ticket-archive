Hello,

I'm contacting the fGOC support center due to the continued failure of
this resource listing fGOC as the support center :

FIU-PG

While this site passes the verification tests for the MIS VO that we use
here at the Grid Operations Center, it reliably fail tests from the
NanoHUB VO.  This site has published its acceptance of NanoHUB VO jobs via
the grid3-user-vo-map.txt file.  This is currently the only way that
resource users have to determine a resource's support for a particular VO.

Please contact the administrator for this resource and ask them to do one
of the following:

1. If the resource desires to allow jobs from the NanoHUB VO then enable
the NanoHUB user account.  The NanoHUB VO uses the following certs and
CA's:

The credential DN is:
/CN=Invigo Service04/OU=Purdue TeraGrid/O=Purdue University/ST=Indiana/C=US

The certificate is signed by this CA (hash 67e8acfa):
/CN=Purdue TeraGrid RA/OU=Purdue TeraGrid/O=Purdue University/ST=Indiana/C=US

That CA's certificate is signed by this root CA (hash 95009ddc):
/CN=PurdueCA/O=Purdue University/ST=Indiana/C=US

The files for both of these CAs are included in the latest OSG/VDT
certificates packages.

or...

2. Don't publish support for this VO by commenting out the nanohub user in
the grid3-user-vo-map.txt file.  Do this by placing a '#' character in the
first column of the line containing the mapping for that VO.  That file
can be found on the gatekeeper of the resource in the file:
$VDT_LOCATION/monitoring/grid3-user-vo-map.txt

Thank you for you time in this matter.  If you have any questions, please
contact me.

Lastly, there are some issues with the publication of VO's that you may
    
not be aware of but will have an affect on your validation process.
    
First, VORS relies on the presence of a file on the gatekeeper called
    
grid3-user-vo-map.txt to determine the supported vo's.  It is generally
    
correct when a site is using edg-mkgridmap for authentication but if a
    
site is using GUMS then that file may not be kept uptodate (or never
    
modified from the default) by the resource's admin.  Currently, there is
    
no good way that I know of to detect the VO's supported by a resource when
    
it authenticates via GUMS. A mid-February report on the sites then using
    
GUMS is available at:
    
http://www.ivdgl.org/~leighg/gums-report.html
    
There has been quite a bit of discussion on this topic lately and 
    
hopefully a more reliable means of determining vo support will be 
    
found in the future.
