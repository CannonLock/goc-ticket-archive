    Jan 19, 2010 07:14 PM UTC by Ian Stokes-Rees 
    I sent the following update at 1:35pm EST but it does not appear on the
ticket, so I will send it again:

Unless I am mistaken, globus-job-run should only be used for "immediate"
interactive/synchronous jobs.  It should not have a job manager
specified (which I believe will cause it to default to executing a
"fork" job on the CE head node).

globus-job-run will wait for a result, and if the command is sent via a
batch system, then it will be queued and may not run for hours or days,
depending on the depth of the queue.  You will not get any output until
it runs.

globus-job-submit should be used to submit to a batch system (and then
globus-job-status and globus-job-get-output), although within OSG you
should use the OSG Condor-G based job submission mechanisms instead.

Right now there are free slots, so the command completes relatively
quickly for me:

$ globus-job-run abitibi.sbgrid.org:2119/jobmanager-condor /bin/date
Tue Jan 19 13:13:30 EST 2010

I hope that provides some useful information.  I cannot comment on the
Globus error 17 issue, except to say that it would be helpful to have
timestamps for when the user executed the command, which will allow us
to cross check with our logs.  We have had a few periods when the system
has not been available.

Regards,

Ian
