    Mar 16, 2015 11:36 PM UTC by OSG-GOC 
    Hi,

We would like to unify the structure of the sub-VO entries under the Fermilab VO. Some experiments have multiple groups and local accounts, while others don't. We'd like a single structure in the template for each experiment to simplify things across the board. Below we describe the changes that we would like to see.

1) This is a list of expmeriments/sub VOs that have Fermilab as their umbrella VO
• annie
• argoneut
• cdms
• chips
• coupp
• darkside
• fermilab (this is for small-scale efforts not being enough to have their own group)
• genie
• gm2
• lar1nd  (NOTE: this is a replacement for the lar1 group)
• lariat
• minerva
• miniboone
• minos
• mu2e
• nova
• numix
• patriot
• seaquest
• uboone

2) For each of the above groups we'd like to have the following structure (using annie as the example):

<groupMapping name='annie' accountingVo='fermilab' accountingDesc='FERMILAB'>
<userGroup className='gov.bnl.gums.VOMSGroup'
url="https://voms.fnal.gov:8443/voms/fermilab/services/VOMSAdmin";;
persistenceFactory='mysql'
name='annie'
voGroup="/fermilab/annie"
sslCertfile='/etc/grid-security/http/httpcert.pem'

    sslKey='/etc/grid-security/http/httpkey.pem'
matchFQAN="exact" />
<accountMapping className='gov.bnl.gums.GroupAccountMapper'
groupName='annie' />
</groupMapping>

<groupMapping name='annieana' accountingVo='fermilab' accountingDesc='FERMILAB'>
<userGroup className='gov.bnl.gums.VOMSGroup'
url="https://voms.fnal.gov:8443/voms/fermilab/services/VOMSAdmin";;
persistenceFactory='mysql'
name='annieana'
voGroup="/fermilab/annie"
voRole="Analysis"
sslCertfile='/etc/grid-security/http/httpcert.pem'
sslKey='/etc/grid-security/http/httpkey.pem'
matchFQAN="exact" />
<accountMapping className='gov.bnl.gums.GroupAccountMapper'
groupName='annieana' />
</groupMapping>

<groupMapping name='anniegli' accountingVo='fermilab' accountingDesc='FERMILAB'>
<userGroup className='gov.bnl.gums.VOMSGroup'
url="
https://voms.fnal.gov:8443/voms/fermilab/services/VOMSAdmin";
;
persistenceFactory='mysql'
name='anniegli'
voGroup="/fermilab/annie"
voRole="pilot"
sslCertfile='/etc/grid-security/http/httpcert.pem'
sslKey='/etc/grid-security/http/httpkey.pem'
matchFQAN="exact" />
<accountMapping className='gov.bnl.gums.GroupAccountMapper'
groupName='anniegli' />
</groupMapping>

<groupMapping name='anniepro' accountingVo='fermilab' accountingDesc='FERMILAB'>
<userGroup className='gov.bnl.gums.VOMSGroup'
url="https://voms.fnal.gov:8443/voms/fermilab/services/VOMSAdmin";;
persistenceFactory='mysql'
name='anniepro'
voGroup="/fermilab/annie"
voRole="Production"
sslCertfile='/etc/grid-security/http/httpcert.pem'
sslKey='/etc/grid-security/http/httpkey.pem'
matchFQAN="exact" />
<accountMapping className='gov.bnl.gums.GroupAccountMapper'
groupName='anniepro' />
</groupMapping>

So each group has four entries, <group>, <group>ana, <group>gli, <group>pro, with the corresponding vo roles. Now the only exception here would be "fermilab", where the voGroup would be just be "fermilab" instead of "fermilab/fermilab." Again, we'd like this structure to be in place for all groups listed in 1). Most of them are already in place but a few are not.

3) There are other standone VOs that, while managed by Fermilab, are not under the umbrella VO. Those are

• cdf
• des
• dzero
• lbne
• lsst

For these, we would like the same 4-entry structure (no suffix, ana, gli, and pro). The difference with respect to the groups within the Fermilab umbrella VO is that the voms url would be different. Instead of being  https://voms.fnal.gov:8443/voms/fermilab/services/VOMSAdmin, it would be https://voms.fnal.gov:8443/voms/VONAME/services/VOMSAdmin

4) As mentioned in part 1, the lar1 group is deprecated and should be replaced by lar1nd.

5) The "orka" group's entries in the template are all spelled "okra", so that should be corrected.

Please let us know if anything is unclear or you require additional information.
