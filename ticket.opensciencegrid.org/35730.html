    Jan 2, 2018 11:11 PM UTC by letts@.... 
    gftp.t2.ucsd.edu is an alias, behind which sit 6 gftp server machines, gftp-[1-6]. We are getting read errors on all of them, but not all the time. Seems about half of the reads succeed, but to some sites 80% are succeeding (but not much more than that).

We have no significant number of write errors.

There seems to be no correlation with filename or gftp server (i.e. we see errors on all 6 servers, pretty evenly). This makes me think it is a problem reading from hadoop, not a gftp problem or networking problem.

UCSD has also been segfaulting production work (see ticket [1]), but I don’t think this would be related.

It seems that the pool transfer quality started around the beginning of the break [2]. We did have a cooling cut that week, and lost about 10K files. One possible cause of the file read errors is that all of those files are on the bad list. That’s the next thing I will check, but it may take a while since generating a list of failed file names from PhEDEx might be a non-trivial (for me). Time to dust off the data service manual.
