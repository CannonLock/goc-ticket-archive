    Hi,

I'm having a problem calculating the size of a space reservation and
expanding it to fill the linkgroup. The reservation is ATLASDATADISK, id
30000, and it resides in linkgroup atlasdatadisk-link-group.

The reservation reports it has size 20408387809796 (18.5T).  It is the
only reservation in this linkgroup. The linkgroup reports it has only
15329058981778 (14T) reserved.  I've checked the postgres srmdcache db
and it agrees with these numbers.

We've restarted all dcache services, but that hasn't resolved the problem.

Right now this isn't a blocking production, but it will be an issue
later when we have to juggle different reservations.  Is this a known
problem?  Any idea which set of numbers is correct?

- --Sarah

Hi Sarah,

Ahhh... I see now.

1) the resuilt of the query returning the zeroes for "check" is a good
thing, meaning we do not have an inconsistency here

2) Space reservation has size it is 20408387809796 bytes in your case

"Reserved space" in linkgroup is :
sizeinbytes - usedsizeinbytes
of this space reservation.

So, 15329058981778 = 20408387809796 - usedspace in this space
reservation

"Reserved space" of linkgroup is not a useful (and poorly named)
number.

"Reservable space" of linkgroup is a useful number (it tells you
how many more reservations of you can create)

Dmitr

Also:

if you check "srmspace" table for this reservation:

select sizeinbytes, usedspaceinbytes, allocatedspaceinbytes from srmspace
where id=30000;

then

sizeinbytes should be 20408387809796

allocatedspaceinbytes should be 0 (if no one is currently writing a file
inti the system)

usedspaceinbytes = 20408387809796 - 15329058981778

Does it ?

Dmitry
