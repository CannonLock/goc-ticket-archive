    Apr 21, 2009 04:42 PM UTC by Chris Green 
    Hi,

Yep, if you initiate a --append-verify sync from the modern side it will
end up as a --inplace transfer when it finds out the older rsync can't cope.

According to my tests, you can NFS export (eg)
/var/spool/torque/server_priv/accounting read-only with root squashed
even if server_priv is 700 provided that accounting/ is 755. That way
you shouldn't need to export the filesystem with no_root_squash. I
suggest you do your own tests, though.

Let me know what you decide to do.

Chris.

    Apr 21, 2009 04:09 AM UTC by Chris Green 
    Hi Horst,

--append-verify is a feature of more recent versions of rsync. It's to
be preferred if available but even the basic --append much to be
preferred over "standard" rsync operation. Read-only NFS is least
problematic, otherwise you may want to check for the presence of the PBS
probe lock file in gratia/var/lock (I think) with the lockfile utility
(part of the procmail package, historically) to lock out the probe while
you're syncing.

Just a suggestion -- do what you're comfortable with.

Thanks,
Chris.

    Apr 20, 2009 10:15 PM UTC by Chris Green 
    Hi Horst,

It would be perfectly acceptable to sync the logs every hour. We have
*some* protections against problems that might be caused by reading
while the log is being updated. It would be much safer though, if you
were to rsync additively only (I believe there's an option to only
update the bits of the file that have changed rather than re-writing the
whole file --append-verify). If the file were being re-written from
scratch as the probe run kicked off there might be information loss.

Thanks,
Chris.
