By chance, I found there were a lot of duplicates ldapsearch endpoint for OSG site in http://lcg-bdii-conf.cern.ch/bdii-conf/bdii.conf, for example for BNL there are lines:

OSG_BNL-ATLAS_BNL_ATLAS_1 ldap://is.grid.iu.edu:2180/mds-vo-name=BNL-ATLAS,o=grid
OSG_BNL-ATLAS_BNL_ATLAS_2 ldap://is.grid.iu.edu:2180/mds-vo-name=BNL-ATLAS,o=grid
OSG_BNL-ATLAS_BNL_ATLAS_3 ldap://is.grid.iu.edu:2180/mds-vo-name=BNL-ATLAS,o=grid
OSG_BNL-ATLAS_BNL_ATLAS_4 ldap://is.grid.iu.edu:2180/mds-vo-name=BNL-ATLAS,o=grid
OSG_BNL-ATLAS_BNL_ATLAS_5 ldap://is.grid.iu.edu:2180/mds-vo-name=BNL-ATLAS,o=grid
OSG_BNL-ATLAS_BNL_ATLAS_SE ldap://is.grid.iu.edu:2180/mds-vo-name=BNL-ATLAS,o=grid

As you saw, the host and binding part are the same for all the entries, thus the bdii-update process of top BDII will fork one ldapsearch process for each entries, but just to get the same duplicate information. It definitely would overload OSG info system. There are similar issues for other OSG sites too. And other top BDII would get timeout very easily because of this.
