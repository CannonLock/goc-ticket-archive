without a 302 redirect, <http://ca.nordugrid.org/cacrl.crl>;. I believe
that it is no longer a problem.

===== USC CA =====
Problem URL:
<http://www.usc.edu/isd/services/authx/CA/USC-PKI-Lite-CA.crl>; redirects
to <http://www.usc.edu/its/services/authx/CA/USC-PKI-Lite-CA.crl>;.

    
The CA web page, <http://www.usc.edu/isd/services/authx/CA/>; shows the
second URL (the one with ITS) on it. I'm not sure why we have the first
URL (with isd) in it: I assume they must have changed.

I have updated the CRL URL in our CA-Certificates package, and the next
time we do a release and you do an update, you'll get the new CRL. We
plan to do a release within about a week to integrate a new IGTF CA
distribution.

===== ESnet Root CA =====
Problem URL: http://es.net/CA/d1b603c3/CRL/d1b603c3.r0

This redirects to <http://www.es.net/CA/d1b603c3/CRL/d1b603c3.r0>;, which
seems like it should be a safe replacement. The fact that their web page
lists the second URL (with www) in it suggests that it's a safe
replacement.

However, they are part of the IGTF, which distributes CA files. I would
rather not modify anything that comes from the IGTF. So I have contacted
the IGTF and ESnet and requested that in the next IGTF release, they
update the CRL.

===== Summary =====
At this point, I think the only remaining task is to wait for the IGTF
to update their distribution and include it in the VDT. I think it's
very likely that they will do so. I'll let you know what they tell me.

Thanks,

    Jun 13, 2006 04:23 PM UTC by Footprints Remedy 
    The Fermi/Remedy ticket has been updated.

New Work Log Entry : This information did not make it into the VDT ticket, so I am resubmitting.

The issue: the HTTP standard declares that the 302 page is cachable only if it has an Expires: or Cache-Control: field.  If the redirects are supposed to be permanent, it should return a 301 and the crl_url files should be updated (by the CA, I assume).  Otherwise, I'd request that they add the Expires: field so that we can cache the redirection page.

It may be as simple as adding the following lines in the section with their Rewrite Rules if they are using apache:

ExpiresActive On
ExpiresDefault A7200

    The Fermi/Remedy ticket has been updated.

New Work Log Entry : It's not a 404, it's a 302.  That means that it is a temporary redirect -- the URL it points to is not guaranteed to be permanent.

The issue: the HTTP standard declares that the 302 page is cachable only if it has an Expires: or Cache-Control: field.  If the redirects are supposed to be permanent, it should return a 301 and the crl_url files should be updated (by the CA, I assume).  Otherwise, I'd request that they add the Expires: field so that we can cache the redirection page.

It may be as simple as adding the following lines in the section with their Rewrite Rules if they are using apache:

ExpiresActive On
ExpiresDefault A7200
