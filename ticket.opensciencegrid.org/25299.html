    Hi Burt,

Perhaps we're talking at cross-purposes here.

I believe there's two distinct things: BDII and GLUE.

BDII is software that some (most? all?) sites run to provide the LDAP endpoint that may be queried to discover information about their site, such as the services they run.  Typically, top-level BDII services contact site-level BDII services to achieve an aggregate view of currently available services.

GLUE is the information model whereby sites describe their available services in a common information model.  This is usually expressed in LDAP, although other formats ("bindings") exist.

I believe OSG is decommissioning (or already has decommissioned) their site-level BDII in favour of some internal information system.  There is a bridge service that acts like a site-level BDII but delivers information about all OSG sites.  Top-level BDII services can contact this OSG-bridge-service to discover all OSG services.

However, information about OSG services and sites is still available as GLUE information.

Does this match what USCMS is planning?

Cheers,

Paul.

    Jun 2, 2015 09:24 PM UTC by OSG-GOC 
    Hi Elizabeth,

I understand your (and Mat's) concern about breaking something.  I think there is little risk as other (non-OSG) sites are publishing "mailto:" URIs without a space after the colon.  You can verify this visually with the command:

ldapsearch -LLL -x -H ldap://lcg-bdii.cern.ch:2170 -b o=grid '(objectClass=GlueSite)' GlueSiteSecurityContact|grep ^Glue

The specific problem for me was when I wanted to contact the ~20 sites that are running a version of dCache that is about to run out of support.  I wanted to make sure they realise the full implications of not upgrading.  With GLUE, this is relatively easy to do.  However, because of the badly formatted URIs (and other problems), I needed work-arounds to reformat the badly published contact URIs.

The deeper issue here is that, unless the problem is fixed, all subsequent attempts to use these GLUE values will suffer similar problems.  This will require anyone using GLUE information to maintain a set of brittle, undocumented set of work-arounds for the various "peccadillos" of the sites (or grids) that publish information.

Also, as a courtesy to whoever wants to use this information next, I generally opened GGUS tickets to report GLUE problems as I find them -- generally sites fix such problems in a timely fashion.

I imagine OSG publishes GLUE information so that people can make use of it.  Publishing badly formatted information makes this hard.  I would guess this problem is pretty trivial to fix but, by not fixing it, users may become frustrated with GLUE and start developing some alternative ("more reliable") information systems.  This would waste effort by duplicating what already exists.

HTH,

Paul.

    May 7, 2015 11:21 AM UTC by OSG-GOC 
    Hi,

The "GlueSite" object describes a grid-site and includes contact information, like site's sys-admin contact, their security contact, user support contact, and some generic email contact.

Currently SPRACE is publishing some broken email values. You can verify this with the following command:

paul@celebrimbor:~$ ldapsearch -LLL -x -H ldap://lcg-bdii.cern.ch:2170 -b o=grid GlueSiteUniqueID=SPRACE
dn: GlueSiteUniqueID=SPRACE,Mds-Vo-name=sprace,Mds-Vo-name=local,o=grid
GlueSiteWeb: http://osg-ce.sprace.org.br
GlueSiteSecurityContact: mailto: jadir.marra.da.silva@....
GlueSiteDescription: OSG Site
GlueSiteSponsor: cms:100
objectClass: GlueKey
objectClass: GlueSchemaVersion
objectClass: GlueSite
objectClass: GlueTop
GlueSiteSysAdminContact: mailto: jadir.marra.da.silva@....
GlueSiteName: SPRACE
GlueSiteUniqueID: SPRACE
GlueSchemaVersionMinor: 3
GlueSiteLongitude: -46.66769
GlueSiteLatitude: -23.511344
GlueForeignKey: GlueSiteUniqueID=SPRACE
GlueSiteEmailContact: mailto: jadir.marra.da.silva@....
GlueSchemaVersionMajor: 1
GlueSiteUserSupportContact: mailto: jadir.marra.da.silva@....
GlueSiteLocation: Sao Paulo, Brazil

paul@celebrimbor:~$

Notice that GlueSiteSecurityContact, GlueSiteSysAdminContact, GlueSiteEmailContact, and GlueSiteUserSupportContact have values that start "mailto: ", with a space after the colon.  These values should be valid URIs, without the space after the colon; for example "mailto:jadir.marra.da.silva@....".
