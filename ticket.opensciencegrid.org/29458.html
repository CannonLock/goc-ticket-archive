    Hi,
The object is more than 72 MB.  It is 95800205 bytes.  It was read from your squid disk (or possibly the system disk cache) and served in 3.051 seconds.   The record looks normal.  What does not look normal is setoption("maximum_object_size_in_memory", "72 MB").  This is not what you think it is.  On https://twiki.cern.ch/twiki/bin/view/Frontier/MyOwnSquid  the recommended value is 128 KB.  The reason is quoted on that same page:

The default cache_mem is only 8 MB and is probably a very old default and should be increased some, but we have found that squid performs better for large objects out of the disk cache than the memory cache. We recommend at most 1/8 of the physical RAM and no more than 128 MB, leaving a lot of memory for disk buffering.

GOCTX Source: https://ggus.eu/ws/ticket_info.php?ticket=120922
    Apr 25, 2016 02:21 AM UTC by OSG-GOC 
    Please also see https://twiki.cern.ch/twiki/bin/view/Frontier/InstallSquid#Running_multiple_squid_processes to run multiple squid processes with frontier-squid-2.  This is usually only helpful when the network is greater than 2gbit/s.  It is not a feature of squid-2, but frontier-squid-2 has a wrapper script that handles it.

squid-3 is not strictly multithreaded, it is also multiprocess just as frontier-squid-2 is although it is builtin and better integrated.  There is one remaining bug in squid-3 that is blocking us from being able to use it for frontier, however; the details are on the https://twiki.cern.ch/twiki/bin/view/Frontier/MyOwnSquid page as Barry pointed out.  We're now contracting with a consultant to hopefully get it fixed in the next couple of months.

Disk drive speed usually has little do with squid application performance, because the frontier & cvmfs hit rates are so high that it almost always has all of the data in filesystem buffer cache (in memory).  However the disks are exercised a lot for logs on high-traffic machines.

    Apr 23, 2016 12:29 PM UTC by OSG-GOC 
    Hello,
Please see  https://twiki.cern.ch/twiki/bin/view/Frontier/MyOwnSquid
for the status of squid-3.x

One short term possibility is that your site-local-config.xml currently has

<proxy url="http://10.3.11.62:3128"/>
<proxy url="http://10.3.255.184:3128"/>

which uses the first squid as a primary and the second as a backup.
If you add one line:

<load balance="proxies"/>
<proxy url="http://10.3.11.62:3128"/>
<proxy url="http://10.3.255.184:3128"/>

It will try to use the two of them equally and spread the load.
The usefulness of this depends on whether the two machines are comparable in power.
