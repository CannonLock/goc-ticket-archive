    Hello Anand,

I have added the attachments using the green button on the ticket web page. I guess attaching them to the email message was not sufficient to have them attached to the ticket.

You asked, "when you that your VO users run with their own credentials with glidein jobs, is it because they are required to use glexec by the VO? Do you have any controls in place to make sure that your users are following your recomendation?"

When I set up a user and grant her access to my submit host, I give her a standard condor submit file which she is to use for all job submissions. Certain lines are marked as modifiable by the user (executable, input, output, error, transfer_input_files, etc.) but others are marked as not to be modified. Among those not to be modified is the line x509userproxy that points to the user's local voms proxy in the /tmp filesystem. If this does not exist or is out of date, I tell them, the jobs may fail. So far this has seemed to work ok for our users. Each job ships out to the pilot execute directory with a copy of the x509proxy, and if I happen to check and see that this file is expired or invalid, I may purge the jobs from the queue. So far I have not actually seen this problem, so I have not purged any user jobs for this reason.

    Dear VO Managers,

The OSG security team is conducting a drill/audit of OSG VOs to check if VO users are actually signing Acceptable User Policy (AUP). As we mentioned in an email sometime ago OSG VOs are expected to ensure that they have an AUP and that their users are made aware of it periodically. Specifically, three conditions were outlined

1. If VO users have a certificate that is registered in its VOMS server, then the associated VOMS Admin interface or another equivalent system should have a valid VO AUP and the users are expected to read and accept the AUP atleast once a year, failing which the user much be suspended from the VO. The acceptance of AUP by the user needs to be recorded.
2. If the VO uses a pilot job framework (e.g. glideinWMS), and its users have ssh access to a glidein frontend (or equivalent) from where they submit jobs to OSG, then the login to the frontend system should display the usage conditions and provide a link to the VO AUP. Since it would be difficult to record acceptance during every login, a message displayed immediately after login should indicate that if users don't accept the AUP they should immediately exit and not submit jobs to OSG.
3. If VO uses a Science Gateway (or equivalent) system, then users of such gateway should be asked to accept the VO AUP during registration, and a link to the AUP should be made available during login. The acceptance of AUP by the user needs to be recorded during their registration, and text indicating that they accept AUP by logging in to the system should be displayed on the login page.
