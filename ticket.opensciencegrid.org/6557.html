    Scot,

There is no change in the behavior with the update from 1.5 to 1.6
certificates

We update certificates into an afs RW directory.  This is replicated to
RO copies, mounted at a different mount point, that are the ones
actually used by all of our working machines.  However, the link that
was created for this update is "absolute" in the sense that it is a full
path, and points into the RW directory, not the RO path, like so:

[gate02:globus]# cd /afs/atlas.umich.edu/Certficates/

[gate02:Certficates]# ll
total 136
lrwxr-xr-x  1 root    49 May  9 00:12 certificates ->
/afs/atlas.umich.edu/Certficates/certificates-1.6/

Certficates is the RW mount point, so even though the directory is
properly replicated into the RO mount points, the "certificates" link
re-directs back to the RW mount point, and that is incorrect for us.

This is the behavior that was, apparently, introduced back around the
1.4 or 1.3 release, and which we request to be changed.  The link that
is created should look, instead, like this:

[gate02:Certficates]# ll
total 136
lrwxr-xr-x  1 root    16 May 11 09:16 certificates -> certificates-1.6/

This, in my mind, is the meaning of a "relative" link.  The first link
above is "absolute" in my mind because it completely specifies the path.

Can this behavior be changed as I request?


    bob

    Mar 13, 2009 07:31 PM UTC by Christopher Pipes 
    We distribute certificates out of a central location in afs, replicating the RW
volume to differently-named RO volumes for use throughout our distributed system.

In some recent release of the certificates distribution, what used to be a relative
link was changed to be an absolute path link, and this has (once again) broken our
certificate distribution here.

For example, the RW path is
/afs/atlas.umich.edu/Certficates
and within this directory, it used to be that the link named "certificates" would
point to, for example, "certificates-1.4".

Now, this link "certificates" points to the absolute path
"/afs/atlas.umich.edu/Certificates/certificates-1.4" and this defeats the entire
purpose of having RO volumes as the RO mount points are at
"/afs/atlas.umich.edu/OSG_certificates".

Can this be repaired?  Does it HAVE to be such an absolute link where the previous
relative link was much more flexible?

Thanks,
bob
