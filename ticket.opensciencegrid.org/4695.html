Forwarded message:
Hello GOC,

FNAL_GPFARM is advertising some very broken data. It appears that the problematic
data appeared when they started running a version of the GIP which attempts to
include information about a GT4 container. It broke my GIP recording for several
days while I was out.

1) This was not coordinated with the OSG-GIP team to my knowledge (or, at least,
there was no emails about it...).
- This broke GIP monitoring for several days while I was on vacation.
- If it was at least informally announced that new breaking attributes were going
to be added, I could have caught the problem much sooner.
2) This unnecessarily breaks the GLUE schema, as there are already places for the
added data (set GRAMVersion=4, ImplementationName=WSGRAM,
ImplementationVersion=4.0.5, Info.ContactString=fngp-osg.fnal.gov:9443 Condor).

If there are places in the existing schema to add the data - without even abusing the
underlying schema! - I would prefer those.

3) FNAL_GPFARM now has funky data in the BDII:

bbockelm:trunk brian$ ldapsearch -h is.grid.iu.edu -p 2170 -xLLL -b
'GlueCEUniqueID=fngp-osg.fnal.gov:2119/jobmanager-condor-cigi,mds-vo-name=FNAL_GPFARM,mds-vo-name=local,o=grid'
dn: GlueCEUniqueID=fngp-osg.fnal.gov:2119/jobmanager-condor-cigi,mds-vo-name=F
NAL_GPFARM,mds-vo-name=local,o=grid
objectClass: GlueCETop
objectClass: GlueCE
objectClass: GlueSchemaVersion

    objectClass: GlueCEAccessControlBase
objectClass: GlueCEInfo
objectClass: GlueCEPolicy
objectClass: GlueCEState
objectClass: GlueInformationService
objectClass: GlueKey
GlueCEHostingCluster: fngp-osg.fnal.gov
GlueCEName: cigi

There should be a whole lot more in that entry.

So, I would appreciate it if the GOC:

1) Cleaned out the entry above with the bad data from the production BDII.
2) Kindly ask FNAL_GPFARM to coordinate any schema breaks a bit more widely, or at
least pass them through the OSG-GIP team.
