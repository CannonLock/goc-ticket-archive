    John,

Late yesterday I found the problem.

In 3.1.19 a change was made in condor_meter to verify that PER_JOB_HISTORY_DIR was defined.
It uses "condor_config_val" to verify this value has been set in the condor configuration files

Unfortunately, if a site (such as MWT2_CE_UIUC) is using a non-standard installation location for condor,
the condor_config_val will fail due to the fact that $CONDOR_CONFIG is not defined and the master
condor configuration files cannot be found.

In /etc/osg/config/20-condor.ini, there are two definitions for non-standard condor installs,

condor_location = /opt/condor
condor_config = /opt/condor/etc/condor_config

osg-configure does put the condor_location value in ProbeConfig (as CondorLocation)
However condor_config is ignored and no definition is placed in ProbeConfig.
Also condor_meter does not check nor attempt to use any such definition for condor_config

As a result, even though condor_meter can find the location of condor_config_val, the location of the
master condor config is not defined nor known, condor_config_val then fails and the gratia records are not uploaded.

As a "workaround", I have added the definition for $CONDOR_CONFIG in the cron file, /etc/cron.d/gratia-probe-condor.cron

[root@mwt2-gk ~]# more /etc/cron.d/gratia-probe-condor.cron
CONDOR_CONFIG=/opt/condor/etc/condor_config

0,15,30,45 * * * * root /usr/share/gratia/common/cron_check /etc/gratia/condor/ProbeConfig && /usr/share/gratia/condor/condor_meter
-s 900

On a site with condor installed in a standard location (/usr/bin, /etc), no variables need be defined. Thus the CE for MWT2_CE_UC
and MWT2_CE_IU
properly uploaded their gratia records. The MWT2_CE_UIUC is part of the Illinois Campus Cluster. We are required to install condor
on a common

    GPFS file system and not on the nodes themselves. This is why only this site has the problem with a non-standard condor installation.

I believe Suchandra is going to bring this up within your group to find a general solution?
Perhaps it is a simple as osg-configure putting the value for $CONDOR_CONFIG as defined in 20-condor.ini into the cron file?

I suspect this ticket can be closed, at least at my end. I assume someone will file an internal jira ticket?
