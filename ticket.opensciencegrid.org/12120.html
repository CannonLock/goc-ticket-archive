    Keith,

The issue is not so much caching lifetime, but that we are using wget in two different ways: (1) to actually fetch files, and (2) to get request permission from a central manager for privs to carry out certain site-wide changes to our code repository on the site's $OSG_APP area.  We don't want multiple workers on a site stepping on each other trying to install a package that is needed for their job.  These queries are routed to a thread running in our Apache server (mod-perl registry) and do not go to disk files on our web server.  It looks to me like what I should do is use the "Expires : <date>" entity-header field in the HTTP response returned by my web server.  That will tag each thing it returns with a lifetime stamp beyond which the squid will automatically request a fresh copy from my server.  I can customize each type of response with an appropriate lifetime, and then have all of my queries go through the local site squid.  Does this sound reasonable?

A while ago someone mentioned that you might be reluctant to use the site squid
due to worries about overly long caching of results.  To control this, you can implement
a .htaccess control file on your web server.  The following URL gives a reasonable

GOCTX Source: https://fermi.service-now.com/nav_to.do?uri=incident.do?sys_id=INC000000251285
    May 17, 2012 06:16 PM UTC by OSG-GOC 
    There is a site-independent way to discover the squid, namely use the variable OSG_SQUID_LOCATION
which is set in the environment of every job.  If that OSG_SQUID_LOCATION is not defined then you presume the site does not have a squid.

Running your own private squid is frowned upon.

You should also note that Brian Bockelman, part of the OSG Technology group, is currently working on a document with a set of requirements
for centralized code distribution on the OSG which might be an alternative to what you are doing now, you should contact him.
