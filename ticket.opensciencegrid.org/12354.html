    Hi, this is Tim Cartwright from the OSG Software team.  There seems to be a lot of information in this ticket already, but it is hard to get a clear picture of the current state.  I will try to summarize what I see.

It seems that the core problem is that Engage glidein jobs are failing 100% of the time.  Just making sure: This is still the problem, right?

Now, there was a lot of discussion about the state of the certificates on brgw1, which appears to be the gatekeeper for the glideins.  From what I can tell, it looks like things are OK with the certificate install there.  That is:

* The certificates are actually installed in /opt/osg/1.2.13/osg-1.2.13/globus/TRUSTED_CA
* There is a symlink from /etc/grid-security/certificates to that directory
* The certificates directory contains a symlink at d1b603c3.signing_policy
* The symlink points to ESnet.signing_policy, which does also exist

So from the profiler output, at least, it seems that this particular file is OK.

Also, according to the profiler output, fetch-crl itself seems to be installed, enabled, and running.

Alain had me look at the RSV information on MyOSG, and it does have one outstanding issue: The certificates themselves are not up to date.  This is puzzling, since it does appear that vdt-update-certs is installed, enabled, and running.  The one potential clue in the profiler output is this:

vdt-update-certs info:
Last run: 2012-07-18T08-47-02
WARNING: last update time unknown
Status: Installed and running

That warning is a concern.  vdt-update-certs writes to a log file, which should be located at "/opt/osg/1.2.13/osg-1.2.13/vdt/logs/vdt-update-certs.log".  Could you please look for that file and, if available, send it into the ticket?
