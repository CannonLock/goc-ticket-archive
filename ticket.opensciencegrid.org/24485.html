To add some hopefully clarifying information:

There are currently 3 methods of using cvmfs:

1) Repository hosted at the OSG GOC.  (This is the case for this ticket).
In this case, the authorized library/code manager needs to log in to a
system at the GOC to make a release.  For this case, a cvmfs client node
will see the release located at, for example,
/cvmfs/oasis.opensciencegrid.org/lbne

2) Repository hosted at Fermilab and distributed by Fermilab.  This is
a deprecated (though functional) method that utilizes a set of machines
to host repositories and the cvmfs stratum-0/stratum-1 servers.  These
releases were (intentionally) not made available externally to Fermilab.
If one uses this method, the repository will be located on a Fermilab
node with certain users having authorized (via entry in local user "cvmfs"
.k5login file) access.  LBNE had a test node, lbnecfs.fnal.gov, which has
not been used for production.  If using this case, a cvmfs client node
would see the release located at, for example,
/cvmfs/lbne.fnal.gov

3) Repository hosted at Fermilab, signed and distributed by OSG (OSG "external"
repository).  This is the preferred method for Fermilab-based VOs.  The
repository will be on a Fermilab node, e.g. oasiscfs02.fnal.gov, with a user
name associated with the VO and access controlled by the .k5login file.
The cvmfs client will see the release located at, for example,
/cvmfs/lbne.opensciencegrid.org

To repeat, Case 1 is the one referenced by this ticket, as the active LBNE
repository is at the OSG GOC.
