    All indicators suggest VOMS is running properly on port 15004 and that this is *not* blocked by the firewall.

Furthermore, I've discovered that there are 3 different pointers to the VOMS client configuration information:

1. According VDT documentation:

http://vdt.cs.wisc.edu/VOMS-documentation.html

the VOMS client looks for configuration in the following in turn:

    * /etc/vomses
    * $X509_VOMS_DIR evnironment variable
    * ~/.edg/vomses

Of these, only the second is defined, and set to:

setup.sh:X509_VOMS_DIR="/opt/vdt/glite/vomsdir"

This directory exists, but is empty.  I dutifully created a file called "SBGrid" and put in it the required values, and am left wondering how my voms-proxy-client was even able to *guess* at what VOMS server to contact.

2. My VOMS installation:

https://hpc.sbgrid.org:8443/voms/SBGrid/webui/config

suggests I should be looking for a path more along the lines of:

/opt/glite/etc/vomses/SBGrid-hpc.sbgrid.org

I don't have /opt/glite, but I do have /opt/vdt/glite, however when I look there I find that etc/vomses is a file, not a directory.

3. So, the final answer is that the VOMS config file is $VDT_LOCATION/glite/etc/vomses (a file).  Helpfully, this already has an SBGrid entry, answering the mystery of where voms-proxy-init got its information from.  Unhelpfully it doesn't give me many clues as to why voms-proxy-init is failing to connect to my VOMS server which seems to be running in the expected place.  Furthermore, the OSG (and VOMS) documentation does not seem to be correct regarding where to find/set this information in the client.
