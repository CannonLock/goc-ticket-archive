    Dear Aurilien

In our site forwarding gateway, but also in some other OSG sites, the
executable is not local to the worker node but it is transferred from
the head node to the final destination when you do a globus-job-run.
The best way to get around this is to submit a shell script.  That
way you are guaranteed to get the utilities that are at the destination.
Try this one, for instance.

[root@fnpc3x1 log]# cat /home/timm/gridsleep.sh
#!/bin/bash
#
echo "Checking the environment"
/usr/bin/printenv
echo "sourcing the setup"
. ${OSG_GRID}/setup.sh
echo "dumping my proxy"
grid-proxy-info -all
echo "output of pwd"
/bin/pwd
echo "output of hostname"
/bin/hostname
echo "user id"
/usr/bin/id
/bin/sleep 7200
---------------------------

I am not very familiar with the WLCG job submission mechanism, but I know
there are people from other WLCG VO's who do use it, and are able
to run on FermiGrid.  So there must be a way.  since you are a d0
collaborator we will be glad to help you as much as necessary to get
this working.  The shell script is the safest way. And there are a lot
of opportuntistic cycles on FermiGrid that are open to D0.

Steve Timm

    Aurielien

Our site gateway fermigridosg1.fnal.gov runs Scientific Linux 5 but
most of our worker nodes still run Scientific Linux 4.  So whenever
you do a globus-job-run fermigridosg1/jobmanager-condor
it will try to run the SL5 executables on SL4 nodes and fail.

Compile your executable on SL4 nodes and submit it using condor-G submit
scripts and it will work fine.  Alternatively there are
ways to select the few SL5 nodes we do have, I can give you information
if you are interested.

Steve Timm
