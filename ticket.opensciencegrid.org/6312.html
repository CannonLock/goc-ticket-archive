    Feb 10, 2009 06:40 PM UTC by echism 
    Jay,

Do you have any thoughts on this?

We are chasing a problem where our GUMS server has higher loads once per hour, with
major peaks every 3 hours. This GUMS server is primarily for our OSG 1.0 gatekeeper,
gate01.aglt2.org. The problem is that gums load peaks result (apparently) in
authorization errors, and in turn these cause "lost heartbeat" failures for some
ATLAS jobs.

Errors I see in the Condor-G pilot logs are the following:

012 (17689542.000.000) 02/02 12:01:02 Job was held.
Globus error 10: data transfer to the server failed
Code 2 Subcode 10
At the same time as this batch of lost hearbeats this morning, an rsv probe also
failed, with the following error:

timestamp: 2009-02-02T17:01:00Z
metricStatus: CRITICAL
serviceType: OSG-CE-General
serviceURI: gate01.aglt2.org
gatheredAt: gate01.aglt2.org
summaryData: CRITICAL
detailsData: Job submission to "gate01.aglt2.org/jobmanager-fork" failed in
OSG_RSV_Probe_Bases::Set_Remote_Env () with error: "GRAM Job submission
failed because data transfer to the server failed (error code 10)"; Bailing now ...

When the gatekeeper is examined, it is also having load peaks (much less obvious) at
the same, 1hr intervals.

Do you have any clues at all about what may be going on here? Is there some periodic
process within the OSG 1.0 suite that could be contributing here?

Any thoughts or pointers at all will be very helpful for us.
