Hi John

Thank you for all the effort you have put into this.  It is a mystery as to why it is not working.  I have two slightly random suggestions that you may already have tried.

1) Can you check the cache.log files (probably in the same directory as the access.log file /var/log/squid/ or whatever you set it to).  There is a change there is a useful error message in there.

2) Some sites, for reasons that weren't understood found adding:
acl SNMPHOSTS src 128.142.0.0/16 localhost
to the config made it work.

If neither of these work, I suggest we put the ticket on hold until you have setup your new squid (we can grey out your site on the monitoring so you don't get accidental tickets).  I know you have also been in contact with John de Stefano, he may have some ideas to try as well.

    Hi John

I see no reason why a host key would cause a problem.  Can I just double check that the squid is definitely being used successfully by jobs?

If I do:
[squidmon@vocms168 ~]$ squidsnmpwalk heroatlas.fas.harvard.edu:3401
Timeout: No Response from heroatlas.fas.harvard.edu:3401
which isn't terribly useful in narrowing down the problem.

There should be something listening on port 3401 (assuming you haven't changed the default listening port). On a squid at RAL I get:
# netstat  -tulpn | grep :3401
udp        0      0 0.0.0.0:3401                0.0.0.0:*                               11843/(squid)

I notice you are running a slightly older version of the frontier squid package.  Maybe the easiest thing would be to upgrade that and see if it fixes things.  The latest stable version is:
frontier-squid-2.7.STABLE9-16
You can find the instructions here:
https://twiki.cern.ch/twiki/bin/view/Frontier/InstallSquid
and the ATLAS specific tweaks here:
https://twiki.cern.ch/twiki/bin/view/Atlas/T2SquidDeployment

Alastair
