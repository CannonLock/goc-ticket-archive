Brian Bockelman wrote:

On May 15, 2009, at 12:00 PM, Ted Hesselroth wrote:

Regarding the timeouts, do you have

cache-lifetime=180

in dcachesrm-gplazma.policy? The default is 0, which is to have caching turned off. With caching on, there may
still be some timeouts users are first authorized. The socket connection for the Axis callout times out at 30
seconds. That was hard-coded as the upper limit for the settable gPlazmaRequestTimeout, which also has a
default of 30 seconds. gPlazma was timed at 7 authorizations per second with a nearby GUMS server at the
time, so 30 seconds seemed like plenty.

Out of curiosity:  The stated goal for CMS for 2009 is 50Hz from an SRM endpoint.  Is this 7 auths per second
the limit (and if so, is it a per-component limit or a site-wide limit)?  Certainly this seems slow.

Brian

We didn't get timeouts between Fermilab and BNL during development and testing, but that was not a
production environment. If with the new GUMS server the timeout is not long enough, I can increase it, though
it will take a new dCache release for it to be seen.

I'd also want to check from the production logs to be sure that the timeout is not in the cell messaging. The

    timeout there is hard-coded at 60 seconds, on the assumption that if the process takes any longer then
"something's wrong". That could also be changed if it were determined that a cell message overhead of around
30 seconds was reasonable.

Of course, if we see authorization timeouts on an srm which is calling gPlazma modules, then we know it is in the
Axis call. I can check if the setup works.

Ted
