Unfortunately, this is never safe to do. But let me be more precise:
In general, Pacman can install software from multiple caches. But if
you install software from two different VDT software caches, you'll
get all sorts of weird problems.

>Here is a the beginning of the pacman command history which shows this
>clearly (though I don't recall why there were multiple attempts to
>update the PPDG-Cert-Scripts package):
>
>[root@stargrid02 var]# more o..pacman..o/logs/commands
>% /opt/pacman-3.20/bin/pacman -get OSG:ce finished at Mon Mar 19
>19:09:03 2007
>% /opt/pacman-3.20/src/pacman -get OSG:Globus-Condor-Setup finished at
>Mon Mar 19 19:14:55 2007
>% /opt/pacman-3.20/src/pacman -get OSG:ManagedFork finished at Mon Mar
>19 19:16:23 2007
>% /opt/pacman-3.20/src/pacman -update VDT::PPDG-Cert-Scripts finished at
>Tue Mar 20 17:48:28 2007
>% /opt/pacman-3.20/src/pacman -update VDT::PPDG-Cert-Scripts finished at
>Tue Mar 20 17:50:19 2007
>% /opt/pacman-3.20/src/pacman -update VDT::PPDG-Cert-Scripts finished at
>Tue Mar 20 17:52:30 2007
><snip>

Offhand, I'm not sure if you actually installed from multiple VDT
caches or not. It depends on what version of the VDT was installed by
the OSG cache on March 19th. I'm just not sure. The version that
"VDT" points to changes over time. I never recommend using it.

    
I *think* that you only installed from one VDT software cache, so
this might not be the problem.

Is this problem really from last March?

>What is the best way to clean this up?

If you installed from multiple VDT software caches, there is no good
way to clean up.

Can you send me your vdt-install.log file, and I'll look more carefully?

Thanks,
-alain
