    Hi Tim,
it is the first time, as far as I know, that we try installing CEMon on
a VM: it is an interesting use case. Did you get the wrong hostname from
the log file of the consumer?

I've looked at the code of the consumer (i.e. the information gatherer
code... it must be very similar to the C clients that you use for BDII):
it seems that the consumer subscription interface (...called by CEMon
when it comes up...) gets the CEMon host as input and builds the CEMon
url ( URL address= new
URL("https",host,8443,"/ce-monitor/services/CEMonitor"); )

When starting up, CEMon subscribes to the consumer using information in
the cemon config file:
$VDT_LOCATION/glite/etc/glite-ce-monitor/cemonitor-config.xml

For example:

<subscription id="subscription-http___is_grid_iu_edu_14001-OSG_CE-RAW"
monitorConsumerURL="http://is.grid.iu.edu:14001";
sslprotocol="SSLv3"
retryCount="-1">
<topic name="OSG_CE">
<dialect name="RAW" />
</topic>
<policy rate="300000">
</policy>
</subscription>

It is not clear to me what function CEMon uses to get its host name and
pass it to the consumer subscription interface. If you want, I can ask
the details to the CEMon developers in Italy. If it is something like
(...which is pretty common...):

InetAddress.getLocalHost().getHostName();

    
this gets the IP address of the local host, then the corresponding
hostname. The IP and local hosts are resolved using the standard OS
local configuration and DNS/NIS. Are you sure these are correct?

I've tried to give more details on what system call the java function
uses, but the java documentation on this is pretty vague: see "Host Name
resolution" at
http://java.sun.com/j2se/1.5.0/docs/api/java/net/InetAddress.html

I've also tried to resolve the hostnames in the ticket, but I cannot
even ping them: maybe the administrators are working on the machine.

Please, keep in the loop.
Cheers
Gabriele

Normally I would add this Resource to monitoring, but it fails to pass the authentication. We would

To resolve this issue, please be sure the MIS VO is supported. If you're using a grid-mapfile and not
GUMS for authentication you set this in the $VDT_LOCATION/edg/etc/edg-mkgridmap.conf file by
removing the commenting hashes from the lines in the file for those VOMS and VOs you wish to
support, namely, the MIS VO, which runs the monitoring jobs. And then make local accounts for
those VO's, 'mis' in this case. Any VO you wish to support will require a local account that is
mapped to its VO. If you're using GUMS, a separate authentication mechanism, and you need
assistance, please let us know.
