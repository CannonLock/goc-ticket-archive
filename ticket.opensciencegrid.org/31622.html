    Oct 28, 2016 01:13 PM UTC by Christopher Pipes 
    Sarah,

System level checkpointing is not possible on OSG due to the many slightly different types of hardware making up the system.

If your application has support for application level checkpointing, which is basically writing out state files periodically, that should work. There is no default mechanism for the transferring of those state files, but we have helped users in the past with wrapper script which transfers the files periodically. Let me see if I can find an example of somebody on our team with more experience on this.

    Oct 26, 2016 07:10 PM UTC by Tim Theisen 
    > I was wondering if it was possible to incorporate checkpoints into programs in the vanilla universe in OSG.

Yes, but it depends tremendously on how your program does checkpointing, how long it takes, and what you're trying to avoid.  If you're just worried about preserving progress across scheduling interruptions, and your program can checkpoint on receipt of a signal, and it can do so fast enough (shorter than the eviction-to-kill delay (HTCondor default: 10 minutes)), something like

when_to_transfer_files = on_exit_or_evict
kill_sig = <checkpoint signal>

will probably get the job done.  It will re-run the job using the preserved sandbox, so the program will also have to deal with that without changing its arguments.  (Some can, others require a wrapper script.)

> Alternatively, if there is a simple way to transfer data back to me and keep a job from restarting after an eviction, that would be sufficient for our purposes.

Something like

when_to_transfer_files = on_exit_or_evict
requirements = NumJobStarts == 0
periodic_remove = JobStatus == 1 && NumJobStarts > 0

is as close as HTCondor can come to once-and-only-once; see https://htcondor-wiki.cs.wisc.edu/index.cgi/wiki?p=HowToAvoidJobRestarts for the disclaimers.
