Andrew Hanushevsky wrote:
Hi Sarah,

When you do a peering arrangement, as you noticed, the redirector
gives
both copies equal preference. The notion about a meta-manager is that
you normally ask for files in your local cluster. Only when you can't
find a file there do you go to a meta-manager. The way this
arrangement
is normally used is to trigger a copy of the missing file into your
cluster so that you can locally continue. In your case, I assume you
just want to access it over the WAN. Fabrizio Furano (who I
copied) is
the expert in this area. So, Fabrizio, any way top configure the
client
for a 2-tier access arrangement?

Also, I forgot to explain the use of the "port wan" option. That's in
there for networks which need to be optimized differently for local
usage vs wide area usage. When you configure a WAN port you can
specify
a different window size (though of late, using Linux auto-tuning
seems
to work best for everyone). The client automatically chooses the WAN
port (after ngotiating with the server) when connecting over the WAN.
Both ports need to be accessible through the firwall. I doubt you
need
this for clustering IU with UC as the network latency is probably not
big enough to matter.

Andy

Hi Sarahm

I think what I really want is the xrd.port wan command, from
here:
http://xrootd.slac.stanford.edu/doc/dev/xrd_config.htm#_Toc235524916

But, to use it for both xrootd and cmsd I need them to use
different
config files.
Not at all. In the single config file use the construct:

if exec xrootd
xrd.port <whatever for xrootd>
else
xrd.port <whatever for cmsd
fi

The config file supports generalized if/else if/else/fi
constructs. For
detailed information refer to:

http://xrootd.slac.stanford.edu/doc/dev/Syntax_config.htm

which is available for any post-June 23rd release (go to the
xrootd.slac.stanford.edu web page for earlier documentation).
