    Sep 29, 2010 06:55 PM UTC by Soichi Hayashi 
    Injecting my comment to this conversation..

I believe our DNS-RR based failover never works... Even if we increase our buffer hours from 1 hours to 24 hours, some site will still fail because they cache their DNS entries for up to 24 hours. Repeated request to increase their cache refresh frequencies has so far failed. Even if DNS-RR information is up-to-date, the actual application that uses the DNS information can cache old IP address - and it will still fail. DNS-RR also fails to correctly load-balance request sent to GOC - as GOC is already aware (is1 gets 2/3 of all requests and is2 gets only 1/3 of all requests)

I am currently working setting up a LVS balancer at GOC, and if this idea becomes a reality (and I will push for this as hard as I can), we can replace DNS-RR with LVS and most of these problems will disappear.

    The most recent SAM tests seem to be returning OK status. Maybe the issue has been resolved by itself?

I believe that this issue was due to CERN DNS server caching old DNS aliases for is.grid.iu.edu during today's production release. We service only 1 instances at a time and update our DNS alias to point to working instance while servicing on another. We give at least 1 hour so that DNS servers around the world will have enough time to update its cache. However, some DNS servers are known to not honer requested keep-alive value, or the actual application caching the IP address for the duration of the process - thus voiding DNS alias changes.

Does everyone agree with this explanation?
