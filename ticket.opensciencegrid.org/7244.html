> Personally I think we should not be in the business of supporting email clients. Signed messages can be verified on the command-line using openssl on any system with the
> OSG CAs installed and maintained by vdt-update-certs:

I apologize if my request was confusing -- I was not suggesting supporting an email client per se. I was suggesting you offer a plugin/add-on to commonly used clients. For example, we provide a ticket id based search tool for Firefox that allows searching on GOC tickets.

> $ openssl smime -verify -in message.txt -CApath $X509_CERT_DIR -signer sendercert.pem
> $ openssl x509 -in sendercert.pem -noout -subject -email -serial -dates

While the above method might work on the odd occasion, it is hardly a scalable solution, is it? Other GOC staff have indicated this privately to me as well, but I personally do not find this approach very user-friendly. On a busy day while dealing with hundreds of incoming emails: the number of steps involved in taking an email, separating the sender cert, moving it to a Unix machine that has the CA certs installed, makes it less than ideal especially if the email in question contains sometime time-sensitive.

Also, how do you suggest we deal with websites that use a cert issued by one of the OSG-approved CA? Perhaps there is a path to that too using wget/curl and then using command line openssl but I hope you see that this method is even less practical to validate websites.

If you expect recipients and website viewers to accept the validity of certificates issued by one of the numerous CAs approved by the OSG, then it is only fair, in my opinion, to provide mechanisms to importing the CA certificates into commonly used tools. In the absence of user-friendly mechanisms to setup the validation infrastructure in email clients/browsers, I bet, recipients will request re-sends of emails with a more commonly used CA in the respective context/community.

Cheers,
-ag

PS: Kyle, given the security team does not plan to offer any automation mechanism(s), this ticket is likely ready to be closed.

    Personally I think we should not be in the business of supporting email clients. Signed messages can be verified on the command-line using openssl on any system with the OSG CAs installed and maintained by vdt-update-certs:

$ openssl smime -verify -in message.txt -CApath $X509_CERT_DIR -signer sendercert.pem
$ openssl x509 -in sendercert.pem -noout -subject -email -serial -dates

    Jim/Security team [adding Rob to ticket],

Kyle brought this up in our group discussion today, and I wanted to start a discussion with the security team.

I personally think that given we accept certificates signed by any OSG-approved CA for persons registering to OIM, it would be be only fair to also accept emails signed by a cert issued by any of those CAs. However, I also would like to add that (personally I think) there needs to be a clean mechanism for any one to install these CAs into their browser/mail client, and to keep the CA certs up to date. Randomly downloading a CA when needed is not a very scalable solution from various perspectives.

Would the security team be willing to investigate this process (of importing the entire OSG CA distribution) into browsers/mail clients)? and to perhaps provide a solution we could all use? (similar to vdt-ca-manage for OSG sites running CEs/SEs) ??

I know this could get tricky but perhaps you could provide a mechanism for the top 2-3 browsers and mail clients, while asking users to use one of those if they are using a very peculiar browser or mail client. I suspect that will be a fair expectation.
