    Mar 4, 2009 07:31 PM UTC by Ted Hesselroth 
    This problem, "lcp-cp hung", and "intermittent failures in storage probes" all have
the same symptom, a failure to read network io after a connection is made. So since
the clients are running from different places, the problem being at the pool node is
more likely. The first two explicitly use a tls library, and the java native calls in
the last likely do, too, though they don't show up in the stack trace.

Restarting the pool works for dccp because the client will try to reconnect.
Evidently lcp-cp doesn't do that. The fact that restarting the pool works doesn't
necessarily point to a problem in the pool itself; it may be just undoing a system
problem. For that matter, the act of reconnection could be doing the same on the
client side.

The name of the last dccp call looks like it is stuck reading something locally. I'll
try to find out.

Ted

Doing some  more research on the particular point at which dccp is hanging, the most
common explanation is in thread locking among processes accessing the kernel. For
example, if another process has the lock and hangs, then dccp will wait here forever
for the lock. So the cause could be from misbehaving processes apart from dccp. I
believe the problems with lcp-cp and srmping may have the same explanation.

So the same test as I suggested before, for the rsv probe, of running it without
contending processes would still apply. Also suggested is to turn off hyperthreading.
The problem could even be a bad driver for the network adapter. Tracing the exact
cause of the hang requires kernel level debugging, which I do not currently have
instructions for.

Ted

> If the kernel hang is causing the dccp client to hang, why does re-enabling the
pool in the PoolManager cause the problem to resolve itself?

If it is really true that a thread is waiting on a lock, it isn't doing anything.

    It's hard to see any recovery by the server being acted on by that thread. Maybe it
does time out, but the next attempt ends up stuck in the same place because of the
lock. Somehow the restart causes the process with the lock to release it. If the
scenario is correct, it points to contention within dccp. Maybe there are separate
control and data threads, and they are not correctly synchronized. I don't know
enough about the code to say. Tigran is out this week, maybe he'll be able to tell us
something when he gets back.
