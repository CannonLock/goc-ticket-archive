    My first worry is what I see in the list of Pacman commands that have been executed:

> % /share/apps/pacman-3.28/bin/pacman -get http://software.grid.iu.edu/osg-1.2:;ce finished at Tue Feb  2 13:43:36 2010
> % /share/apps/pacman-3.28/bin/pacman -get http://software.grid.iu.edu/osg-1.2:;Globus-Condor-Setup finished at Tue Feb  2 13:46:51 2010
> % /share/apps/pacman-3.28/bin/pacman -get http://software.grid.iu.edu/osg-1.2:;ManagedFork finished at Tue Feb  2 13:51:50 2010
> % /share/apps/pacman-3.28/bin/pacman -update-check finished at Tue Feb  2 15:48:22 2010
> % /share/apps/pacman-3.28/bin/pacman -get OSG:CEMon finished at Tue Feb  2 17:17:47 2010

The first commands installed from OSG 1.2, while the second one attempted to install from OSG 1.0. This can potentially mess up an installation. It certainly explains why you got the following error message:

> Package [/opt/osg-1.2:OSG:CEMon] not [installed]:
> Package [/opt/osg-1.2:OSG:CEMon] has not been installed.  Can't
> setup.

You previously tried to install that, it failed, and Pacman is still confused about it. I'm sorry Pacman is so sensitive. Basically, you told it that there is more than one correct place to install the VDT, and so it may not get updates from the correct place. In fact, that's what happened--during the update process, Pacman attempted to get updates from VDT 1.10.1, and everything got completely confused.

Unfortunately, there is no easy way to solve this problem. Pacman's database was confused before you started this update, and there is no way to remove that confusion.

I hate to tell you this, but the only reliable way to fix this is to do a fresh installation. If you've got your backup, you can restore it then do an new installation that automatically copies the configuration from it.

Sorry to be the bearer of bad news. :(

-alain
