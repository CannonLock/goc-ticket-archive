    Jun 3, 2015 08:42 PM UTC 
    Ken,

I spoke with Jeff at lunch. Effectively, the GLIDEIN_MaxMemMBs attribute set on the factory acts like a hard cap on what is advertised back to the frontend. It has no direct influence on glidein submission at Hyak. That is, e.g., let's say we bumped GLIDEIN_MaxMemMBs up to 256 GB. These glideins would be submitted to both the 2GB/core and 16GB/core nodes. It sounds like the problem with this would be that the frontend would then in some cases match user jobs to memory starved (2GB/core) glideins thinking that they are in fact memory rich (16GB/core) glideins, likely resulting in a steady stream of user job failures in the mismatched cases.

One outstanding question is whether or not the glideins will auto-detect the memory starved cases and adjust the per slot memory accordingly at startup and, of course, properly advertise that back to the frontend. I don't have an answer for that. Maybe someone else does? Or maybe we should just give it a try.

Marty Kandes
UCSD Glidein Factory Operations
