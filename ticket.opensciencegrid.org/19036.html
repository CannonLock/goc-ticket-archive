    Maybe I should explain how the proxy gets used for us.  While GSI can be used to access more resources on OSG, that's not a noticeable error condition for users (they just have less jobs running).  The important use case for us is to use gridftp to a central server to move files back and forth instead of using condor file transfer.  When the key is only 512 bits the gridftp process fails.

The easy test is to just run grid-proxy-info on the worker node and see what it returns (512 bits or 1024 bits).  With +x509userproxy in the condor submit file, jobs running on GLOW or CHTC have 512 bit keys.  Glidein jobs have 1024 bit keys, so they finally work.

    David,

I just took a look around the glidein server, and this is
what I found:

From our frontend.xml file:

<proxies>
<proxy absfname="/etc/condor/certs/pilot.proxy" security_class="frontend"/>
</proxies>

The proxy cert is defined as such for each group.  Taking a
look at the key for that certificate:

[moate@glidein certs]$ sudo sed -n -r
'/^-+BEGIN RSA.*/,/^-+END RSA.*/{p}' pilot.proxy
| openssl rsa -text
| head -n 1
writing RSA key
Private-Key: (1024 bit)

It looks like our glidein is generating a 1024 bit key, but
my understanding of the gwms frontend workings is *not* stellar.
Anybody at OSG GOC want to chime in?

Heath Skarlupka contacted us with this issue earlier this
month.  Here was my answer to him:

> Heath,
>
> I just talked with Zach Miller, and the specific issue
> is that when HTCondor talks to globus, globus creates a
> 512 bit key by default. There is a way to force HTCondor
> to tell globus it wants a specific key size. That is the
> following configuration value:

    >
> GSI_DELEGATION_KEYBITS = 1024
>
> Zach is also going to make a change to HTCondor to
> explicitly ask globus for keys of that size by default. That
> will be in future versions of HTCondor.
>
> Cheers,
> -Moate

    As noted in this ticket (https://jira.opensciencegrid.org/browse/SOFTWARE-1335), proxies with only 512 bits are failing with new OSG server certificates.  We can get around this problem by forcing grid-proxy-init to make 1024 bit proxies, but when submitting to condor with the x509userproxy option this gets ignored.

I'm not sure of the exact infrastructure behind the glideins, but my guess is that the glidein server is running an older openssl version and generating a new proxy with only the default of 512 bits.  Is there a way to fix this?

Right now we're forced to use the file transfer mechanism to transfer the proxy manually and set the appropriate environment variable so it gets used.  While this works for knowledgeable users, it would be best to solve the general case.
