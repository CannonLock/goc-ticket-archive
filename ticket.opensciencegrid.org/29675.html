    May 5, 2016 07:41 PM UTC by Carl Edquist 
    PS, the issue we are suspecting is perhaps rsv is running with a more restrictive umask that creates files non-world-readable. (For example, a umask of 0022 would create a file with 0644 permissions, which should work, but a umask of 0066 would create a file with 0600.)  But rsv relies on it being world readable so that the file, owned by rsv, can be read by the cndrcron user...

    I just installed a new certificate for RSV and unsure if that or some other change has broken RSV.

Attached is output from "rsv-control --on -v 3".  I have also attached rsv-control --profile tar ball.

This is the basic error:

# /etc/init.d/rsv restart
Stopping RSV: Stopping all metrics on all hosts.
Stopping consumers.
Starting RSV: Starting 12 metrics for host 'osg.osc.edu'.
ERROR: Problem submitting job to condor.  Command output:

ERROR: Failed to open command file (Permission denied)
ERROR: Problem submitting job to condor.  Command output:

The ERROR lines repeat for each metric and consumer.

What's strange is I can force things to start if I do this:

rsv-control --on -v3 gratia-consumer
<lots of output>
DEBUG: commands_getstatusoutput: command='condor_cron_submit /tmp/gratia-consumer.sub' user='rsv'
INFO: Condor submission:
ERROR: Failed to open command file (Permission denied)
DEBUG: Condor submission completed: 1 (256)
ERROR: Problem submitting job to condor.  Command output:

ERROR: Failed to open command file (Permission denied)

install -o rsv -g rsv /dev/null /tmp/gratia-consumer.sub

rsv-control --on -v3 gratia-consumer
<lots of output>
DEBUG: commands_getstatusoutput: command='condor_cron_submit /tmp/gratia-consumer.sub' user='rsv'
INFO: Condor submission: Submitting job(s).

    1 job(s) submitted to cluster 36998.
DEBUG: Condor submission completed: 0 (0)
DEBUG: Condor job cluster ID: 36998

By manually creating the tmp file for rsv, I was able to get the consumer to start.  Subsequent restarts still fail as the tmp file is removed once submitted.  The /tmp directory is world writable, so rsv should have no problem accessing /tmp.
