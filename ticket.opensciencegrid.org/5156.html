There apparently exists a dcache bug which caches tokens when it should
not.  If one token, eg, /atlas/Role=production, or
/atlas/usatlas/Role=production, is the only one in use, then all seems
well.  However, if both exist at a site, then dcache ends up with problems.

We also really did fill the space, but this was repaired on Monday.

For the atlas vs atlas/usatlas problem, we have done the following.  On
Friday, Hiro did:

insert into srmspace values (570010,
'/atlas','production',2,1,210000,70744055021871,1211393142260,-1,'ATLASMCDISK',0,9238859692590);

insert into srmspace values (570011, '/atlas','production',2,1,390027,
96463499984500, 1211393310031, -1, 'ATLASDATADISK',0,10709978213487);

and also he said "Ok.  I missed one more value (allocatedspaceinbytes)
which should be 0"

Today, Wenjing did:
since there are still upcoming request with
/atlas/usatlas/Role=production to ATLASCALIBDISK which resulted in the
recent failure.., maybe it is the dcache bug, so I just add another
space token with
/atlas/usatlas/Role=production ..


    cache=> select * from srmspace where description='ATLASCALIBDISK';
