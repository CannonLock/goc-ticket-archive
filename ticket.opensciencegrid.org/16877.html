    Hi all,

It appears the user is submitting using a VOMS proxy with
"Role=production" extended attribute:

"/auger/Role=Production/Capability=NULL" according to the logs.

The GUMS definition in the OSG template for the Auger VO specifies to
match the FQAN *exactly*, so it refuses to map.

If the user submits with a VOMS proxy without a role, it will succeed:

[root@gridgk01 ~]# gums-host mapUser -s
'/DC=org/DC=doegrids/OU=Services/CN=gridgk01.racf.bnl.gov' -f '/auger'
'/DC=es/DC=irisgrid/O=ugr/CN=mdserrano'
auger

vs.

[root@gridgk01 ~]# gums-host mapUser -s
'/DC=org/DC=doegrids/OU=Services/CN=gridgk01.racf.bnl.gov' -f
'/auger/Role=Production/Capability=NULL'
'/DC=es/DC=irisgrid/O=ugr/CN=mdserrano'
null

The other option is to change the requested configuration in the OSG
template from 'exact' to 'vo'. That will let any "/auger" VOMS proxy
map
to the auger user, whether it has associated roles or not.

But as long as the production role is unnecessary, then the simplest
way
to run here (and at any site using the OSG template) is to not request
the production role for the proxy used.


    Cheers,

--john
