    Oct 9, 2017 01:59 PM UTC by garhan.attebury@.... 
    We’re continuing to debug this, but believe we may have narrowed it down to an issue with the latest stable kernel in the EL 7.4 series. In particular, after a downtime when we had updated to the latest stable packages we were running kernel 3.10.0-693.2.2.el7.x86_64. Yesterday (October 8th ~1600 UTC) we removed all running jobs and brought up a smaller set of worker nodes running the previous kernel version 3.10.0-514.26.2.el7.x86_64. Since then we haven’t seen any of this strange failure happen.

The failure itself seems to have something to do with the volumes we bind mount inside the docker containers where jobs run. In particular, it seems that at random some or all of the volumes disappear from the view of a running job. This ends up causing our local SE mount /mnt/hadoop to disappear in some cases, which cause the jobs to try to read data via xrootd, at which point they try to load a shared library from /cvmfs/cms.cern.ch/ which has also disappeared ultimately causing the job to fail. I can’t say for certain that all mounts disappear at the same time or not. It also seems that the mounts may not actually permanently disappear either, as attaching an interactive shell to running containers that failed jobs often still show the mounts as being fine and the files which the HC job logs indicate were missing are 100% available, even though the container is exactly the same one which ran the job. Very, very odd behavior.

At this point it seems to be related to the kernel version – we just now need to determine what changed between the previous and current stable versions.

- Garhan Attebury
