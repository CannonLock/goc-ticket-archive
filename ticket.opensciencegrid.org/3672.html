    Jun 25, 2007 10:32 PM UTC by Chris Green 
    Based on diagnostics, this appears to be another case of bad site configuration by
configure_gratia.sh and/or configure-osg.sh.

Problem synopsis:

1) SGE Probe has a bug: new versions of Gratia.py and sge_meter.py will be attached
to this ticket tomorrow.

2) Both SGE and condor probe configurations have the following problems:
* SOAPHOST is configured wrongly.
* SiteName is configured wrongly.

Corrected ProbeConfig files, .py files and a repair script to recover unsent records
will be attached to this ticket tomorrow along with instructions for running the
repair script.

Shaowen, Suchandra: did some diagnostics and discussed them with Alain. It looks like
they have two batch systems on this CE: Condor and SGE. Note that this is not that
unusual any more. It seems that (to quote Alain) there are two issues:

(5:18:46 PM) Alain Roy: Issue 1: configure-osh.sh is clearly insufficient when there
are multiple batch system that must be configured.
(5:19:07 PM) Alain Roy: Issue 2: why did configure-osg.sh fail in this case? Was it
due to Issue 1, or was there a different problem?

configure-osg.sh appears only to be able to configure one batch system; also even
though (from osg-attributes.conf and vdt-install.log) they seem to have used "SGE",
configure_gratia.sh was invoked with no arguments and neither the SGE probe nor the
Condor probe were configured properly.

I consider this a hot-button issue for both the 0.6 and 0.8 streams -- I don't want
to be fixing probe configurations for > 50 sites by hand every time somebody runs
configure-osg.sh.

Comments appreciated from the troubleshooting team.

    
Thanks,
Chris.
