    Ian,

CMS jobs will run on CMS-owned nodes - where they won't get preempted - if there are slots available.
If a job starts to overflow the dedicated slots and runs on opportunistic nodes - as is happening below - then there is no way to prevent preemption. Preemption is the price for opportunistic access.

You're welcome to use something like Requirements = 'ClusterName == "CMS" '  in your RSL, but be aware that throughput will be reduced.

And finally, the messages below show that jobs are coming from the HCC VO - not CMS.

Hi,
We've noticed something funny on glideins submitted to Purdue from the glidein Factory at UCSD on behalf of the VOs CMS and HCC.
We submit with the following configuration:
gatekeeper="osg.rcac.purdue.edu:2119/jobmanager-condor"
gridtype="gt2"
rsl="(queue=cms)(jobtype=single)"

We get the following errors for about 10% of our jobs. It's a little opaque. So some explanation follows.
Starting monitoring condor at Wed Jan 12 20:45:18 EST 2011 (1294883118)
/home/ba01/u127/osghcc/.globus/.gass_cache/local/md5/a1/f3/32/2dd22ac464b28f7848340ad4ed/md5/d9/83/d3/25b08946ccaf89e195156fe461/data: line 1039: 32014 Hangup                  "${gs_id_work_dir}/$last_script" glidein_config
Wed Jan 12 20:51:45 EST 2011 Error running 'condor_startup.sh'

The important part of the error is the "Hangup" that is reported.

From another log we find the following lines to also clue us in.
=== Condor starting Wed Jan 12 20:45:18 EST 2011 (1294883118) ===
=== Last script ended Wed Jan 12 20:51:45 EST 2011 (1294883505) with code 129 after 387 ===

We contacted the glideinWMS support team and Doug (cc'd) had the following to say:
"Exiting with code 129 means that the process was sent a Unix signal 1, which is a SIGHUP (Hangup), which explains why it is printing out "Hangup".

I am not sure why the process was sent a SIGHUP though.  That's usually for reconfigures.  If the glidein expires or timeouts, I think its a SIGTERM.
Also, the new signal handling code uses SIGTERM as well, so I am not sure why it would receive SIGHUP."

We've seen this occur only on the following nodes (many other nodes run fine, and no jobs at all run on these nodes):

2 Running on steele-a226.rcac.purdue.edu
9 Running on steele-a245.rcac.purdue.edu
2 Running on steele-a250.rcac.purdue.edu
2 Running on steele-a259.rcac.purdue.edu
1 Running on steele-a448.rcac.purdue.edu
15 Running on steele-a599.rcac.purdue.edu


    We were wondering, if there is anything you can find out at the site about why we would receive a SIGHUP signal for some of our jobs.
