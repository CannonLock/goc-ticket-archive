    Nov 18, 2011 03:20 PM UTC by Thomas Lee 
    My thinking was hasty in my previous reply.  Thinking about this some more, installing this or any OSG application requires two different sets of packages:

1. The support packages not part of the application but required by it.  These are packages that are commonly included with the supported Linux distributions; however, there are multiple supported distributions, and some sites may have internal repositories, so there's no way to specify where the installation script should get these from in a way that would work anywhere.
2. The application-specific packages.  These come from a single known repository, so there's no problem with specifying this one.

It is possible to issue two separate YUM commands, and the command for group 2 could ignore all repositories other than the OSG one, but that still leaves group 1, which would have to be installed first.  For group 1, one possibility would be simply not to limit the repositories, but this risks the potential problem that the target system may have divergent repositories defined containing packages that cause conflicts.  An attempt to limit the repositories exclusively (with "--disablerepo") encounters the problem that there is no way for the script to predict what "bad" repositories the target system has defined.  Limiting the repositories inclusively (with "--disablerepo=*" followed by "--enablerepo") has the same problem; the script can't predict which "good" repositories to use.  We would seem to have run out of options for doing this automatically.

At this point the only thing I can think of is for the person doing the installing to be aware of the repositories in use on the system ("yum repolist"), and for the script to give them some sort of ability to disable some of them for this installation.  Perhaps the script could identify the "distribution" repositories by searching for one or more of the support packages among the defined and enabled repositories like this:

for i in `yum repolist | grep -Ev '^(Loaded|repo id|repolist:)' | cut -d ' ' -f 1`; do echo -n "*** $i: "; if yum --disablerepo=* --enablerepo=$i info java-1.6.0-sun.x86_64 >& /dev/null; then echo t; else echo f; fi; done

Then, if there were more than one repository that seemed to have "several" of the support packages, the user could be prompted to choose which one was correct, or if not, the script could merely select that one.  This seems overly complicated, though.

Another thought I had was that the creator of the various application-specific RPMs could be more specific about their dependencies and the versions they require.  This would keep YUM from running off to some divergent repository to install a bleeding-edge version of some package, but it would also limit the distributions and update states on which the application could be installed.

Well, if you are in a situation where you have little control over what other repositories might be configured and enabled on the system, it's probably best for you to ignore all of them except for the ones known to contain the necessary packages.  Suppose someone had an rpmforge repository defined and enabled by default, and suppose it had an incompatible mysql package -- perhaps the latest bleeding-edge development version, perhaps dependent on newer versions of libraries -- yes, you can advise people not to have such things, but you already have control over which repositories your installation script will access.

It's what I have to do here -- various OSG software installs strange repositories and configures them to be enabled by default, so our OS update script does "--disablerepo=*" followed by "--enablerepo" statements activating only the repositories relevant to the update.  That way I avoid silently replacing the RHEL packages with, for example, some bleeding-edge openssh package from UNL, on certain servers and not on others.

    We have an internal YUM repository server containing a mirror of the IU Satellite server's RHEL5 Server x86_64 packages -- this is because it can be updated just before doing ITB OS updates and then left alone for a week, so we can ensure that the following week's production OS updates use exactly the same packages.  (Packages that Red Hat released between our ITB and production updates would appear on the Satellite server, but not on our local mirror.)

The goc-rhel-x86_64-server-supplementary-5 repository is one of these repositories hosted on this internal server.  We just had ITB OS updates on Tuesday, so this repository was synchronized with the RHEL Satellite on Tuesday morning.  It should (*should*) contain every package that the Satellite contained as of then.  I synchronize using Cobbler.

But I find it strange that adding "--disablerepo=goc-rhel-x86_64-server-supplementary-5" worked ... because YUM is still obtaining packages from IU's RHEL Satellite, which should be the same ones that are in the internal mirror.  Perhaps the internal mirror malfunctioned, because I just looked at the internal mirror's directories, and I'm seeing the files that are reported "not found" below (and the internal mirror hasn't been updated since Tuesday).

An unusual thing, though, is the fact that the Java update that recently came out was 1.6.0.29, which is available from the IU Satellite, whereas the most recent version on the internal mirror is 1.6.0.26, despite the fact that the internal mirror was supposedly synchronized with the Satellite as of Tuesday.  1.6.0.26 is also the version that the installation procedure below was trying to install from the internal mirror, not the more recent 1.6.0.29 (released October 19).

There are some unanswered questions about our internal RHEL mirror (why didn't it update with these recent Java packages?  why does it report java-1.6.0-sun-1.6.0.26 not found when it's obviously there?) but also some questions about the intended VOMS installation (what versions of the java packages is it meant to install?  where does java-1.6.0-compat come from?  why did it select java-1.6.0-sun-devel and not select java-1.6.0-compat?).

I would suggest that the installation script be very specific about the repositories it includes -- a command like "yum --disablerepo=* --enablerepo=repo1 --enablerepo=repo2 ..." avoids surprises.  This allows YUM to heed only the repositories specified and ignore all others.
