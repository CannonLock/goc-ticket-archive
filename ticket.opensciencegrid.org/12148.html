    May 23, 2012 08:45 PM UTC by Thomas Lee 
    I do remember that back in March, we removed a local cron job that we have on all OSG Operations servers called crlsync-rsync, because it brought in conflicting CRLs.

crlsync-rsync doesn't work very well because every server tries to rsync the CRLs at the same time, which means that each time, some servers don't get them.  So I've been trying to improve it by replacing it with a Puppet rule.  However, I didn't recall that osg-xsede should be excluded from this, and Tuesday the Puppet rule went into effect, as it was production update day, bringing back the crlsync-rsync cron job on osg-xsede.

I'm sorry I forgot about that, but I've now put a specific exception to the Puppet rule, making the exclusion part of the institutional memory that is the Puppet system.  The crlsync-rsync job will not be coming back.  But I'm still not sure whether this caused the problem mentioned.

Besides changing the Puppet job, I also ran the job in /etc/cron.d/fetch-crl by hand, removed crlsync-rsync (again) and moved all expired CRLs in /etc/grid-security/certificates to /root/expiredcrls.  If crlsync-rsync caused this problem, it should be gone again.  If not, I haven't done anything that wouldn't have happened every six hours anyway.
