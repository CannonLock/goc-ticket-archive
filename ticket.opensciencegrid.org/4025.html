    Sorry, Alain. I set an option to send you the full ticket description. But since it didn't make it to RT,
I'll just copy and paste it into this update:

Dear GOC,

I tried to raise this issue in the osg-int mailing list and did not get
the response I was looking for.

What is happening is this:
I installed the host cert on ligo-web.aset.psu.edu on 08/2006 when the
value of CN was "host/ligo-web.aset.psu.edu"

I installed a new certificate on the machine on 08/2007 and this time
the CN was "ligo-web.aset.psu.edu". NOTE THERE IS NO "host/" in the CN.

So the vomses files distributed with the OSG stack has the old DN with
CN=host/ligo-web.aset.psu.edu and therefore voms-proxy-init fails for
LIGO users unless they edit vomses to update the value.

What I would like to see happen is voms-proxy-init trust both
CN=host/FQDN and CN=FQDN format of the host cert, OR, have the vomses
file contain only the hostname and CA, not the DN of the VOMS machine.
Could you contact the person responsible for voms-proxy-init and find
out the best form of action?

The short term solution is to update the vomses file and redistribute it
via a routine OSG software stack update. The value of the host
certificate on ligo-web.aset.psu.edu currently is:
/DC=org/DC=doegrids/OU=Services/CN=ligo-web.aset.psu.edu

Thanks,
Murali

I installed a new certificate on the machine on 08/2007 and this time

I have installed a OSG:client and test the "voms-proxy-init", it seems when I issue
the following command, it fail with the following message

[lhuang@osg-vtb ~]$ voms-proxy-init -debug -voms LIGO
........
Authorization denied: The name of the remote entity
(/DC=org/DC=doegrids/OU=Services/CN=ligo-web.aset.psu.edu), and the expected name for
the remote entity (/DC=org/DC=doegrids/OU=Services/CN=host /ligo-web.aset.psu.edu) do
not match

Based on the error message, therefore I changed/edit the file
"/usr/local/client/glite/etc/vomses" from "CN=host /ligo-web.aset.psu.edu" to
"CN=ligo-web.aset.psu.edu".  And then init process was successful.

I am not sure which part should responsible for this mismatch problem? But either the
file need to be fix, or otherwise Murali need to change his server DN (not sure
whether this will affect how many existing LIGO users). So if fixing the file is not
a problem, I would suggest let's fix the file.

Thanks
