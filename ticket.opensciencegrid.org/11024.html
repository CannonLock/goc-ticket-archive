Hello,

When I attempted to convert my CE to using GUMS, then reverted back to my grid-mapfile, something got broken and I can't track it down.  RSV tests are passing and I'm able to submit jobs via Condor glideIn, but not via gLite.

I used condor_q -l to get the "Cmd" ClassAd, which gives me one of the directories storing the job state.  I then looked at the tag file to get the globus job ID:
https://hepcms-0.umd.edu:40886/17541/1316354064

Strangely, sometimes when I called globus-job-status on this job ID, I
was told the job is DONE and sometimes I got an error:
ERROR
GRAM Job status failed because an authorization operation failed (error code 7)

More recently, I have consistently started getting the error:
ERROR
GRAM Job status failed because the job contact string does not match any which the job manager is handling (error code 156)

though I know this job is still present on my system.  Perhaps it's been there so long it's entered some unrecoverable state.

I do see at least some of my authorizations in my globus-gatekeeper.log file and everything from me within that file looks OK - my DN is being mapped to the user that my grid-mapfile has configured.  This is not the same as the user my GUMS server maps to (for testing purposes), so I know my grid-mapfile really is being used, at least some of the time!  However, as far as I can tell, when I get the above error, those authorization attempts aren't even showing up in my CE's globus-gatekeeper.log (where else might they be??).

I also have an astonishing number of pings from the a CMS testing DN (SAM), some of which have an odd statement at the end:
Failure: ping successful

I did modify my config.ini back to using grid-mapfile, specifically, in the [Misc Services] section:
gums_host = %(unavailable)s
authorization_method = gridmap
and I did call configure_osg -c (stop services, configure, start services).

Could there be some detritus somewhere that is occasionally trying to use GUMS instead of gridmap, now that I tried to get the CE to use GUMS once?  I note that the file /etc/grid-security/prima-authz.conf was left behind, which I moved out of that directory, though I didn't do a service restart after moving the file.  I also checked that gsi-authz.conf is using globus_gridmap_callout.

Thanks in advance for any guesses you may have,
Malina
