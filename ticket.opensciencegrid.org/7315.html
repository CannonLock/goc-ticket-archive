Work Info Type* : Resolution Communications

Summary*        : Log analysis
Notes           : Hi,

First, I apologize for the very long delay. I have finally analyzed the logs you sent back in September.

Out of the >2M logs, approximately 443K have no endtime. These records also have no starttime, which is the problem. Based on the general tenor of these records, for example:

lnxfarm:lnxfarm238.colorado.edu:man:nlc:runBDK.pl_27275:1754235:sge:0:1251298088:0:0:19:0:0:0:0:0.000000:0:0:0:0:0:0:0:0.000000:0:0:0:0:0:0:Users:defaultdepartment:NONE:1:0:0.000000:0.000000:0.000000:-l lnxfarm_quad4=1 -P Users:0.000000:NONE:0.000000

I conclude that these represent jobs that were canceled without ever being run. It is a minor inefficiency that these jobs are sent to the collector; but we reserve the right to do *something* with them at some point. We have certainly quieted-down our collector-side complaints. I would say therefore that this is not a problem. However, while analyzing the code, I noted two important limitations of the current SGE probe:

1. The probe cannot cope with the corner case where it reads a currently-incomplete log line.
2. The probe cannot cope with any form of log rotation or truncation.

I have fixed problem .1. and will raise problem .2. with the original author of the probe, which was an external contribution to the Gratia project.

To address your other concern, that of all the files you had in lost+found. These files only build up during an extended outage of the collector. Recent versions of the probe (most notably the one just about to be released to the VDT) are much more intelligent about combining files that have failed to be sent in a way that can be easily dealt with when the collector comes back online. I would urge you to update the probe software as soon as it is announced by the VDT.

Again, I apologize for the delay. Please don't hesitate to contact me if you have any further questions or concerns, or if I have failed to answer any questions you have already posed.

Thanks,
Chris.

    Aug 28, 2009 04:41 AM UTC by doug johnson 
    Greetings,

Sorry I didn't respond sooner.

The SGE log file, actually accounting log, is over 2 million lines
long. It contains all the jobs run on the system for the past ~3
years. If you really want to look through it, I put a bzip2'd copy of it
at:

http://www-hep.colorado.edu/computing/gratia-logs/sge.log.bz2

It is a 60MB download, that will be ~600MB when unpacked.

I don't know if this will save you time, but I found further evidence
that what gratia did was try to parse the >2 million batch jobs listed
in the accounting file. When I tried to reboot the machine that was
running the test CE, the disk with the installation failed fsck. After
fsck'ing the disk by hand, I found ~6GB of small files in lost+found. I
am not positive what a gratia "data" file looks like, but these
certainly seemed like what I would imagine one to be. So I assume there
was 1 file for every job that gratia had processed. I have since removed
the files from lost+found, so I am unable to send you a sample.

So in my opinion, either gratia should be designed to not try to back
fill so many jobs or folks should be warned so that they can truncated
the SGE accounting file before starting the SGE gratia probe.  I am also
not sure what to do about jobs submitted to SGE via the CE versus jobs
to SGE by local users? Currently, we are not running our CE with SGE,
but that is the future intent.

Hope this helps,
doug
