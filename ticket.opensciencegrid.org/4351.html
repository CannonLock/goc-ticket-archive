New Work Log Entry : The following was e-mailed to the Requester:

thomas@.... sent this Notes To Requester:

As a follow-up:

There seems to be a second problem with the automatic mapping that was
generated during the gums installation/upgrade.  The automatic mapping
was of the form:

<hostToGroupMappings>
<hostToGroupMapping
groupToAccountMappings='gums-test'
description=''
cn='*/?*.ultralight.org'/>
</hostToGroupMappings>

Note that it assumes the host certificate from the CE contains a / in
the CN field:  */?*.ultralight.org

Our CE host certificate subjects do not look like this.  Instead, they
look like:

/DC=org/DC=doegrids/OU=Services/CN=cithep169.ultralight.org

After modifying the pattern (actually, I just appended 'gums-test' to
the existing host to group mappings), I was able to verify through the
gums management web page that this test mapping works.

--Mike

New Work Log Entry : The following was e-mailed to the Requester:

thomas@.... sent this Notes To Requester:

It seems that our new GUMS installation from OSG 0.8.0 did not correctly
set up the GIP test user mapping correctly.  Our original gums
configuration had two host -> group mappings, one for '*.ultralight.org'
and the other for '*.cacr.caltech.edu'.  During the GUMS upgrade, the
mapping to the gums-test group was only made for *.cacr.caltech.edu, not
*.ultralight.org.  Since our CIT_CMS_T2 gatekeeper is
'cithep169.ultralight.org', the mapping failed.

I've added the mapping for *.ultralight.org to our gums config.  Let me
know if this continues to fail.

--Mike
