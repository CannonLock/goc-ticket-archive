    Jul 22, 2010 05:48 PM UTC by Philippe Canal 
    Hi,

After further analysis (thanks to John for your infinite patience and
help :)), the problem is indeed
a deficiency in the pbs/lsf probe that affects extremely active sites
during a Gratia Collector downtime.

The pbs/lsf probe is a two stage process.   In the first stage, we use
the urCollector (contributed by
EGEE) to transform the accounting record into xml files.   In the second
stage, we take those xml
files, update them with OSG specific extra information and upload them
to the Gratia Collector.

The 2nd stage has appropriate mechanisms to prevent disk filling during
extended Collector outage.
When used by the pbs/lsf probe, it stops creating new xml files if the
local input queue is more than
100000 files (by default), however this also means that it stops
deleting the xml files created by the
urCollector.

The reported problem stems from the fact that the 1st stage (the
urCollector part) does NOT have any such
control mechanism thus the probe can fill up a disk in the case where
the Gratia part is no longer consuming
the input and there is a lot of jobs running on the batch system.

In John's case there is about .5 million local jobs every day, thus the
couple of days (last week)
of down time lead to more than 1 millions  file accumulated in the
urCollector outbox which
where not consumed by Gratia and thus kept accumulating.

Once the disk was filled, the directory containing the data was deleted

    by the administrator
so that it could be soft linked to a different physical.   This deletion
resulted in a 'resetting' the
pbs/lsf probe which then created a record for all the jobs still in the
lsf accounting file, resulting in
more than 4 millions files in the urCollector outbox (since the
Collector's queue was
full due to the catch up needed after the latest Gratia release).

The backlog of data should now resolve itself with time since the
Collector is now
taking inputs.

The underlying problem will be solved in the next Probe release by
introducing
proper throttle mechanism in the pbs/lsg probe.

Cheers,
Philippe.

    Hi,

Since there was already data loss, I propose that you reset the lsf
probe (by deleting gratia/var).

Then you ought to install the probe version 1.06.16c-1 which contains
additional checks
against a filled disk.

Then restart the probe which should reprocess your old data and
eventually catch up.

Philippe.
