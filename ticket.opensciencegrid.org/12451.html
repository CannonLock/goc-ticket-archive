    Aug 14, 2012 06:52 PM UTC by Scot Kronenfeld 
    Hi Bob,
The proble is that the Gratia Metric probe on gate02 is not configured correctly.  This was a problem with osg-configure versions 1.0.7 and earlier.  The 1.0.8 release has code that configures the Gratia Metric probe properly.

You have two choices:

1) Update osg-configure to 1.0.8 or later on gate02 and re-run osg-configure to let it properly set up the Gratia Metric probe.

OR

2) It sounds like you are running RSV only on gate01 and using that to monitor both gate01 and gate02.  Since you aren't running the RSV service on gate02 you aren't using the Gratia Metric probe.  You can just disable checking the Metric probe on gate02.  This can be configured in /etc/osg/config.d/30-rsv.ini.  Look at the 'gratia_probes' setting.  It has a comment explaining how to do this configuration.  Holler if you have questions.

-scot

    Hi Bob,

I've looked it over carefully, and this is my best interpretation.

Gratia is used in two ways and you'll see both if you run RSV on the CE itself.

1) Gratia is used to upload accounting information, like number of jobs run and how long they run for.
2) Gratia is used to upload RSV results. This is not accounting data, but we share the underlying data transfer mechanism to reliably upload the data to the central RSV service.

When I looked at the directories where the Gratia data collects, I see two directories:

> subdir.condor_gate02.grid.umich.edu_gratia-osg-prod.opensciencegrid.org_80
> subdir.gridftp-transfer_gate02.grid.umich.edu_gratia-osg-transfer.opensciencegrid.org_80

Each one of those is for case #1, the accounting information. The first directory is for Condor job information, the second is for GridFTP information.

When I looked at the error message from RSV, I see:

> WARNING: The expected directory containing records waiting to be re-sent does not exist (/var/lib/gratia/tmp/gratiafiles/subdir.metric_gate02.grid.umich.edu_rsv.grid.iu.edu_8880).

This is case #2: the RSV result data does not appear to have a directory. It is correct that it's port 8880 (that's buried in osg-configure). This is a sign that RSV isn't pushing data to Gratia. It should be, since you apparently have Gratia reporting enabled in your config.ini (unless you've changed it recently--I looked at your osg-system-profile from July 31st.)

So the question is why RSV isn't reporting this. Let's check out a couple of possibilities.

1) RSV's "gratia consumer" isn't running. Can you please run "rsv-control -j" and share the output? On an RSV host, I see in part:

> # rsv-control -j
>
> ...
>
>      ID OWNER      ST CONSUMER
>    11.0 rsv        R  html-consumer
>    12.0 rsv        R  gratia-consumer

I would expect to the same (with different IDs) on yours.

    
2) Did RSV report any errors in the log file for the Gratia consumer? There are two relevant files to look at:

/var/log/rsv/consumers/gratia-consumer.out
/var/log/rsv/consumers/gratia-consumer.output
/var/log/rsv/consumers/gratia-consumer.output
/var/log/rsv/consumers/gratia-consumer.err

Some of these might be very long, so I'm most interested in the ends of the files.

This will probably lead us to the underlying problem.
