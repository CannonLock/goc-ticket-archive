Gabriel,

We are sorry for the troubles.  We'd like to work with you to try and resolve the problems.  There are several things we can do to adjust how our jobs run.  If these don't solve the problem, then there are a few things you can try which should help.

We are running the same jobs at many other sites, so if the problems are due entirely to us, other sites are probably experiencing the same issues and simply haven't narrowed them down to us (yet!).

I will give you some details about our jobs in case that helps.

We are using glideinWMS which submits "pilot" jobs to UNESP.  Each pilot job will run on a WN job slot for as long as possible (queue time limit or until evicted, I believe, although it may have a built in time limit after which it exits "gracefully").  Each pilot job will fetch from our job server an "actual" job.  These jobs are typically 5-10 minutes long, although occasionally they can run for 40-60 minutes.  Each will "relocate" to a sub-directory of $OSG_WN_TMP (if set) or to /tmp.  This means the job should run on local disk on the WN, not a network file system.  Each job uses Condor file transfer mechanisms to transfer 1-3 MB of data in 2 files.  The larger of these is the same file for a big set of jobs (in other words if you have lots of SBGrid jobs running at your site, one of the two files each job accesses is going to be the same).  Previously we were using a combination of Condor-G and curl/wget (with http_proxy, if Squid or some other web-cache was setup).  This meant that the "common" file was fetched from GASS cache or Squid cache, if available.  I do not know what happens (if anything) with caching in the glideinWMS pilot job system -- we have only been using it for 2 weeks.  The running job will create 10-15 files in a total of about 10 MB, but only send back to us about 500 bytes.

My guess is that the problem comes from the file transfers.  It is also possible that there is a problem due to the high number of "internal" jobs we have.  A final possibility is that we have a lot of failures at UNESP (I'm not sure about this -- I can check our logs tomorrow -- I know we have had lots of successful jobs at UNESP in the past week), and a lot of failures can certainly lead to problems of high load as jobs start and stop rapidly.

Hopefully Igor (from glideinWMS) and Peter (from SBGrid) can provide some more insight.  Tomorrow (Wednesday) I'll investigate improvements to our file transfer system.  We can then discuss how best to try and test out the changes at UNESP in a limited fashion to ensure we don't create more problems for you.

Regards,

Ian
