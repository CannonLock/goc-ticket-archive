    Hi all,

Just to confirm, I see several hundreds of jobs with the error
"ORA-12520: TNS:listener could not find available handler for
requested
type of server" in the schedconfig query output in jobs from most US
sites (MWT2, AGLT2, SLAC). The result of the error would be to not
source whatever is in AGIS for envsetup for the queue.

As for why this didn't cause the scale of failures at other sites as
we
saw at BNL, I suspect it has to do with different local site movers.
But
it would take a detailed comparison of jobs everywhere to be sure. The
BNL failure is failing to get TiersOfATLAS from dq2 within the local
site mover. A job at AGLT2 succeeded even after the Oracle failure and
doesn't try to get TiersOfATLAS in its local site mover code.

In any case, we are adding content validation to the wrapper and
exitting immediately if we don't get proper data from the schedconfig
query.

GOCTX Source: BNL Ticket: Queue=GridServices ID=25195
    Mar 18, 2015 01:14 PM UTC by OSG-GOC 
    Hi all,

The underlying cause is a DB problem at CERN. The wrapper is asking
what
files to source, and the curl request returns successfully. BUT, the
contents of the reply is the DB error, not the name of the file to
source.

So parts of the pilot environment are not properly set up, and DQ2
isn't
set up, and thus Rucio info isn't retrieved properly.

All the wrapper currently checks is that curl returned successfully
(RC=0), which it did. We (Jose and I) need to add some curl response
content validation as well, and if there is a failure, exit the
wrapper
entirely.

I just manually checked the curl request, and the DB error seems to be
gone (at least on the back end my request hit). So things should begin
to recover.

Cheers,

--john
