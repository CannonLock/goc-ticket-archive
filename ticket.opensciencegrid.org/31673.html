    Note to Andrew: keep in mind that one thing is what the pilot requests and actually uses from the WNs, another thing is what it considers to manage for job scheduling purposes. This second parameter has to be corrected (increased) to compensate for the inefficiency that would come from the memory request auto-rounding (2.2, 2.3 -> 2.5 GB). So, as a general rule, as we switched to multicore pilots but kept running single core jobs, we needed to configure pilots to assume 2.5GB/core, even though we are supposed to respect the (VO card) 2 GB/core rule. Not doing so was noticed to result in never filling 8/8, cores but 6/8 or 7/8 cores in our regular 8-core pilots.

Priority has been changed from urgent to less urgent.
Hi all-

This might be a known issue, but there's a unit mismatch between SI and CMS that's confounding things. SI/gWMS/Condor are using power-of-ten units for memory while CMS uses power-of-two units for memory.

I opened this ticket because the factory entry was for 4,000MB (4 * 10^3) but the CRAB/CMS default request was for 2,048MB (2 * 2^10). Though both of the units were described in terms of "GB", the different units made it so that 2x2GB CMS jobs wouldn't fit in one 4GB pilot.

Now if I look at the monitoring, I see that there is a lot of cores being occupied by CMS production jobs, each requesting 2.5GB. However the memory request is in power-of-two units, 2.5GB jobs are requesting 2,560MB. So, even though the pilot is 5GB, two production jobs won't pack in since 5,000MB - 2,560MB = 2,440MB.

It may be a known issue that's considered to be unimportant, but I wanted to bring your attention to it if not.

Cheers,
Andrew
