    Hey Scott,

A few things:

1) Your gridftp transfer appears to be timing out after 10 minutes.  I'm not sure what the command-line option you need to adjust, but I know that
until very recent versions of the client, this was not an adjustable parameter.
2) Upon retry of the gridftp client, it does not delete the previous attempt first.  That's what is causing your hadoop "File under construction"
problems.  I'd suggest that for now you do -retry_num=0.  Alex / Junmin - could you issue a delete for the partial file prior to retry?

I have CC'd the OSG-storage support list.  Alex or Junmin, you can see the error messages here:

http://cmsweb.cern.ch/phedex/prod/Activity:;:ErrorInfo?tofilter=UCD&fromfilter=Nebraska

On 12/14/09 7:15 AM, Brian Bockelman wrote:
On Dec 13, 2009, at 11:24 PM, Alex Sim wrote:

could you issue a delete for the partial file prior to retry?

bestman does not delete the partial files.
if this has some specific issues with Hadoop, I say users should "rm"

    the problematic files before overwriting it.

But this is not possible!  This is happening:

1) srm-copy starts gsiftp transfer
2) Partial file is transferred
3) srm-copy times out and then stops transferring.  Sometimes this causes issues with Globus GridFTP, causing it to hang; hadoop internally will
keep the file under construction until GridFTP is killed.
4) srm-copy sleeps for 60s.
4) new gsiftp transfer is retried by srm-copy.

So, it's really an issue with -retry_num being set to >0.  srm-copy never exits, so it's not plausible to insert a "rm" in the middle of it.

Brian,
what I meant is that this is not general enough that we can put into
srm-copy.
srm-copy would not go through rm interface to remove the partial files.
It'll remove the entire SURL and it has to go through from the P2P/P2G
in general.  bestman gateway server has no idea if anything is partial
or not. Besides, it's not the client tool to decide to remove the
remaining file if it's known to be partial. Although we don't do it yet
and other tools may be, if 99GB out of 100GB file is dropped, you don't
want to re-transmit the file, but to "resume" in any possible ways.
One thing we can do is to make the default retry-num to be 0, so that
users have a chance to use srm-rm if they know specifics of the target
storage.
--Alex
