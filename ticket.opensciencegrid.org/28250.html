    Hi,

For example: when user transfer is failing due to quota problem, T2_CH_CERN is reporting [1] and T3_US_FNALLPC is reporting [2].

Natalia, I know this is clear users quota issue. I seen this ~1year ago, but was not creating issue as it was stated if it is user problem, he has to know how much space he will need and he has, it is up to user to fix his problem.  But FTS3 back-off mechanism lowers number of transfers and affects other users in FTS3@RAL. Please see my longer reply below:

PhEDEx is not looking into latency and it can wait 1,2 days, but ASO/CRAB3 not. Due to this error message [2] FTS3@RAL is considering as site issue (It is correct because error message is not telling anything useful) and lowering number of active transfers between source/dest pair. This affects all users and if transfer not progressing in next 24h, job is marked as failed. In the end this lead to wasted CPU, other users were affected due to one user and their transfers failed.

If error message reported is correct like T2_CH_CERN, FTS3@RAL would not lower number of active transfers, as it is not site issue. [3] is the full log file. if you want to get same error, execute gfal-copy to fnallpc where quota is exceeded.

So until LPC EOS will report this error message[2], it will be considered as site issue which can affect any other users.
