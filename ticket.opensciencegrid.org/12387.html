    Hi,

Here is what I did for the original configuration:

The Torque server is on a different machine but exports server_logs, and server_priv to
the CE so the accounting files are visible.  This works without any issues.

For the CE install:
From bare metal, I install an SL 5.5 image and then YUM update to get security patches,
and reboot.

For the CE I installed osg-ce-pbs and the managed fork jobmanager.

I went through the process of hacking apart my old config.ini and placed the appropriate
pieces in the various /etc/osg/config.d/*.ini files.

I had ran a few jobs before turning gratia on.

I did not run the command by hand but hovered over the system to watch when the cron would
kick off the job.  The initial setup did not include bringing the urCollectorBuffer.pbs
file into the system from the previous OSG installation.

When the cron kicked off, I was continually running 'ps' commands to track what it was
doing.  When I noticed that it was running through all the old data, I started killing
processes to stop the probe which I was eventually able to do.  I then turned off the
cron, deleted items in /var/lib/gratia that seemed to be left over from the run ( but not
in /var/lib/gratia/data, which seemed to be related to very recent jobs).

I then went over to my Torque server and started hiding all the old data files so that
gratia could only see the past few days.  This is what I meant when I mentioned "fiddled
with the logs" in my original post.

I then reran the cron and started seeing the problems that I mentioned.  I then tried
doing things like bringing in the old urCollectorBuffer.pbs file and I could tell that it
was reading the file from the logs in /var/log/gratia, but did not update it.

    
Due to problems with CEMon, I reinstalled the entire system after adding some tweaks
concerning the tomcat user and the straightening out a java JDK issue. Neither of which
should have affected the gratia probe.  After the reinstall things are running correctly.

Let me know if this helps.

    [by Patrick McGuigan]

It looks like something has gone wrong with an osg3 installation concerning the pbs gratia probe.

After the probe began to run, I noticed that I had no way of stopping it.  This is a problem since the accounting logs contain 2+ years of data.  I killed the probe, cleaned up debris, and fiddled with the logs so that only a couple of days worth of jobs would be visible.

This is where things get weird.  On every iteration of the probe execution it will seemingly process all of the grid jobs. After it finishes it does not write the expected buffer file /var/lib/gratia/tmp/urCollectorBuffer.pbs.  I found the buffer file from the last gratia run from the previous installation, put this in the location in /var/lib/gratia/tmp and watched the probe run.  Again it processed all of the jobs up to the job mentioned in the buffer.  Upon completion it did not update the buffer.

It does not seem to be my configuration files, since the probe finds the existing buffer file.

The installed gratia rpms are:
gratia-probe-common-1.12-1.osg.el5
gratia-probe-pbs-lsf-1.12-1.osg.el5
gratia-probe-gridftp-transfer-1.12-1.osg.el5
gratia-probe-gram-1.12-1.osg.el5
osg-configure-gratia-1.0.14-1.osg.el5
