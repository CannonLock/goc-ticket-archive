Dear all,

I think we may be hitting the same scenario described in https://ggus.eu/ws/ticket_info.php?ticket=97607

If those CEs were put for maintenance for a while and published status "queueing" at some point, the FCR mechanism could have blacklisted them. Since the queue belongs only to CMS:

GlueCEAccessControlBaseRule: VO:cms

Once this ACBR is removed, there are no more GlueCEAccessControlBaseRule attributes in the object GlueCEAccessControlBase, which is a schema violation (thus the errors in the log), so the blacklisting cannot actually take place and the status "queueing" keeps on being published by the top BDII.

At some point the status gets bak to "production", but as long as the FCR mechanism blacklists the CE, the same problem persists. So I'm afraid it is not a bug in the BDII itself but a bug in the FCR mechanism.

Actually, after restarting the BDII, the CMS queue is not published at all:

ldapsearch -LLL -x -h lcg-bdii.cern.ch:2170 -b o=grid GlueCEUniqueID=rossmann-osg.rcac.purdue.edu\* GlueCEUniqueID

dn: GlueCEUniqueID=rossmann-osg.rcac.purdue.edu:2119/jobmanager-pbs-ncn,Mds-Vo -name=Purdue-Rossmann,Mds-Vo-name=local,o=grid
GlueCEUniqueID: rossmann-osg.rcac.purdue.edu:2119/jobmanager-pbs-ncn

And this is for the same reason explaind above, the FCR is blacklisting the resource, so the whole CE object is not published as there are no other ACBRs to be published, with the consequent schema violation.
