    Hey Britta,

I would add that various versions of the GIP have had questionable
data quality.  I can't control the versions sites deploy, but
generally believe the data can be more and more trusted moving forward.

The GlueCEPolicyMaxRunningJobs is usually 0 because the GIP didn't
know how to query the batch system for this data (I think this is now
added in Condor in beta releases, for example).  Older versions have a
value of "0", but we're starting to use a string of "9"s to indicate
attributes which are not known (anything more than 4 "9"s should
indicate "not known").

I would suggest that this particular attribute is not useful at most
sites; one thing you'll want to look at is the number of free CPUs and
number of waiting jobs (note that free CPUs at a lot of sites will
never go all the way down to zero due to batch system configurations,
but waiting jobs will definitely increase).

I would recommend submitting to sites with (LIGO waiting jobs < 20)
and doing a fuzzy ranking against using the # of free CPUs, or maybe
(# of free CPUs) / (# of total CPUs + 10).

Gabriele, I would also note that having folks look for "VO:<voname>"
will be dangerous in the future.  GLUE supports access rules by FQAN,
and I'd like to eventually (next 6 months) support this.  My
suggestion is to normalize all the ACBRs into a common format to allow
for adjustments like this in the future.

Brian

    Nov 17, 2008 07:51 PM UTC by Gabriele Garzoglio 
    Hi Britta,
let me cc the GIP group: parts of your questions are more on the
implementation of Glue Schema attributes, rather than on features of
ReSS. OSG GIP will be able to give you the details, but I can already
give you some answers.

First, this might just be a cut and paste issue, but your query below
lacks the final 's' to GlueCEPolicyMaxRunningJobs. This results in
GlueCEPolicyMaxRunningJobs not being printed out. In reality, I've
noticed it by using '%d\n' instead of '%d\t' in the formatting string.

For both GlueCEPolicyMaxTotalJobs and GlueCEPolicyMaxRunningJob, I
believe that 0 is equivalent to a large number: in these cases, there is
effectively no policy limitation on the maximum number of total jobs OR
on the maximum number of running jobs. On the other hand, for example, I
see that both TTU-ANTAEUS and FNAL_DZEROOSG_1 have meaningful limits
forGlueCEPolicyMaxRunningJobs (64 and 250 respectively), which should be
the maximum slot capacity of the queue, for a pbs system. Brian should
be able to give more details on the actual implementation also for other
batch systems.

As for what attributes are specific or independent of a VO constraint,
the general answer is that the attributes in the VOView (for the Cluster
info tree) and VOInfo (for the SE info tree) entities can all override
attribute values common to the CE and Storage Area, respectively. You
can see the attribute list at p7 and p14 of the Glue Schema
specification document:
http://forge.cnaf.infn.it/plugins/scmsvn/viewcvs.php/*checkout*/v_1_3/spec/pdf/GLUESchema.pdf?rev=49&root=glueschema

When general attributes are overriden by VO-specific attributes, they
are kept in the classad with the same name with "Original" appended. For
example, IF GlueCEStateRunningJobs is the VO value, then
GlueCEStateRunningJobsOriginal is the original value. Note that if a
value is NOT overridden, there is no "Original" attribute in the
classad. You can see a discussion on this on the document "Glue Schema

    mapping to old classad format" (p6):
https://twiki.grid.iu.edu/twiki/pub/ResourceSelection/GlueSchemaToOldClassadMapping/glue-v1.3-oldclassad-mapping-v1.1.pdf

In practice, not all VOView and VOInfo attributes are implemented to
have different values for VOs. I know that the important ones, like
GlueCEStateRunningJobs and similar, are different for the VOs vs. the
general values. Brian can probably give the details of all such attributes.

Let us know if you have further questions.
Cheers
Gabriele
