    Dec 15, 2015 05:34 PM UTC by OSG-GOC 
    I am testing an upgrade of our compute nodes that are currently CentOS 6.5 / OSG 3.2.16 to CentOS 6.7 / OSG 3.2.32.  I have noticed that a 'yum update' on my compute nodes is wanting to install the entire tomcat6 stack.  This seems a bit excessive for a compute node.  The package that appears to depend on tomcat6 is jglobus.  It seems the dependency structure from osg-wn-client package is this:

osg-wn-client -> bestman2-client -> bestman2-common-libs -> jglobus -> tomcat6

What's odd is my compute nodes, which are on OSG 3.2.16, do not have tomcat6 installed but do have osg-wn-client-3.0.0-22.osg32.el6 installed.  So sometime between OSG 3.2.16 and OSG 3.2.32 the dependencies changed that result in tomcat6 wanting to be installed onto compute nodes.

Right now a 'yum update' on my compute nodes consists of 501 package upgrades (expected) and 71 installs (unexpected).  I would expect some installs because a few packages got renamed so there are "replace" operations, as well as kernel installs, but 71 seems excessive.  Fortunately my systems do not rely on RAM disk or other technologies that would make lots of packages installed a real problem.

I don't know if this issue has a resolution.  I would be interested to know when or why the dependencies changed to result in tomcat6 getting installed indirectly by osg-wn-client package.

Thanks,
- Trey
