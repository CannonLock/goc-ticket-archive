    Aug 10, 2011 02:36 PM UTC by OSG-GOC 
    There are a lot of issues in this ticket. If I'm following the ticket
correctly, I hope these will help you:

1) Sites are free to choose edg-mkgridmap or GUMS.

2) If a site chooses to use edg-mkgridmap, users who are in multiple VOs
cannot control how they will be mapped at a site: one particular mapping
will be chosen by edg-mkgridmap, and the user will only be mapped that
way. The underlying reason is that a grid-mapfile is not expressive
enough to include the VO information, only the base DN of the user. My
understanding is that a large number of sites (50%?) use grid-mapfiles,
so some users who are in multiple VOs have chosen to have distinct
certificates (with distinct DNs) to ensure that they know how they are
mapped.

Finally:

> I'm not sure. Let me ask VDT, I haven't used edg-mkgridmap much.
>
> VDT,
>
> Is there a way to add a manual mapping to a grid map file that won't be
> overwritten by edg-mkgridmap?

You can add a manual mapping with to edg-mkgridmap. It's done by adding
a "gmf_local" (read "grid-mapfile-local") line to your
edg-mkgridmap.conf that specifies an extension to the grid-mapfile.
Documentation on this can be gotten by doing "man edg-mkgridmap.conf".

If you use this option, it will be used in preference to all other
mappings the user could use. So if you want the original user (Marco?)
to be mapped to SBGrid, the site can use the local grid mapfile option
to force the mapping to SBGrid. That's perfect if Marco only ever wants
to show up as SBGrid to the site, but if Marco wants to be in different
VOs at different times it's insufficient.

    
There are two solutions.

1) Marco can use multiple certificates, one per VO.
2) Sites can use GUMS. GUMS is definitely a better solution, but it's
more complex and an extra service to run. Sites without much effort
devoted to maintaining the site may choose not to run GUMS.

Does this help?
