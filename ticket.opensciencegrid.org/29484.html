    Apr 27, 2016 08:11 PM UTC by mgheith@.... 
    Hello Hiro,

It's important for Fermilab to collect this information so we can track jobs running on, and offsite, for statistical and troubleshooting purposes.  Since all of the jobs that Fermilab launches contain a tool, called IFDH (the swiss army knife for data transferring) it makes sense to be able to make use of the information it sends back to Fermilab, via UDP, to one of our central logging servers.  As of today, I would say 97% of the sites that Fermilab uses allows for UDP packets to be sent to Fermilab via this tool, however we noticed that we are not receiving this log information from usatlas.bnl.gov nodes based on an anomoly a user was having with one of our systems while tracking a job.

Some of the information that is being logged by IFDH via UDP:

-VO

-Site name

-Node name

-User name

-jobsub_job_id (contains cluster and process number)

-CPU type

-Bytes transferred

-Transfer time

-Bogomips

-Files being transferred

-Timestamps on when scripts runs and when they end along with its exit code

Also, Fermilab has currently implemented a system called POMS (Production Offline Management System) for the OPOS (Offline Production Operation Services group) which relies on this data to keep track of jobs.  Currently this system is only used by this group, but eventually all of Fermilab will be using this tool to submit and monitor jobs.  It's important for Fermilab to have IFDH be able to send this information to our central logging server via UDP.  Information is only being sent OUT of the tool.  Eventually this information will be visualized, and available to OSG, so various VO's can see a high level view of things like transfer rates, success rates, etc.

Please do let me know if you need any other additional information from me, or the Scientific Computing Division here at Fermilab.

Many thanks,

Michael Joseph Gheith
