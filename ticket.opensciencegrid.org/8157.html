    This is a problem with GUMS 1.3.17 and the generation of the
osg-user-vo-map.txt file using:
VDT_LOCATION/gum/scripts/gums-host generateOsgUserVoMap

Both client and server are 1.3.17.

It appears to be a problem in processing the mapped users selected from
the MAPPING table in the GUMS database.  I believe these will always be

Show More

pool account users.  The very last user (record) does not get processed
and populated in the osg-user-vo-map.txt file.

This is not an issue of UNIX accounts not existing.  I tested this
independent of those filter processes executing the command as shown in
the beginning.

Not sure if this will help or not, but if I query the GUMS data base
with;
select * from MAPPING order by ACCOUNT
... at least in my instance, the last non-NULL DN never comes across in
the output.

I have done this on 2 of my GUMS servers with the same results.
