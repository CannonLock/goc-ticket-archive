Manoj, that only will work in the case of directly mounted local PNFS access.

You can compare SRM access to access to a FTP server.  If you had two files on a FTP server, the only way to join
them is to download them both locally, join them, then re-upload them.

Specifically, you cannot access SRM in general like you would if it was a network or distributed file system.  In
some specific cases, if you are running a job on a worker node at a particular site (and really, there's a small
number of site set up in this way), you can use the trick that Charles showed you.

    Sep 17, 2008 07:47 PM UTC by Neha Sharma 
    Neha Sharma writes:
Charles

I have not seen this before.

See for example the dCache FAQ: http://www-dcache.desy.de/manuals/FAQ.html

This is also discussed in other places in the dCache documentation.

Can you explain in more detail how and where to use this?

libpdcap is a library which intercepts calls to "open", "read",
"write", etc and replaces them with calls to dCache.  (note
this is libPdcap, not libdcap.  The P is for "preload").

By using

LD_PRELOAD=/path/to/libpdcap.so some-prog

you make some-prog work with direct access to PNFS.  This
is not effficient, but it is good for one-off tasks like
what Manoj is trying to do.

Also note that I do *not* advise setting this in the
environment (I've seen some users do this!) which will
cause all sorts of strange, unexpected behavior, and
will make commands like "dccp" and "srmcp" fail.  But
it does let you use standard Unix utils like cp, mv,
grep, tar, etc on files in PNFS.  Just put
LD_PRELOAD=/path/to/libdcap.so,
on the command-line, ahead of the command you want to run,
just like in the example "tar" command I provided.

The path is probably something like /opt/d-cache/dcap/lib/libpdcap.so,
you should be able to find it.

    
Do *not* set LD_PRELOAD as an env. var.

- Charles
