> Yes, please create a JIRA ticket for fixing the event.srm.log file

I think the unexpected way in which event.srm.log is rotated might be worth
opening another ticket.

However,  this bug report subject is bestman2 filling up partitions b/c of too
much logging.  The first part was addressed by using logrotate.

The second part is that the default value for BESTMAN_EVENT_LOG_COUNT is not
sane.

BESTMAN_EVENT_LOG_COUNT is an integer which controls how many
event.srm.log.[0-9]* files exist. 50000000 is the default value in the code.

BESTMAN_EVENT_LOG_SIZE is an integer indicates the maximum number of bytes
event.srm.log should be before it is renamed to some event.srm.log.[0-9]* and
a new event.srm.log written. I could not find what the default value of this
is in the code, but it too can be overriden by /etc/sysconfig/bestman2
setting.

BUT, the default value is large enough that 50000000 log files will fill up
many partitions.  E.g. if each log file was 1 KB then 50000000 of them would
fill 48 GB.

So I would suggest adjusting BESTMAN_EVENT_LOG_COUNT to around 10
and BESTMAN_EVENT_LOG_SIZE=20971520 (20MB) for a total of 200MB log files.

    Mar 4, 2013 05:16 PM UTC by OSG-GOC 
    Hello,

We've run in to a problem of the partition containing /var/logs/bestman2
running out of space.

I've tracked the problem down to a few issues in how bestman2 rotates it's
logs.

The logrotate configuration for bestman2-2.3.0-9.osg.el5.src.rpm is this:

/var/log/bestman2/*.log {
    weekly
    rotate 10
    missingok
    notifempty
}

However when /var/log/bestman2/bestman2.log is removed by log rotate, the file
continues to be written to because the server started like so
in /etc/init.d/bestman
daemon --user $SRM_OWNER "/usr/sbin/bestman.server $BESTMAN_OPTIONS 2>>
$BESTMAN_LOG  >> $BESTMAN_LOG"
does not close it's file handle.  This can result in the log file filling the
partition.

I fixed this by using the "copytruncate" option of logrotate.

The second issue was with the rotation of /var/log/bestman/event.srm.log*
files.  
These logs in the default configuration are managed by both logrotate and
bestman2.  The result was a confusing mish-mash of event.srm.log* files. I
fixed this by changing the logrotate configuration to only match the
bestman2.log file.
(e.g. /var/log/bestman2/bestman2.log { )


    Bestman2's rotation of the event.srm.log file is controlled by these
parameters:
/etc/sysconfig/bestman2:
# BESTMAN_EVENT_LOG_COUNT=<total number of logs>
# BESTMAN_EVENT_LOG_SIZE=<int>  (in bytes)

I could not find a default value for BESTMAN_EVENT_LOG_COUNT, while
BESTMAN_EVENT_LOG_SIZE if unspecified is 50000000 (bytes).  We had many 44MB
event.srm.log.* which also file the /var/log/bestman2 partition.

Setting the above two parameters, especially the BESTMAN_EVENT_LOG_COUNT to
some sane number might be something to consider.

Thanks!
  Chad.
