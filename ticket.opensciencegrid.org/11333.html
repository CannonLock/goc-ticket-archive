    Dec 9, 2011 01:49 PM UTC by Scott Teige 
    I use our in the broadest sense, the OSG. RPMs are used by the install script (which I'll send along via email). gratiaweb.grid.iu.edu is not covered by an SLA so it's not clear to me that it has the status of a "service" but we keep it running as if it were. None of the software GOC maintains is technically open source. The same is true of the VDT software. Both groups freely distribute software when appropriate but I have the constraint that it is distributed "as is" and I cannot offer support without higher management approval. To send off the files needed for the install script to actually do something would require that I review the files to insure I compromise no passwords or certificates, a procedure that would take sufficient time to qualify as support.

The good news is that Brian Bockleman has told me there is a version specifically designed to be stand alone. He could not recall the name but thought it was "BasicGratiaWeb" or something like that. This is certainly more usable for gratia stakeholders, since what GOC has won't work outside GOC without changes and what Brian refers to is designed for the purpose.

Please close this ticket if this is sufficient.

    First the easy one, gratiaweb1.grid.iu.edu is a virtual machine with 1G of RAM and 30G of disk, configured as a single CPU. So, the hardware requirements are modest.

The documentation for all our services is the install script. We consider this definitive and sufficient. We also have a private TWiki where we document experiences, etc associated with a service. Downloading is from an internal SVN repository and sundry rpm repositories.

Exporting a GOC service for use outside the GOC has been considered before (ticket exchange for XSEDE, MyOSG for MyWLCG are examples) but I feel that this would need to be discussed on a case by case basis at a level outside the GOC. I'm happy to initiate and participate in these discussions but am unable to provide effort to make this happen without discussion.
