    -- by Peter DohertyHello,
In a previous ticket ( 8958 ) it was discussed how sbgrid can manage access to our software and datasets at BNL.

This was suggested by Xin:
"A preferred way is to use a "software" role in your proxy
for installing the software for the whole VO, like other VOs do.
We can have a mapping in our GUMS for a sbgrid VO "software" role attribute,
that way the local account will always be the same
every time some new software needs to be installed."

I'd like to ask that this be implemented.
- What I'd like is for SBGrid users with no VOMS roles/attributes to just be mapped regularly to the BNL pool accounts (grd24733, etc...)
- But I'd like Ian and myself to be able to be mapped to the "sbgrisft" user account when using the "/SBGrid/Role=software" VOMS attribute.
- When not using that attribute, Ian and I should just be mapped to the grd### pool account.

Here are Ian's and my DNs:
/DC=org/DC=doegrids/OU=People/CN=Peter Doherty 869198
/DC=org/DC=doegrids/OU=People/CN=Ian Stokes-Rees 411174

Right now, both our DNs (with or without DNs) are being mapped to "sbgrisft" user.
So $OSG_DATA/sbgrid and $OSG_APP/sbgrid are owned by a user that we can't access, and I can't write to these directories.
- Please recursively change ownership of both those directories and contents to the sbgrisft user, after setting up the GUMS mappings.

I hope this solution will resolve issues we've had with OSG_DATA and OSG_APP at BNL by giving Ian and I shared ownership of these two directories when using the VOMS software role.

Thanks!
