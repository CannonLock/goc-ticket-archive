    Jul 17, 2014 02:17 PM UTC by Scott Teige 
    * Who is affected?
Sites with a job manager other than HTCondor are *not* affected and can ignore this.
The bug affects the Gratia HTCondor probe from version 1.13.24 to 1.13.29 (current production version) and flags all your jobs as Local (non grid jobs).

*  Severity: MILD to SEVERE
- If you kept the default probe configuration (i.e. you *did not* manually edit /etc/gratia/condor/ProbeConfig), then MILD,  you are reporting correct numbers to WLCG (if enabled). If you check your /etc/gratia/condor/ProbeConfig and see a line SuppressGridLocalRecords="0", your level is MILD, your jobs are misclassified but few people look at that level of detail.
- If you changed the probe not to report local jobs (i.e. you edited /etc/gratia/condor/ProbeConfig and changed to SuppressGridLocalRecords="1"), then SEVERE, please take action URGENTLY to apply the workaround below. Without it you are not reporting correctly to OSG and WLCG.

* What you should do:
If you have a HTCondor job manager (very likely if you received this notification), evaluate your severity level (point above).
- If the severity is MILD: you may apply the workaround if you like (please see below) or wait for the new production release next Tuesday (22/Jul).
- If the severity is SEVERE: apply the workaround (below) as soon as possible and contact Gratia support (open a GOC ticket or email gratia-operation@....) for instructions to recover the jobs that have not been reported.

* Workaround
Edit the probe configuration (/etc/gratia/condor/ProbeConfig) and add the line:
NoCertinfoBatchRecordsAreLocal="0"
It can be anywhere in the ProbeConfiguration element, e.g. right below the line EnableProbe="1".
This workaround will make your reports correct: Grid jobs are correctly classified as Grid jobs and reported correctly to WLCG
Local jobs are not reported as you desired

* Solution
gratia-probe-common.noarch 1.13.30-3 and gratia-probe-condor.noarch 1.13.30-3 are currently in osg-testing and will be part of an urgent release next Tuesday 7/22.
Everyone (using the HTCondor Gratia probe) is recommended to upgrade to that release.
The upgrade will involve a regular update of the package (yum update) and a merging of the new configuration file (/etc/gratia/condor/ProbeConfig.rpmnew) with your current one (/etc/gratia/condor/ProbeConfig):
- this will add the line NoCertinfoBatchRecordsAreLocal="0" to your configuration file
- if you applied the workaround above you can simply delete the /etc/gratia/condor/ProbeConfig.rpmnew after the yum update

* Additional details
The bug is discussed and tracked in JIRA, https://jira.opensciencegrid.org/browse/GRATIA-149

* More details for the curious ones (skip as you will)
The bug in xml_utils.py, CheckAndExtendUserIdentity() (part of gratia-probe-common). If the job history has already the VO information (FQAN), like HTCondor jobs do, then the certinfo file (files created by the gatekeeper to associate VO information with a job) is not used to extract the information, the attribute 'has_certinfo' is not set and the job is considered local. You can see this in the Gratia database. Anyway all jobs with VO information are used for WLCG reporting, so if they are in the Gratia database, they are counted even if labelled as Local.

    The new version of the probe correctly identifies the presence of the certinfo files anyway it is good to set NoCertinfoBatchRecordsAreLocal="0" because HTCondor does not really need certinfo files and they may go away in the future.
The bug is discussed and tracked in the Jira ticket https://jira.opensciencegrid.org/browse/GRATIA-149

    Here is an URGENT configuration change that will fix the reporting while we are working on changes to the probe:
in /etc/gratia/condor/ProbeConfig within ProbeConfiguration (e.g. right above EnableProbe="1") add the line:
NoCertinfoBatchRecordsAreLocal="0"

Optionally you could do also the following in the same configuration file:
- I'd recommend to increase DataFileExpiration to 30 days if you have a shorter value, e.g.:
DataFileExpiration="31"
- You can put SuppressGridLocalRecords back to 1 if you had so before this ticket:
SuppressGridLocalRecords="1"

This will restart the regular reporting.
If you have quarantined records, you can move them back to the data directory after you made the changes above:
mv /var/lib/gratia/data/quarantine/* /var/lib/gratia/data/
