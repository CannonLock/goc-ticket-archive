    Hello,
Yes, Dave is correct, I should have updated this ticket.  As I said in the original posting, my job environment setup script was doing many wget's for small files that are part of the job execution environment.  They are not shipped with the job because they change on a regular basis, typically 1-2 times per day, but at this low update frequency they are better housed in the /cvmfs filesystem.  This change has been made.  The startup script now looks for these files in /cvmfs/oasis.opensciencegrid.org/gluex and only attempts to wget them if they cannot be found there.  As soon as this change was put into place, the wget failures on rcac.purdue nodes disappeared, and job startup is now succeeding 100%.  I think the reason for the timeouts before was contention for the squid during periods of rapid job startup.  This ticket has been resolved by an improved strategy by the user, and can now be closed.
thank you, -Richard Jones

    Update on this issue of failing http gets to http://zeus.phys.uconn.edu through squid2.rcac.purdue.edu:

I would still like to understand the reasons for the failed transfers, but I now believe that my job startup script was putting too much stress on the local squid.  I just counted up 17 individual files that were being fetched over http (through the squid proxy) from my httpd server on every job startup.  These are mostly static files (change twice per day, max) so the right access method would be over the cvmfs filesystem.  Yesterday I put a change in place, so that the script first attempts to find the file on the /cvmfs filesystem and only if it fails, to fetch the file from the http server.  This ultimately results in the same basic mechanism (latest copy fetched over http from central service) but I believe that the caching that /cvmfs uses is much more intelligent than what I was doing, which was hitting the squid on every access.

Follow-up questions: Do you see the load on the squid from Gluex jobs is looking more reasonable now?  Might this solve the issues I was having, or do I need to keep looking?

thank you, -Richard Jones
