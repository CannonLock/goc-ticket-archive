machine, and the cert/proxy never leaves the machine it was generated for).

Prakashan,

You have two options: using a user certificate's proxy (for example,
your own cert) or using an RSV service certificate. The choice is
explained in this section of the RSV configuration guide:

https://twiki.grid.iu.edu/bin/view/MonitoringInformation/InstallAndConfigureRsvAdvanced#Should_I_use_a_service_certifica

(or in the pre-requisite list of the CE install guide:
https://twiki.grid.iu.edu/twiki/bin/view/ReleaseDocumentation/PreparingComputeElement#RSV_Prerequisites)

Can you decide which way you'd like to go, and let me know? and we can
take it from there. I'd recommend using a service cert since  you're
only monitoring one resource.

Yes, you'd have to add a DN for that cert to your gridmap file if that
is what you use for authorization (and not GUMS, etc.).

Btw, I assume you stopped osg-rsv till you figure all this out on your
CE. Is that correct?

Cheers,

Hi Prakashan,

Sorry about the delay in getting back. The key is whether or not the
rsvuser is able to run globus jobs on the CE using the rsvproxy. I think
your RSV user is not able to run globus jobs on the CE.

If you'd like you could try the following:

su - rsvuser ## or whatever username your RSV client runs as
. /path/to/rsvOrCe/setup.sh
export X509_USER_PROXY=/path/to/rsvproxy
globus-job-run saxon.hosted.ats.ucla.edu:2119/jobmanager-condor /bin/date

You could also try running one of the probes by hand:

cd $VDT_LOCATION/osg-rsv/bin/probes
perl osg-version-probe -u saxon.hosted.ats.ucla.edu -x /path/to/rsvproxy
--verbose

Thanks,
