    Hi Yujun,

-onwrite=off and -ontransfer=on

instruct dCache pool to calculate checksum in flight and not at the end of the transfer.

Therefore there must be no time spent for checksum calculation at the end of the transfer that would
trigger client side timeout.

Andrew,

I looked at the transfers "how there's timeouts when FTS shows that all
the bytes ended up on the destination site", especially the one showed
up in this ticket.
Basically, dCache pool has set to do checksum policy (csm )

csm set policy -onread=off -onwrite=off -onflush=off -onrestore=off
-ontransfer=on -enforcecrc=on -getcrcfromhsm=off

This means checksum will be enforced when doing the gridftp transfer.
For example, for the transfer shown in [1]. All the bytes finished at
Wed Oct 26 13:48:56 2016, but the checksum need to be calculated before
the transfer was complete (Post-processing). This can take longer time
when a file is large (and the server is very busy). Anyway, this was
