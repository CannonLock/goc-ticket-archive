Hi John

One of my colleagues Hyunwoo is trying out the command line clients for GUMS for some of his testing.

The version he is using is:

gums-client-1.3.17
gums-service-1.3.17

these come from vdt-2.0.0 install.

In process of helping him, we found 2 issues

1) If user proxy doesn't exist, the client dies in peace and does not complain

example:

-bash-3.2$ gums mapUser -d -f /fermilab -s '/DC=org/DC=doegrids/OU=Services/CN=fermigridosg1.fnal.gov' '/DC=gov/DC=fnal/O=Fermilab/OU=People/CN=Neha Sharma/CN=UID:neha'
-bash-3.2$

so it makes one wonder, what happened?

2) The value of FQAN specified in the command line, gets dropped and doesn't make it to server

example:
-bash-3.2$ gums mapUser -d -f /fermilab/Role=NULL/Capability=NULL -s '/DC=org/DC=doegrids/OU=Services/CN=fermigridosg1.fnal.gov' '/DC=gov/DC=fnal/O=Fermilab/OU=People/CN=Neha Sharma/CN=UID:neha'
LocalId[userName: fgtest]
-bash-3.2$

corresponding entry in server logs

12 Apr 2012 10:54:25,807 [INFO ]: GridID[/DC=gov/DC=fnal/O=Fermilab/OU=People/CN=Neha Sharma/CN=UID:neha] - Mapped on host '/DC=org/DC=doegrids/OU=Services/CN=fermigridosg1.fnal.gov' the user '/DC=gov/DC=fnal/O=Fermilab/OU=People/CN=Neha Sharma/CN=UID:neha' / 'null' to 'fgtest'


    A consequence of this is, that GUMS maps a user to a local user that is unexpected/incorrect

I am wondering if these are known issues and if a timeline is already determined to have them resolved.

Thanks!
