    It seems to be working fine now as I see the deletion coming in .

Although I am not 100% sure, this is probably the reason.

The BNL webdav point, dcgftp.usatlas.bnl.gov is a alias for 18
different
hosts to provide a redundancy .
This can be easily seen by dig dcgftp.usatlas.bnl.gov which returns
the
18 IPs. (see below).
However, unfortunately, Rucio deletion for BNL must be using one
physical host.   In this case, the client (Rucio deletion service
hosts
for BNL endpoint) host will cache the result of the DNS lookup to be
particular one host using the nscd service in that host.    In this
particular case,  if Rucio deletion was hitting one particular host
(out
of 18) which is having a issue, the problem will stay forever until
the
DSN cache is expired.  And, I think this was the case, we had one host
(dcdoor15/192.12.15.236) with the minor network issue.   Although the
host is not used for the production transfer (not included in dCache
SRM
selection), it is still included in the DNS of dcgftp.usatlas.bnl.gov.
Therefore, if the deletion was hitting this host and this host only,
it
will keep failing.  Anyway, the network of this host is fixed.  But,
meantime, I can see that the deletion service is now hitting dcdoor17.
Anyway, it is working.
