    Aug 14, 2015 01:04 PM UTC by Scott Teige 
    The solution I proposed (1c below) is the only short term solution available. Internal discussions have converged to the point where I can give the following list of options with *estimated* times:
(the "I" in the below is Dave Dykstra. I, as in Scott Teige, have no opinion as to the relative virtues of these solutions. That is entirely up to you) The timescale on these is sufficiently long I believe
the additional administrative permission issues are no longer critical path.
---
1. Those that don't require a client change:
a. Ask ILC to rename the repository to ilc.egi.eu.  They
would either have to host at RAL or RAL would need to add the
ability to host the repo remotely.  The latter is not yet
supported, but the egi.eu CVMFS administrator at RAL would
like to support it and I explained to him how to do it.
b. Ask ILC to rename the repository to ilc.opensciencegrid.org.
This would set a new precedent of having a remote OSG
repository in Europe.
c. Re-publish the contents of the repository as an oasis repo
subdirectory.  I discourage this option because I believe we
should be moving away from that older mechanism for a number
of reasons.  The reason most affecting the user is that it
requires completely relocatable code.
2. Those that require a client change once:
a. Use the cvmfs-2.1.20 mechanism of a configuration CVMFS
repository by defining one in the cvmfs-config-osg rpm.  We
would need to decide where and how to host it, whether at the
GOC or at Fermilab, and who would be responsible for
maintaining it, whether the GOC or the OSG Security team.
Also note that even though the default non-OSG installation
of cvmfs-2.1.20 from both CERN and from EGI's UMD yum
repository configure a cvmfs-config.cern.ch repo for this,
they have not yet chosen to put anything in it, including
ilc.desy.de.  So if we were to consider this route we should
collaborate with CERN and EGI to see if they agree.
b. Use a new mechanism from John Hover for distributing client
configurations authenticated by x509 PKI.  I don't understand
the details, and Jose said in any case it is not a short term
solution since code hasn't yet been written.
3. Option that would require a client change every time a new domain

    is added:
-  Add the desy.de domain public key to the cvmfs-config-osg
rpm.
---
Dates:

1a.) 1 week, Catailin (RAL admin) is pretty responsive, plus response time from ILC to get the work done.
1b.) Depends on response time from ILC to get the work done.
1c.) Immediate.
2a.) Several weeks. No sooner than next release date (8/Sep) probably the one after that (13/Oct)
2b.) Long term, months? Add delay due to release schedule.
3.) Same as 2a. There may be resistance to implementing a one-off solution.
