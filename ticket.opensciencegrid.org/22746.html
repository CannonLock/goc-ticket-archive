As an experiment, I also tried doing the upgrade when the sysconfig file was actually modified, and, in that case it does still preserve the modified version (and dropping in the .rpmnew). Otherwise, a packaging workaround would be to make bestman2-common-libs own the shared config files instead.

Carl
    Oct 17, 2014 07:10 PM UTC by Carl Edquist 
    Hi Horst,

Thanks for reporting this.

Yes, that line did change from 2.3.0-18 to 2.3.0-20, but I'm also surprised that upgrading caused the .rpmnew file to be created.  I tried it and observed the same behavior, even though the old and new ones are clearly the same, with identical mtimes/ctimes.  That's interesting about multiple sub-packages owning the file -- that very well may be what is confusing yum/rpm.

I remember reading in the rpm packaging guidelines that multiple subpackages are allowed to own the same file, provided that they're all from the same source package build (and as a result are also always identical files).  But it looks like the %config(noreplace) mechanism doesn't work so well for this case  :)

Probably we can refactor the packaging a bit so only a single subpackage owns them.
