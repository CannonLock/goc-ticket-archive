    Hi Horst,

Lets start debugging why slurm wouldn't send reports for completed jobs.

Multiple steps to do the first bit of debugging.

1. Increase the debug level of the Probe.  In /etc/gratia/slurm/ProbeConfig, look for the line: LogLevel="2".  Increase the 2 to "10".
2. Set a time back in history.  In the file /var/lib/gratia/tmp/checkpoint, edit the file to make the contents just 1493856000.  That should the start of 5/4/17, in unix epoch.
3. Run the probe by hand: $ /usr/share/gratia/slurm/slurm_meter -c

The probe may run for a while.  Once the probe has completed, upload the complete log file for today located in /var/log/gratia/ somewhere that we can get to it.  It may be large.

    > So how do you get to the "Batch" records instead of "Payload"from the main site summary page,

There's a drop-down near the top labeled "Job Type."

> I might argue that that should be changed to Batch, if not all sites show Payload records ...

Yes, I agree. It was originally, then we changed it at some point because for sites which have a lot of opportunistic usage the Payload records provide more interesting information, like the organizations and field of science. I think a fair compromise is to set the default to batch, but have the organization and field of science tables fixed to Payload.

> How do you get to this Gracc page from the main page, by the way?

From the main page (or any page), click on the page title in the upper left corner (e.g. "GRACC Home"), and you'll get a browsable and searchable list of all the dashboards. We only highlight the more commonly used ones on the main page.

> And what's the difference between batch records [1] and job records [2]?

The difference between my two links was the timestamp used for the timeline. [1] showed when the records were received, while [2] showed when the jobs took place (specifically, ended).
