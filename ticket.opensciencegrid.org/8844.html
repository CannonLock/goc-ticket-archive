    GUMS bug
---------
This GUMS bug was identified as the cause of an authorization problem at
CalTech.  There were 2 goc tickets that were issued as a result of this bug:
https://ticket.grid.iu.edu/goc/viewer?id=8750
which is also related to this ticket:
https://ticket.grid.iu.edu/goc/viewer?id=8816

Caltech (CIT_CMS_T2) is running GUMS 1.3.14.  The problem also exists in
GUMS 1.3.17 as that is where I re-created the problem.

Problem statement
-----------------
In a User Group, if you have..
Match VOMS certificates FQAN as "vo"
and do not specify a VO/Group attribute (it says it's optional),
then any User Groups defined in the Host-to-Group-Mappings that follow that
mis-configured User Group will NEVER get authorized.

It appears to hit this condition and then terminate.

Important point
---------------
It is not just the user group that has this invalid condition
that results in  authorization problems.  It is all the user groups that
follow it during the evaluation process of the host-to-group-mappngs.

GUMS behavior with this condition
---------------------------------
1. authorization request is received.
2. service requesting the authorization (gatekeeper in this case), receives
a "DENY" response.
3. There is no log message in gums-service-admin.log of the DENY
4. There is no java stacktrace in catalina.out indicating anything.


    In summary, you are clueless as to the problem.

Additional comments
-------------------
Either this is a bug in the edits being performed on the UserGroup element or
this is a problem in evaluating an authorization request with these
attributes... or both.

This may be a problem with other combinations of attributes as well.

John Weigand
