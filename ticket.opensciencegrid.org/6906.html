    May 20, 2009 07:33 PM UTC by Neha Sharma 
    Authorization takes place some time after a connection is made to the server and a secure context is
established. None of those previous steps have anything to do with gPlazma code. So if they are not happening,
it's hard to see how any gPlazma settings have an effect. Can you confirm that the problem occurs again, that
only changing the useGPlazmaAuthorizationModule and useGPlazmaAuthorizationCell lines in dCacheSetup
causes the error. I know you are in production, but you should be able see the effect immediately, so at worst
you'd be down for only a minute. If changing dCache setup, having the print level set to 3, and executing
"/opt/d-cache/bin/dcache restart srm" brings back the srmcp failures, then please send the log of the startup
to me.

On the document page, the only mention of 1.7 is to say that delegation used to be the default and starting in
1.8 it was no longer the default. I think for the other parts the version dependence is pretty clear, since the
headings are "Validating User Attributes in dCache 1.8" and "Validating User Attributes in dCache 1.9". Also,
both sections say that the default for validation is false, so you should be able to have a working gPlazma without
validation.

If you have the same configuration as the doors and authorization on the doors is working, your setup is
probably OK.

Ted
