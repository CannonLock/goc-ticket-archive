Hi Ruth,

I've been slow to respond to this because I don't know how to summarize
this properly. Let me try now...

- The ultimate goal of the RSV team (and the OSG management presumably)
is to get every OSG resource to be monitored by site-level RSV clients

-- However, I realize that there are constraints:

1) Really small sites may not have enough man power to actually
install one more service

2) RSV v1 was a bit flaky; v2 was better; and I expect v3 to be a lot
more stable -- this will help get more site admins interested in
installing it.

    
- Given the above scenario, Rob Quick wants to get sites to get
interested, and *want* RSV (instead of his or top management throwing a
mandate that they need to install it); Rob, please correct me if I am
mis-speaking

-- We've had quite a few admins who've thanked us for providing RSV as
a service -- and that it helped them identify problems, etc. We need to
get that word out more.

- I'd personally be ok if OSG mandates sites to install RSV v3 when we
release it [early summer] OR if admins just fall in love with it, or
both :-)

- In the list you provided:

1) there are a couple of sites where RSV clients had stopped, and we
have tickets open to get them restarted; NERSC has told me via email
that they plan to install RSV sometime soon to monitor all their OSG
resouces;

2) I do not have any resource in particular that I think should RSV
installed -- see point above about how I want it installed everywhere :)

- Indeed some resources like VOMS servers will not be monitored by
site-level RSV clients ... at least not in the current setup (We -- GOC
-- will use RSV in some form to montor VOMS servers ... )

Does this all make sense? Would you like more explanation?: Would you
like me to call and explain this on the phone?

Cheers,
Arvind
