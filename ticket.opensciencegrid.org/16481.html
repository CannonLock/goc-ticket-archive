    After upgrading to OSG 3.1.22, an "old" problem I fixed reappeared
and I realized I never submitted a ticket on this bug

I believe there is a bug in the nagios-consumer python module

/usr/libexec/rsv/consumers/nagios-consumer

I believe the line in the process_record section which is

PLUGIN_HOST = record.get("serviceURI", record.get("hostName", ""))

should really be

PLUGIN_HOST = self.config.get("RSV", "RSV_HOST")

What happens is that the script is pulling out the hostname from the gratia record.

For most of the probes, the serviceURI hostname is the same as the gatekeeper running the probe
However, for SRM probes, this entry is the SRM server name with the port attached.

For example, in our case, the serviceURI for an SRM probe running on osg-gk.mwt2.org would be

uct2-dc1.uchicago.edu:8443

When the script tries to use this as the host name in the report to Nagios....well...it does not work.
At least not the way I think it should.

If the probe was against the gatekeeper itself, the serviceURI would be the gatekeeper name,
in this case, osg-gk.mwt2.org (sans any port) and all those probes would report to Nagios as
expected under that host name.

My change above pulls the RSV_HOST from the /etc/rsv/rsv-nagios.conf file and
reports to Nagios using that name, which is what I suspect was intended??

Also, even if I did create a Nagios host/service for uct2-dc1.uchicago.edu:8443, then I would have other problems.

    
We have four gatekeepers running RSV and probing the SRM.

Each is at a different geographical site (UChicago, Indiana, Illinois), thus
we want to monitor access from each site.

If all the GKs  reported the results of their probe to the same host/service, they would overwrite each others results.
If one GK had a problem probing the SRM, but the others were ok, I might not see the problem in Nagios.
Or the service might flap back and forth from ok to bad to ok to bad to ok...etc....

I have put this fix in place on my OSG 3.1.22 gatekeepers (MWT2 and MWT2_CE_UIUC) and they now report correctly again

thanks, dave
