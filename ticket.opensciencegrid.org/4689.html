    Hi Alain-

On Fri, Mar 28, 2008 at 09:58:43AM -0500, Alain Roy wrote:
The problem is that installation-time environment variables don't
work as well as I would hope. This is because they need to be set
if you ever update the package, not just when you install the
first time.

That's correct. Our local documentation is clear on this point, and
this is actually more or less how I'd expect things to work. Caching
environment variables could lead to unexpected nastiness.

I think that modifying the environment setup is the the simplest
and most reliable method. You can do it in a way that doesn't get
stomped on by Pacman. Pacman normally rewrites setup.sh any time
you run a Pacman command.

When we noticed the addition of DCACHE_IO_TUNNEL, we added a script
to post-setup to unset it. This is an acceptable solution. I created
the ticket because I couldn't find any announcement of the change
and wasn't sure if it was intentional/expected. It's an entirely
legitimate choice to make and we're happy adjusting our installation
to accommodate it.

Thanks for your help!

    Hi Will

This is a very tricky situation -because if this variable is not set, then users accessing the gsidcap doors will
complain that dcap door is broken!

The setting of this variable was done to make user environment more friendly for grid-related purposes. It is my
understanding that, if users are contacting a dcap door in a grid environment, it could be for both read and
write operations and therefore  the dcap doors will have to support gsi based  authentications. This is why this
variable has been set.

If the door is a plain dCap door (as in case of your site) , this variable has to be unset.

    Hello, GOC-

We just installed the (relatively) new wnclient package at our site.
After the installation, we noticed that dcap was broken. Upon
further examination, it became clear that the Unsup-Dccp package had
been configured to set DCACHE_IO_TUNNEL in setup.sh:

#
#-- package
/afs/hep.wisc.edu/osg/wnclient:http://vdt.cs.wisc.edu/vdt_181_cache:;Unsup-Dccp
#
#
#-- package
/afs/hep.wisc.edu/osg/wnclient:http://vdt.cs.wisc.edu/vdt_181_cache:;Unsup-Dccp-Questions
#
DCACHE_IO_TUNNEL="/afs/hep.wisc.edu/osg/wnclient/dccp/lib/libgsiTunnel.so"
export DCACHE_IO_TUNNEL
PATH="/afs/hep.wisc.edu/osg/wnclient/dccp/bin:$PATH"
export PATH

This is appropriate only at sites using gsidcap; on other sites
(such as ours), setting this variable to the path to libgsiTunnel.so
will cause dcap failures. Our previous installation did not have
DCACHE_IO_TUNNEL enabled in setup.sh, and we were not prompted to
enable or disable it during wnclient installation.

The Unsup-Dccp-Questions package only appears to contain licensing
information; at least, that appears to be its function when reading
its filelist. For now, we've worked around this problem by adding a
simple 'unset DCACHE_IO_TUNNEL' to a script in the post-setup
directory.

Is DCACHE_IO_TUNNEL set unconditionally? Or did we somehow miss the
prompt for it?


    Thanks!
