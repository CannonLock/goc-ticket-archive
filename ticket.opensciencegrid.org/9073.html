On Aug 5, 2010, at 11:07 PM, Alex Sim wrote:

Hi Wei,

The most immediate cause of running out of file descriptors is the
number of concurrent connections and the number of certificates for GSI
authN, and this has been addressed a few times in OSG already by
increasing the file descriptor limit. We have not seen unclosed pipes
cases yet.  Junmin will take a look at the logs, but bestman itself
doesn't open many file descriptors and it'll hard to find any causes.

As for the queues you need to use the full mode by the configuration
options. Gateway mode would not support queues.
If you deploy bestman2, Jetty has a way to queue certain amount of
connections until active connections are done. bestman2 package has
configuration options for those although it's not for the bestman itself
but just passing those to jetty. You may want to take a look at those
options.

Regards,

On 08/06/2010 12:38 AM, Yang, Wei wrote:
Hi Tanya, Alex and Junmin,

As LHC began taking data we are starting to see intensive, simultaneous activities of
data transfer and user analysis. Two bestman (gateway) related issues occurred recently
that US ATLAS would like to seek help from OSG storage and Bestman team.

The first issue is that we see large number of unclosed pipes. This eventually exhausts
the file descriptors and prevents bestman from calling external adler32, etc. ("Too
many open files" in bestman log, see attached bestman log and lsof output of
bestman/java). Those unclosed pipes aren't accumulated slowly over time. We suspect
that something triggers the situation but we couldn't tell what is it.

At SLAC and UTA, adler32 calculation and space reporting are performed via external
commands (and therefore use pipes). UTA only saw this issue once, caused by a hanging
NFS server that hosts one of this external command script. SLAC saw this issue multiple
times. By the time I look at them, those external commands were all accessible and
finished but lsof showed more than 1000 open pipes, and bestman no longer functioning.
While there are things SLAC can do to make external commands lighter (such as those at
NET2 and UTA), we think bestman should be able to close those pipe and continue in this
situation.

The second issue is about SRM request queues. In the past, because we worried about the
performance of bestman-gateway itself (and other reasons), the request queues were
removed from bestman-gateway. In reality, we think that the performance of bestman-g
itself isn't an issue, but the impact of backend storage to bestman-g starting to be an
issue --- when the storage is extremely busy, it may not be able to respond to
bestman-g's requests in time. So US ATLAS would like to request backporting the
queueing functions to bestman-g, as an option that can be turn on and off. The goal
here is to allow us to transfer as many parallel files as possible but protecting
bestman from timing out when the storage is busy.
