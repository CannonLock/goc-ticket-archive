    Sep 20, 2006 03:52 PM UTC by Jerry Gieraltowski 
    This ticket can be closed iff the following change is made to the file
srmclient/etc/config.xml.template

<!-- true or false-->
<debug> true </debug>

!-- true or false-->
<debug> false </debug>

The user can easily turn debugging on in the srm-cp command by merely
specifying the option "debug".  This parameter if set to "true" by default
will cause a lot of unnecessary output when debugging is not desired.

All other changes to the installation of srmclient have already been made in
OSG-4.0.1

Jerry Gieraltowski

    Aug 14, 2006 06:29 PM UTC by Jerry Gieraltowski 
    I just installed VDT-Client from http://vdt.cs.wisc.edu/vdt_1311_cache/
This installation completed successfully and created two directories for srm
services:
srm_v1_client
srm_v2_client

I assumed that the directory srm_v1_client points to the older version of
srmclient tools and srm_v2_client points to a completely new and updated
version of the srmclient tools.

If you execute "which srmcp" you will see that it points to the
srm_v1_client install path.

If you source the setup script in $VDT_LOCATION and then try to run "srmcp"
It will FAIL!!

The failures are caused by:
1) The variable SRM_CONFIG which points to the configuration file
srmcp expects is no longer being set in the setup script.  This variable
should have been defined in the setup scripts as:
SRM_CONFIG=${VDT_LOCATION}/srm_v1_client/etc/config.xml

2) In the file $SRM_CONFIG several variables are defined as relative
to the variable ${VDT_LOCATION} but this is an xml file and the variable
VDT_LOCATION needs to be expanded prior to using the xml file.  This used to
be done by an install-srm script which is no longer available.

If I make the changes pointed out above and rerun an "srmcp" command,
everything works fine.

Am I missing something here or was this package not tested as well because
it was meant to be replaced by the srm_v2_client package in the near
future??

Jerry Gieraltowski - ANL
