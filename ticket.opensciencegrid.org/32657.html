    Apr 7, 2017 08:14 PM UTC 
    Kyle,

As I said, the problem with gfal-copy started after I did a system "yum update".  I have not done another "yum update" since then.  I did, however, implement a work-around, as follows:

From a backup of /usr/lib64/gfal2-plugins taken before I did the yum update, I retrieved the following:

libgfal_plugin_srm.so, libgfal_plugin_file.so, libgfal_plugin_xrootd.so, and libgfal_plugin_gridftp.so

I created a new directory on my running system, /usr/lib64/gfal2-plugins/old, and put the retrieved files there.  I also retrieved the entire /usr/lib64 from the backup and saved that as /usr/lib64/gfal2-plugins/old/lib64 on the running system.

I then replaced /usr/bin/gfal-copy on the running system with a wrapper script that sets GFAL_PLUGIN_DIR to /usr/lib64/gfal2-plugins/old, sets LD_LIBRARY_PATH to /usr/lib64/gfal2-plugins/old/lib64, and then executes the real gfal-copy script.

I had to create similar wrapper scripts for gfal-ls and gfal-rm.

It seems to me that the yum update I did must have resulted in a version mis-match of some sort between the gfal-copy setup and the gfal2-plugins, but I'm not expert enough at using yum to know what to do to debug this further.

So, at the present time my system is working properly, but my solution is not very satisfying.
