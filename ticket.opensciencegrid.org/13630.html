The osg_display script is now aggregating only 12 months of data (instead of 13 months), although it is still off-by-one month from my own view - it is aggregating data between 2012-01-01 and 2012-12-31, instead of 2012-02-01 and 2013-1-31, but I think it's ok.. The result is now much closer to the one reported in gratiaweb, although not identical.

display uses following query to pull data

SELECT
MIN(EndTime) AS time,
SUM(Njobs) AS Records,
SUM(WallSeconds)/3600 AS Hours
FROM (
SELECT
ProbeName,
R.VOCorrid as VOCorrid,
MIN(EndTime) AS EndTime,
SUM(Njobs) AS NJobs,
SUM(WallDuration*Cores) AS WallSeconds,
YEAR(EndTime) as Y,
MONTH(EndTime) as M

    FROM MasterSummaryData R FORCE INDEX(index02)
WHERE
EndTime >= %(starttime)s AND
EndTime < %(endtime)s AND
ResourceType = 'Batch'
GROUP BY Y, M, ProbeName, VOCorrid
) as R
JOIN Probe P on R.ProbeName = P.probename
JOIN Site S on S.siteid = P.siteid
JOIN VONameCorrection VC ON (VC.corrid=R.VOcorrid)
JOIN VO on (VC.void = VO.void)
WHERE
S.SiteName NOT IN ('NONE', 'Generic', 'Obsolete') AND
VO.VOName NOT IN ('unknown', 'other')
GROUP BY Y, M
ORDER BY time ASC

> 2013-02-07 14:21:46,189 - DEBUG - {'endtime': datetime.datetime(2013, 2, 1, 0, 0), 'starttime': datetime.datetime(2011, 11, 1, 0, 0)}

<sql>

<filler name="where"> AND WallDuration > 0 </filler>
</sql>

I couldn't quite figure out how this configuration is translated to the actual SQL query, but maybe Ashu can answer that?
