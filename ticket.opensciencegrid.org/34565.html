    Aug 15, 2017 03:49 PM UTC 
    After further testing, I find that ipv6 works out of the box, so the problem is much more obscure than I previously thought.

It turns out that we were passing an additional argument to globus-gridftp-server:

-hostname ::

Prior to globus-gridftp-server-control-3.6-1.1, this did not break things.  From globus-gridftp-server-control-3.6-1.1 and on, it resulted in the problem that I reported.

Removing the custom option '-hostname ::' from the command-line arguments passed to globus-gridftp-server made the problem go away.  Then it is not necessary to set data_interface as we were doing as a workaround.

Presumably, the -hostname argument had some purpose related to getting IPv6 to work in some old version of the software.  We don't have a good record of exactly why that was and what version it was for.

The software works well out of the box, so I have no further issue here.  Sorry for crying wolf!

--Dan

    Aug 1, 2017 09:47 PM UTC by OSG-GOC 
    We upgraded globus-gridftp-server today to the latest in OSG 3.3 (3.3.14 to 3.3.26).  There was a change of behavior that resulted in IPv6 gridftp transfers failing.  The change in behavior might related to the following OSG patch:

https://jira.opensciencegrid.org/browse/SOFTWARE-2413

... but I haven't read the code carefully to see if that is actually true.

The change in behavior that we observed was that when using IPv6, the data channel address sent by the server to the client was "::", which caused the client to fail to connect.  We were not configuring the data channel.  We were relying on the default behavior.  We found that the problem was solved by explicitly setting data_interface in the /etc/gridftp.conf.  Example:

data_interface 144.92.181.234
data_interface 2607:f388:101c:1000::210

Here is example output from globus-url-copy when the problem happened:

debug: response from gsiftp://g19n24.hep.wisc.edu:2811/store/user/dan/test_file:;
229 Entering Passive Mode (|2|::|10743|)

After applying the configuration workaround, the corresponding output was:

debug: response from gsiftp://g19n24.hep.wisc.edu:2811/store/user/dan/test_file:;
229 Entering Passive Mode (|2|2607:f388:101c:1000::210|15212|)
