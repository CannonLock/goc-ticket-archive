    Nov 29, 2011 02:21 PM UTC by Soichi Hayashi 
    I believe this issue is still discussed outside of this ticket. I am extending NAD one more week.
    Nov 17, 2011 03:41 PM UTC by Soichi Hayashi 
    I have discovered that GOC Ticket was not processing correctly the notification contents formatted in non-unix line delimiters. Jim used to use his linux laptop to send the notification, but now Kevin is submitting the security email using his Mac PC causing the content to have "invalid" line delimiter which are translated to extra newlines.

I have made modifications it so that it will convert all non-unix delimited form inputs to unix format. Although I consider this is a bug, I'd l like to wait to release this patch until Dec/13 so that I can test this more adequately (unless someone really wants this to be released ASAP).

Currently, Kevin has to manually sign the message by first converting his message to unix format, then using gpg command line to generate the signed content, then copy & past to GOC Ticket notification form. I think the whole process to be cumbersome and error prone (as this incident has demonstrated). Also, GPG signatures are not widely accepted by various email client.

Meanwhile, GOC Ticket has a built in capability to send S/MIME signed emails using GOC's DOEGrid user certificate. S/MIME is far more standard, and if security team agrees, I recommend that we start sending out security notification using this signature. Only the OSG security staff has access to the security notification form, so I believe it is almost as equally secure as using current security team GPG key.

Soichi

Other types of announcements that originate from ticket1/2 don't have this problem.  They're also not signed with GPG; they're signed with an X.509 DOEGrids certificate.  The code that handles security announcements must be different from the code that handles other types of announcements.
