    Hi Neha,

It turns out that vacuum (autovacuum and vacuumdb both) will not work
correctly if you don't have max_fsm_pages set high enough.  The error
message looks like:

NOTICE:  number of page slots needed (55280) exceeds max_fsm_pages (20000)
| HINT:  Consider increasing the configuration parameter "max_fsm_pages"
| to a value over 55280.
This message will appear on the command line for vacuumdb, and in the
log files if you're using autovacuum.

Here's a reference from the Postgresql documentation:
http://www.postgresql.org/docs/8.1/static/runtime-config-resource.html#RUNTIME-CONFIG-RESOURCE-FSM

To fix it, I edited the postgresql.conf file, uncommented the line
#max_fsm_pages = 20000
And bumped up the value, then saved and restarted postgres
/etc/init.d/postgresql restart
Then I manually ran vacuumdb again.
vacuumdb -z -U postgres --all

For the largest database, I had to repeat the process several times till
I hit on a high enough number.  Once I did, looking at the analysis
output indicate a huge number of unused item pointers, which I'm given
to understand indicate a large number of empty fsm pages.  I re-ran the
vacuum in full mode, which allows it to freeze the tables and
consolidate the files on disk.
vacuumdb -z -f -U postgres --all

The final vacuum freed up ~10G.

- --Sarah
