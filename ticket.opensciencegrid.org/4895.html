Okay, so I've attached the two perl scripts. The first, submit.pl, is used to submit
a series of <Njobs> jobs to a WS-GRAM service on <ExecutionHost> using <FactoryType>
as WS-GRAM factory type. Each job would simply run the command /bin/hostname and
exit. The temporary RSL script generated by submit.pl tells WS-GRAM to also redirect
stdout/stderr into files and stage those out to another host <StageOutHostAndDir>
after job completion:

> ~/tmp> ./submit.pl
> This script submits a series of jobs to a WS-GRAM service and writes the jobs'
> EPR files into the current working directory.
> The jobs' stdout/stderr logfiles are staged to another Grid resource.

    >
> Usage: ./submit.pl <Njobs> <ExecutionHost> <FactoryType> <StageOutHostAndDir>
>
>        where
>          <Njobs                 number of jobs to submit
>          <ExecutionHost>        host to submit jobs to
>          <FactoryType>          job manager factory type for job submission
>          <StageOutHostAndDir>   hostname and directory to stage logfiles to
>
> Example: ./submit.pl 10 udo-gt01.grid.uni-dortmund.de PBS
udo-gt03.grid.tu-dortmund.de/home/ad/ad0006/tmp/

For instance, to submit 20 jobs to BNL_ATLAS_1 and stage the logfiles to UWMilwaukee
you would say:

./submit 20 gridgk01.racf.bnl.gov Condor nest.phys.uwm.edu/opt/osg/home/ligo/tmp

The script saves each job's EPR as returned by WS-GRAM in a file
<ExecutionHost>-<jobID>.epr in the local directory where you called the submit.pl
script from.

While job submission usually works fine, the more interesting (and error-prone) part
is to periodically query the status of submitted jobs. That's what our GEO600
application does every so often in order to find out when and how many new jobs may
be submitted. The script status.pl is doing the same; it takes a list of EPR files
and queries the jobs' status one after another:

> ~/tmp> ./status.pl
> This script checks the status of submitted jobs identified via their EPR files.
>
> Usage: ./status.pl <list of EPR files>
>
> Example: ./status.pl udo-gt01.grid.uni-dortmund.de-*.epr
>

For instance, to check the status of jobs submitted to BNL_ATLAS_1 you would say:

./status.pl gridgk01.racf.bnl.gov-*

You probably have to run this several times because it takes a while until have jobs
have finished. Ideally all of them should be reported as 'Done' in the end.
Practically though it happens (to me at least) that errors are reported for some
jobs, like in the following:

> Checking status of job 'gridgk01.racf.bnl.gov-2.epr':
>   Current job state: Failed
> globusrun-ws: Job failed: Staging error for RSL element fileStageOut.
> Error authenticating user at source/dest hostServer refused performing the request.
Custom message: Server refused GSSAPI authentication. (error code 1) [Nested
exception message:  Custom message: Unexpected reply: 421 Idle Timeout: closing
control connection.] [Caused by: Server refused performing the request. Custom
message: Server refused GSSAPI authentication. (error code 1) [Nested exception
message:  Custom message: Unexpected reply: 421 Idle Timeout: closing control
connection.]]

Apparently there was a problem in the file stage-out phase (the application has
finished already!) where an operation timed out. I believe this should _not_ happen,
and I would like to know whether you encounter the same problem. If so then we have
something for the Grid admins to test with, I guess.
Interestingly, when you check the status of those jobs again after a day or so (as I
did today) the error is different:

> Checking status of job 'gridgk01.racf.bnl.gov-2.epr':
>   globusrun-ws: Error querying job state
> globus_soap_message_module: SOAP Fault
> Fault code: soapenv:Server.generalException

This is precisely the error message I'm seeing in our GEO600 application. Just a
general exception, no indication whatsoever about the reasons for the job's failure.
It would be good to know whether the failure of some of our E@H jobs is is related to
file-staging problems, and _not_ to a bug in the application code.
