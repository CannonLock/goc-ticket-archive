    Dec 23, 2015 12:32 AM UTC by Carl Edquist 
    Ok, so the way I read that, the RES=538m is the resident memory for the process, which (as also listed) is 6.8% of the (8G) system memory.  VIRT is less important, and it includes files mapped to memory that aren't necessarily taking up physical ram.  (I think VIRT can actually legally exceed the total system memory.)

But it makes sense that it happened again.  I would expect the "java.lang.OutOfMemoryError: Java heap space" error to only happen after RES for the java process actually hits 1024m (or whatever BESTMAN_MAX_JAVA_HEAP is set to), regardless of the memory allocated to the VM...

It may actually go a little beyond 1024, since there is some java overhead besides the heap, I am not sure of the exact details, but I think the heap size is the biggest factor.

I guess I would try setting BESTMAN_MAX_JAVA_HEAP to something higher but not the entire VM's ram (maybe 1/2 or 3/4?) and see if the problem continues.

Thanks,

Did you adjust BESTMAN_MAX_JAVA_HEAP in /etc/sysconfig/bestman2 ?  (Or just increase the memory allocated to the VM?)  Because if you just increased the allocated VM memory I'd expect it to still have the same 1024M limit for the bestman2 server process (java).

Unfortunately we don't have any known minimum memory requirements, other than the default of 1024M.

When you said before "its usually close to max memory used all the time", did you mean the bestman2 server process (java) is close to the BESTMAN_MAX_JAVA_HEAP setting, or the VM's memory use is close to the capacity for the VM?
