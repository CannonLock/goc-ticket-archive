    Jun 24, 2010 05:29 PM UTC by Michael Thomas 
    There is a known bug in earlier versions of gums that causes errors when
you have a generic grid proxy rule (no fqan, maybe?) listed before a
full voms fqan matching rule.  The bug generates a null exception in the
gums server, and does not return a mapping back to the client.  Moving
the generic grid proxy entry in the hostToGroupMapping to the end of the
hostToGroupMapping list can work around this problem.

Our version of gums (1.3.14) is known to contain this bug.  We have
plans to upgrade our gums server to the latest version, and at the same
time will be setting up a HA gums configuration.

--Mike

    Jun 24, 2010 12:56 PM UTC 
    This is an update of what has been done so far, at least the parts I was involved with.  I don't believe this has been fully resolved as yet.

The error they are getting, "Failed to talk to the Identity Mapping Server", I have only been able to re-create when the GUMS service is not available.   When we looked at the GUMS logs, there was no indication that the request was getting to GUMS from the CE.  There were no stacktraces in the catalina.out file either.  This means that they were really not getting there or GUMS was getting them and silently  rejecting them.

These were a couple of the things we tried on the CE/gatekeeper side:

1. The OSG version was 1.2.8, however, the gsi-authz.conf and prima-authz.conf were referencing the /osg/1.2.6 path. This should not have been a problem but we corrected to insure that.

2. The gsi_authz.conf was referencing the xacml protocol.  In talking to the Dave Dykstra (PRIMA), testing has shown that xacml is 5 times slower than the older protocol.  Although, the gatekeeper message did not indicate a timeout occurred (and I think they do if that is the case), we changed the gsi-authz.conf and the config.ini file to use the older prima callout.

Both of the above were done on both CE nodes at CalTech.

We also determined that glidein use of glexec was not in effect.

Late Friday, Mike emailed me this as a partial resolution:
"I made one change last night that may have exacerbated this problem.  In
order to try to help fix some CMS SAM test problems, I changed the
hostToGroup mapping order for two VOs, placing them ahead of the CMS
mappings.  I had Dorian revert this change, and so far it seems to have
fixed the CMS gums mapping problem, though we continue to have SAM test
failures."

As of Monday, 6/22, I was advised that all was fine.  I am very curious as to what the problem really was in the event it happens again.  The re-ordering of the group mapping should have had any affect on the errors being seen.
