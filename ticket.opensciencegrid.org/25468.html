    Rob,

Yes it is preventing /cvmfs/icecube.opensciencegrid.org from mounting on worker nodes.  Rolling back will not fix the problem because it is a matter of an expired certificate.  Also I don't think cvmfs clients will allow the transition of the oasis repository back from cvmfs-2.1 to cvmfs-2.0.

The real fix is for the icecube repository server cvmfs-stratum0.wipac.wisc.edu to download a new copy of http://oasis.opensciencegrid.org:8000/cvmfs/icecube.opensciencegrid.org/.cvmfswhitelist

A workaround at the GOC might be to do this on oasis-replica:
# scp oasis:/srv/cvmfs/icecube.opensciencegrid.org/.cvmfswhitelist /srv/cvmfs/icecube.opensciencegrid.org

The only thing is I'm not sure if the next cvmfs_server snapshot iceceube.opensciencegrid.org, run from cron within the next 15 minutes, will overwrite it or not from what it reads from cvmfs-stratum0.wipac.wisc.edu.  That could be avoided by temporarily commenting out the CVMFS_STRATUM0 line in /etc/cvmfs/repositories.d/icecube.opensciencegrid.org/server.conf, causing the snapsot for that repo to fail each time.

    I happened to notice while working on oasis-replica-itb that the icecube.opensciencegrid.org repository has a bad .cvmfswhitelist with a signature that does not match.  It does not match the one at http://oasis.opensciencegrid.org:8000/cvmfs/icecube.opensciencegrid.org/.cvmfswhitelist that it is supposed to download daily.  This means it will not download to oasis-replica.  Also the .cvmfswhitelist currently on oasis-replica expired a few minutes ago so now /cvmfs/icecube.opensciencegrid.org cannot be mounted on clients.  (A partial cause of this may have been that from Tuesday through yesterday, the icecube repository server in Wisconsin would have gotten download errors from where it gets the file at oasis.opensciencegrid.org, and the administrator may have tried to work around that somehow).
