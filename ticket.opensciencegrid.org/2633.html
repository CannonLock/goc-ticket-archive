I had trouble generating a new http service certificate for GUMS today, because
cert-gridadmin generated a certificate that didn't work.  Is this confusion on my
part about how to use cert-gridadmin, or is it a problem with the tool?

Here is how I created the certificate:

cert-gridadmin -email dan@.... -affiliation osg -vo cms -host ${host}
-service http

This generated a certificate with the correct DN, but when I installed it, both PRIMA
and my browser refused to connect to the GUMS service, because they didn't like the
certificate:

PID: 2733 -- PRIMA ERROR  prima_soap_client.c:68  Error receiving SOAP response PID:
2733 -- PRIMA ERROR  prima_soap_client.c:87  SOAP FAULT Code:   SOAP-ENV:Client
PID: 2733 -- PRIMA ERROR  prima_soap_client.c:88  SOAP FAULT String: SSL_ERROR_SSL
error:14090086:SSL routines:SSL3_GET_SERVER_CERTIFICATE:certificate verify failed

Eventually, I tried generating a certificate using doegrids-cert-request and the
web-based Grid Admin Interface to DOEGrids.  This produced a certificate that worked.
The only apparent difference between the two is this:

certificate generated by cert-gridadmin contains:
X509v3 extensions:
Netscape Cert Type:
SSL Client, S/MIME

certificate generated by doegrids-cert-request and web-based GridAdmin Interface
contains:
X509v3 extensions:

    Netscape Cert Type:
SSL Client, SSL Server, S/MIME

I do not know if the presence of "SSL Server" in the X509v3 extensions is required,
but since this is the only noticeable difference between the two certificates, I
assume it is.

--Dan
