Now, the FQAN of the user job proxy in this case would be /des/Role=Analysis (and from the DES VO, not the Fermilab VO, despite the fact that the jobs run inside Fermilab VO pilots.) Carl added a manual map when we saw this problem in https://ticket.opensciencegrid.org/29770, but the FQAN of this job is different, so the fix may not apply. While we were having these problems, we had similar issues at other sites and might have stubled on the true source of the problem. Does your gums config have the following in it?

<vomsUserGroup
name='des'
access='read self'
description=''
vomsServer='des'
matchFQAN='vo'
acceptProxyWithoutFQAN='true'
voGroup='/des/production'/>

<vomsUserGroup
name='des-production'
access='read self'
description=''
vomsServer='des'
matchFQAN='exact'
acceptProxyWithoutFQAN='true'
voGroup='/des/production'/>

If so, then I think that is the problem. The voGroup for the first entry should just be '/des' instead of '/des/production'. I think this got into the gums template file at some point as well, so if you've never modified that part, you might well have it. In the meantime I'll work on getting that fixed in the template file in the next release. As an aside, if you do have it, then it is probably what caused the problem in the other ticket too. If you don't have that in your config, let me know and we can move on to the next possibility.

Thanks,

Ken
