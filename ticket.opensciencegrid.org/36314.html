    Mar 26, 2018 02:29 PM UTC by Thomas Lee 
    We use Puppet to distribute users' SSH public keys to the machines that they should have access to, and it would appear that Puppet wasn't running on oasis-login-itb due to a certificate mismatch.

Puppet certificate mismatches prevent a machine from connecting to the Puppet server; there's a certificate, but it isn't the one that the Puppet server associates with that machine. This means that someone in the recent past has brought down the oasis-login-itb VM that was authorized to connect to the Puppet server and brought up a different one, new or old, that wasn't authorized, and didn't notify me -- either because I was out on PTO or because, when this happens, there simply isn't any immediate indication that the problem exists.

I've fixed this specific issue -- I didn't even see a /home/zvada on oasis-login-itb before, and now there is one, with a .ssh/authorized_keys file containing the same zvada key that is on jump. Marian should try ssh again. The greater issue still needs work, but that's beyond the scope of this ticket.
