    Hi all, sorry for the delay updating this.

I can't find a way in the probe to bump up the verbosity for these warnings/errors, short of adding debugging to the probe itself. BUT...

It looks like the WARNING message (GRACC server ... not responding) is a bug in RSV - it is always shown when there are any hits from the last two days, rather than on a curl error.  (So we need to fix this in RSV.)

The "Some information could not be gathered due to lack of privileges." message is apparently intended to suggest that there might be more useful information if the probe were run as root (instead of the rsv user), but I doubt that is actually the case for this version of the probe, since gracc support was added.  So i wouldn't bother with running it as root.

As for the ERROR message you got (Probe ... has not been seen ... in >1d), that looks like it is (or might be) being honest, as the timestamp for the log message was:

timestamp: 2018-03-15 15:32:05 CDT

but in the json output you provided, the received timestamp is 2018-03-13T21:55:22.907Z (more that 1d).  And if i run the curl command now (with the month adjusted) it also seems to be more than a day old.  Not sure if that indicates lag on the gracc end or genuinely that there is nothing there for whole-day windows.

I'm not sure exactly what the "org.osg.gratia.metric" metric is about, as opposed to "org.osg.gratia.condor".

Anyway it's looking like at least one fix is in order for this rsv probe, and the question for this site in particular is whether it makes sense or not that there are 24 hour gaps in received timestamps for records...
