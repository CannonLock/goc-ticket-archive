>>> On 10/29/2012 09:39 AM, Mátyás Selmeci wrote:
>>>>
>>>> Ugh, not empty-condor again. I'm really starting to hate that thing.

    >>>> Cc'ing Tim because I have a policy suggestion below.
>>>>
>>>> Steve's instructions are the best way of fixing your install--get rid of
>>>> empty-condor, then install the real condor, then osg-ce-condor.
>>>>
>>>> The technical reason this probably happened:
>>>>
>>>> * Like the instructions say, you installed osg-client-condor first,
>>>> which is of architecture "noarch".
>>>> * It needs the dependency "condor", which both empty-condor and condor
>>>> satisfy. The yum dependency solver looks at both of those and scores them
>>>> according to some heuristics.
>>>> * empty-condor is noarch, whereas condor is either arch "i686" or
>>>> "x86_64".
>>>> * Therefore, empty-condor gets points for matching the architecture of
>>>> the package that is trying to bring it in. empty-condor will end up with a
>>>> higher score than condor, so that is what yum will pick to install.
>>>>
>>>> I believe that by changing the empty-condor package to no longer be
>>>> noarch, I can adjust its score to no longer be above condor. I can't really
>>>> change osg-client-condor to get the "real" condor though.
>>>
>>> I was wrong. Changing the arch of empty-condor not only does not fix
>>> osg-client-condor, it _breaks_ osg-ce-condor. This is because if the arch
>>> matches, then yum goes on to the next step of its heuristic: "give more
>>> points to the one that brings in the fewest additional packages", which
>>> empty-condor easily wins.
>>>>
>>>> To elaborate on that:
>>>> * yum/rpm has no "suggests" mechanism; that is, I cannot say
>>>> osg-client-condor prefers one condor package over another.
>>>> * I could change osg-client-condor to depend on something that
>>>> empty-condor does not provide, but that would break installations that
>>>> really _do_ want empty-condor.
>>>>
>>>> Policy suggestion:
>>>>
>>>> * Create a separate repo for "fake" packages -- packages that claim to
>>>> provide a dependency but are lying about it, like "empty-condor".
>>>> * Move the fake packages there, and mark the repo as disabled by
>>>> default.
>>>> * Adjust the documentation for people that have their own installations
>>>> of condor to enable that repo before installing empty-condor.
>>>>
>>> I also tried a variation of this: create a separate repo, have it enabled
>>> by default but with a lower yum priority. That did not work because yum
>>> priority is only considered for packages with the same name.
>>>>
>>>> That will not break people's existing installs -- empty-condor simply
>>>> won't update for them. It _would_ break things for people that automatically
>>>> install empty-condor from the osg repos when provisioning their machines
>>>> (via anaconda/kickstart/puppet etc.)
>>>>
>> I don't particularly like having to do this, but it seems to be the best
>> option we have.
>>
>> What are all our "fake" RPMs?  The CA certs and the Condor?
