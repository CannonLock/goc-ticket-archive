    Aug 5, 2009 08:22 PM UTC by OSG-GOC 
    First thing I should mention is that this is a production problem in OSG 1.2.

The service providers that need to look at this are RSV and configure-osg support.

I am trying to trace down a problem I am seeing when configure-osg is run and the site_name attribute of the config.ini is changed.

The configure-osg script is executing the ./vdt/setup/configure_gratia script twice:
1. once for the job manager probe (e.g. condor, et whatever)
2. and again for the metric probe

In both cased the --site argument is used so both metric and condor ProbeConfig's get updated correctly.

There is a 3rd execution of configure_gratia that I am guessing occurs in the RSV code.  This one executes...
/usr/local/osg-ce/vdt/setup/configure_gratia \
--probe metric \
--report-to rsv-itb.grid.iu.edu:8880 \
--probe metric \
--data-dir /usr/local/osg-ce/gratia/var/rsv/data \
--working-dir /usr/local/osg-ce/gratia/var/rsv/tmp \
--log-dir /usr/local/osg-ce/gratia/var/rsv/logs \
--log-rotate 7 --log-level 1

Note that the --site argument is not used.

The configure_gratia script, if the --site argument is not provided, defaults to using the OSG_SITE_NAME variable in the osg-attributes.conf file. If the osg-attributes.conf does not exist yet, it will further default to using the hostname and puts out a warning message saying this is a problem.

At the point in time this 3rd execution of configure_gratia is performed, the osg-attributes.conf file has not been updated.  Hence, the ./gratia/probe/metric/ProbeConfig file reverts back to the old SiteName.

If this is an initial installation, it is likely that the osg-attributes.conf file will not exist and the metric ProbeConfig will be populated with the hostname.

John Weigand
