Hi Charles,

It is an interesting exercise, but since I did not try this out
myself, I have no idea how to help you with making tomcat use its
native library. But I do not think it is going to make a big
difference. We performed profiling of the srm server and found that
more then 90% of the cpu power is spent in globus gsi ssl code,
performing authentication, delegation and encryption. Globus is
using PureTLS library that has a native implementation http://www.rtfm.com/puretls/gonative.html
. It might be interesting to try to use that. Gerd Behrman tried to
compile the library and he could not do it on latest linux, and he
developed a patch addressing this. In case  you are interested in
trying this out and testing the performance of the globus code with

    Native PuteTLS lib, I will forward his email to you.

Thanks,
Timur

Since SRM performance is an issue, I thought I'd take a stab at
installing this.  I followed the directions here:

http://tomcat.apache.org/tomcat-5.5-doc/apr.html

I found the source tarball at
/opt/d-cache/libexec/apache-tomcat-5.5.20/bin/tomcat-native.tar.gz

After installing the apr-devel-1.2.8-6 RPM and upgrading to gcc4, I was
able to sucessfully compile and install tomcat-native-1.1.3.  I used
--prefix=/usr in my configure command, so the library got installed as
/usr/lib/libtcnative-1.so (symlinked to /usr/lib/libtcnative-1.so.0.1.3)

However, when I restart SRM I still see the same message:

INFO: The Apache Tomcat Native library which allows optimal [...]

I have 2 questions here:

A) What am I doing wrong?  Has anyone successfully run the SRM server
with the native library?

B) Is this even worth doing?  Should I expect a performance increase
from this?  Reading about the native library, I see that it does
things like use "sendfile" to transfer static files.  But the SRM
server doesn't serve a lot of data... so will the native library
help?
