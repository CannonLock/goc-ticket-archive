    May 20, 2016 03:39 PM UTC by Brian Lin 
    Horst,

All the scripts you need are included on your CE (or any machine that
you have the htcondor-ce-client installed)! I really want to update the
troubleshooting docs with my notes and procedures but I haven't had the
time to collect it all in an easy to consume form. The way I grabbed
your logs was through condor_ce_run:

1. Verify permission to run at a site with condor_ping
(https://twiki.opensciencegrid.org/bin/view/Documentation/Release3/TroubleshootingHTCondorCE#condor_ce_ping).
You can actually run condor_ping instead of condor_ce_ping if you're
specifying -name and -pool.

condor_ping -name grid0.oscer.ou.edu -pool grid0.oscer.ou.edu:9619
-verbose WRITE

2. Submit a local universe job, which runs directly on the CE like a
globus managed fork job, with condor_ce_run:

condor_ce_run -lr grid0.oscer.ou.edu:9619 /bin/sh -c 'cat
/var/log/condor-ce/SchedLog' > SchedLog

Lots of the condor tools take -name/-pool options so you can use them
against remote CE's.

Cheers,

    Apr 29, 2016 08:57 PM UTC 
    Hi Horst,

Yep so it means that hat job has been in the held state 39208 seconds which is less than 3600*24=86400.

The Condor-Ce puts them on hold based on this knob:

condor_ce_config_val SYSTEM_PERIODIC_HOLD

Normally it is setup like this:

SYSTEM_PERIODIC_HOLD = (x509userproxysubject =?= UNDEFINED) || (x509UserProxyExpiration =?= UNDEFINED) || (time() > x509UserProxyExpiration) || (RoutedBy is null && JobUniverse =!= 1 && JobUniverse =!= 5 && JobUniverse =!= 7 && JobUniverse =!= 12) || ((JobStatus =?= 1 && CurrentTime - EnteredCurrentStatus > 1800) && RoutedToJobId is null && RoutedJob =!= true)

Then the removal decides when to remove them:
condor_ce_config_val SYSTEM_PERIODIC_REMOVE
(JobStatus == 5 && CurrentTime - EnteredCurrentStatus > 3600*24)

The defualt is they have been held for more than 48 hours.
