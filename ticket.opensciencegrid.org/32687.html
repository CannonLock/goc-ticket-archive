    Update:  I have checked the various host service status history (e.g. https://perfsonar-itb.grid.iu.edu/WLCGperfSONAR/check_mk/index.py?start_url=%2FWLCGperfSONAR%2Fcheck_mk%2Fview.py%3Fhost%3Dpsds1%26view_name%3Dhostsvcevents ) to get a feel for what is breaking in terms of monitored limits.

Using this kind of information I made changes to the configurations as follows

1) All service checks support and option to "delay" service problem notification:   "This setting delays notifications about service problems by the specified amount of time. If the service is OK again within that time, no notification will be sent out."   I have enabled this to have reasonable limits for future use of notifications (I setup delays for 10, 20 or 30 minutes depending upon the service).

2) All checks by default change the "hard" state of the service based upon 1 check.   I have reconfigured the services that are having intermittent problems to require N checks to change the hard state of the service.
i)  I was seeing "stale" NFS mounts daily on /net/nas01/Public.  I set the check to require 8 checks to change the "hard" state (1 check/minute so 8 minutes to "recover" from stale)
ii)  Similarly for checking required process counts for rsv, stomptcl, simplevisor and MaDDash I now require 10 checks to change the "hard" state
iii)  The BI (Business Intelligence) rules for both Production and ITB were changed to use the "hard" state in their checks

3) The rsv count on the production instance (psds0) was changed from 8 min, 420 warn, 500 critical to 8 min, 600 warn and 800 critical.   This should cover the "normal" operating range

4) The rsv count on ITB was reconfigured from 2 min process down to 0 because there is not enough "work" to keep this process always running (so '0' is OK)

5) The amount of process CPU used by MaDDash is changed from 200% warning 300% critical to 250% warning 350% critical

These changes were in place by around 12:50 PM Eastern time today (Feb 16, 2017).    Let's watch how this works overnight and see if additional tweaks are required.

Thanks,

Shawn

    Hi Scott,

As you know the perfSONAR services have a number of components that must all work together to deliver a working top-level system.  The check_mk BI rules were used to determine the high-level summary state of the set of nodes and services that are need to deliver the system.    I have tuned the individual checks in the past based upon feedback from Operations and I would like to continue to do so.   The problems check_mk are finding are real issues that will cause the service not to operate correctly *if they continue*.   It seems that some of the services slip into and out of bad states.   This may indicate a problem that ultimately would need code changes to fix.

I would like to propose that we try to better configure the set of states to be less sensitive to short "failures" that we see usually recover.    I agree that the service BI summary rules should only go critical when a person needs to intervene and I believe we just need to tune the configuration with that goal in mind.  Let me try to revise the checks to provide a more robust indication of problems needing intervention.

Let me know if you have suggestions or questions.   Thanks,

Shawn
