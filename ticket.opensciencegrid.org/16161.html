    Hi all,

I'm looking into this now. I'll post this both in the ticket and in JIRA, and we can do the rest from JIRA.

1) What VO is this cert currently in, so I can run some tests?

2) Everyone should be careful about assuming this is a "bug". I'm looking at several aspects of the string matching code, and reading X.500/X.509 string matching specifications. It is clear from some comments in the code that this behavior has been changed back and forth, and I recall discussions at one point with Mine about the security of matching against multiple certs that differ only in consecutive whitespace, and/or leading & trailing whitespace. The X500 spec says that multiple consecutive whitespaces are to be 'insignificant', but that is talking about comparing one certificate to another. From a security standpoint, this actually says to me that CAs should never issue multiple certs that would collapse to the same string, i.e. differ "insignificantly". In deciding on GUMS behavior, we may have decided to interpret X.500/509 strictly, meaning we collapsed all strings to their shortest form, and reject any actual subject DN that doesn't comply with what *should* be CA policy.   Mine and/or Von should probably think about this and explicitly explain what *they* think GUMS should do. Should it be strict, but leave the possibility of this problem? Or should it be lax, and open up the possibility of authorizing a DN that is not actually in a VO (e.g. "user name" is in VOMS, but "user  name" is not). Or should be implement something more complicated, but which may introduce corner cases we haven't considered yet?

3) It *appears* to me that GUMS code in the current version of GUMS 1.3.18.009 and the 1.3.x HEAD is *not* manipulating whitespace within the stored DNs. As I said, I see commented code from some point in the past that *did* collapse multiple consecutive whitespace characters. Once I know Bockjoo's VO, I'll check directly in the database to see what is ending up there. There is a very real possibility that the VOMS client libraries used by 1.3.18 may be trying to follow the X500 spec as they download user certs from VOMS, while the ones in earlier GUMS versions may not have.

4) Whatever desired behavior for GUMS is decided on, any changes will take time to go through the OSG release procedure. Even if we released a "fixed" version of GUMS tomorrow, there is nothing that will force sites to upgrade. So if Bockjoo wants this to work soon, he should request a new certificate with no leading or trailing whitespace, nor consecutive spaces, in the CN.

5) I definitely think the  OSG CA request mechanism should trim and collapse all strings provided during the request process. Von will have to comment on whether this is possible.

Cheers,

--john

    Hi,
I have recently obtained a DigiCert certificate to be used for my T2 operations.
Thus, I used extra characters during the certificate application to indicate that
it's for the T2 service.
The DN is
/DC=com/DC=DigiCert-Grid/O=Open Science Grid/OU=People/CN=Bockjoo Kim  (UFlorida T2 Service) 118
There are two white spaces between Kim and (.
When I applied the certificate, I had an option to add more strings and I meant to add:
'(UFlorida T2 Service)' between Kim and 118 with a single white space.
But apparently one extra white space was entered between Kim and (.
This did not present a problem during the certificate request.
This did not present a problem during the VOMS registration.
Once the certificate was registered in the VOMS, I checked the certificate with various
US CMS T2 sites.
The cert worked fine except for SPRACE, UCSD, and Vanderbilt.
I had an authentication error with these three sites.
It turns out UCSD has the gums 1.3.18 and updates database only with the single white space.
Later, I had the same issue with FNAL and it turned out they have 1.3.18 GUMS as well.
So, apparently, this seems to be a bug in GUMS 1.3.18 because the certificate works perfectly fine
with all other CMS sites world-wide.

An ugly workaround (I personally don't want to obtain a new CERT with a single white space)
is to insert a new entry of the DN with the correct number of white spaces as presented in
the voms-proxy-info output.
Otherwise, a simple update of the existing entry of the DN as is generated with the mysql db id
is being wiped out and my cert is not authorized with these GUMS 1.3.18 sites.
So, I had to virtually harass these sites to make them add my DN.

At the first order, the DigiCert application web page needs to ensure there is only a single white space
or something like that to prevent happening this again for the gums server 1.3.18.
More importantly, GUMS bug needs to be fixed ASAP.
At the moment, I don't want to obtain a new CERT as this is a GUMS bug.
