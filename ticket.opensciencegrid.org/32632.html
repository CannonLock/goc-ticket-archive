    Mar 16, 2017 03:59 AM UTC 
    Hi all,

I think this is not resolved yet, unless uboone can make use of very short-lived (45 min) glideins.

Gianfranco,

You mentioned that we're not passing a requested wall clock time directive to SLURM when the glideins are submitted the site, which in turn leads to a default 1hr limit being assigned to glideins. Therefore, any glideins that attempt to run longer than this 1hr default will be killed by SLURM. We've seen this happen at other sites before many times. To fix we either have to (1) figure out how to pass a requested wall clock time from the factory to ARC-CE to SLURM. Unfortunately, we've never been able to figure this out for ARC-CEs; or (2) like I mentioned earlier, define a separate SLURM queue we can target with a default wall clock time assigned to glideins, which will in turn be configured at the factory to live within that default time you are assigning to them at the site. This is what we do for all ARC-CEs sites currently.

Until we can resolve the wall clock issue, I've reduced the GLIDEIN_Max_Walltime = 2700 seconds (45 min) to ensure the glidein won't be killed off by SLURM anymore. We usually need a buffer to ensure the glideins clean themselves up properly once the user jobs are done. Hence, the 15 min difference between max glidein walltime and SLURM 60 min default wall time.
