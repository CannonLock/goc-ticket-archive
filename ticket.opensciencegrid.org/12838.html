    Hi

I ran into a funny problem when I updated my bestman service
certificate. I renewed it the gridcert-admin tool using the usual flag
--service=bestman. I then installed the cert and key in the proper place
properly and found that my SE wouldn't authenticate anymore. I
eventually traced the problem through the event.srm.log file to the new
cert which identifies the host as
'/DC=org/DC=doegrids/OU=Services/CN=bestman/srm.hep.fiu.edu couldn't be
used to map user, below is part of log file with error message:
ts=2012-11-10T04:42:17.343Z level=Info
class=gov.lbl.srm.util.ValidatorControl
caDir=/etc/grid-security/certificates tid=qtp8604923-22
ts=2012-11-10T04:42:18.661Z level=Excep
class=gov.lbl.srm.util.GUMSXACMLClient XACML server error: Cannot map
user for host
'/DC=org/DC=doegrids/OU=Services/CN=bestman/srm.hep.fiu.edu' - it is not
defined in any host to group mapping.
org.opensciencegrid.authz.xacml.client.MapCredentialsClient.mapCredentials(MapCredentialsClient.java:37)
gov.lbl.srm.util.GUMSXACMLClient.getMappedID(GUMSXACMLClient.java:201)

So what I did was to generate a new cert/key pair this time NOT using
the --service but still following same procedure to install as bestman
service cert. So now the cert/key do not have bestman in the subject.
This solved my problem and the SE can now authenticate.

So the question is given that this is probably not the way to do this
how does one modify the GUMS config to accept bestman service
certificate. I tried this first but GUMS is pretty much a black box to
me. I know that this SE was working before I renewed the cert/key and
most likely I did use the bestman service flag, however I can't be sure
I did as I deleted the old cert.

Anyone can shed light into this?


    Jorge
