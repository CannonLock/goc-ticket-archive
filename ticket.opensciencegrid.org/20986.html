> OSG Security Contacts,

> The OSG Security Team wants to update you on reported issues with
> the Digicert CA CRLs.
>
> Earlier this week, there was an issue with the "next update" field
> in Digicerts signing certs certificate revocation list
> being set to 3 days. Some sites were still seeing the old CRL after
> running fetch-crl. It appears that fetch-crl was waiting for the
> separate, longer http expiry value, which was still set to 7 days.
> We believe this was causing fetch-crl to keep the cached CRL instead
> of fetching a new one. It appears deleting fetch-crl's cache files

    > located in /var/cache/fetch-crl/* and then re-running fetch-crl
> will fix the problem.
>
> It is also possible to run fetch-crl without state information by
> adding "stateless" to the fetch-crl config file, usually
> /etc/fetch-crl.conf.
>
