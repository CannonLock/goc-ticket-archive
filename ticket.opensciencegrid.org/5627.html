    Sep 30, 2008 09:31 PM UTC by Neha Sharma 
    If you have a file /dcache/pool/data/<PNFSID>, you can
sometimes determine the original pathname in /pnfs
by examining the "storage info" file
/dcache/pool/control/SI-<PNFSID>

This file is a serialized Java class but you can usually recognize
the pnfs pathname if you just use "cat" or "strings" on the SI
file.  Alternately, you can use this little Python script:

------------------------findpath.py--------------------

#!/usr/bin/env python

# $Id: findpath.py,v 1.3 2008/09/29 18:48:59 cgw Exp $
import sys, os

control_dir="/dcache/pool/control"
f=sys.argv[1]
data=open(control_dir+'/SI-'+f).read()
i=data.find("path")
if i<0:
print "Error, cannot find path in control file", f
sys.exit(-1)

p=data.find(chr(0),i+1)+1
l=ord(data[p])
if l<5: ##must be high byte of 2-byte word.
##this heuristic works assuming 4 < len(filename) < 1024
p += 1
l = l*256 + ord(data[p])
p+=1
path=data[p:p+l]
path=path.replace('//','/')
print path


    -------------------------------------------------------

(this lives at http://repo.mwt2.org/viewvc/admin-scripts/dcache ,
you might also be interested in the 'poolcleaner' script  there)

Note that the "SI" file is created when the file is originally
copied into dCache, and if the file is subsequently moved,
it will not be updated.

This may help you figure out what some of the "orphan" files are.

Hope this helps,

- Charles

Brian Bockelman wrote:
On Sep 24, 2008, at 1:13 PM, Michael Thomas wrote:

The latest inconsistency that I've found are pnfs ids with no associated
path (see below).  Is this what is commonly referred to as 'orphaned
files'?  Is there any way to 'reattach' the orphans, or is the only
remedy to delete them?

'orphan' files occur when the logical file is deleted, but the pool
doesn't get the message.

You can:
(a) delete them by hand from the admin interface.
(b) Use a script to automatically find & (possibly) delete them.
(c) Wait until the pool is restarted the next time; they should
automatically be cleaned up then.

We used to do (b), but lately the rate that this situation occurs at has
been declining to the point we can usually just use (c) unless we are
very space-poor.

FNAL does (b), I think; the site is used heavily enough that even at a
lower rate, it causes significant problems.

I recommend getting a feel for how much of a problem this is at your
site, then select accordingly.

    
Brian
