    The dCache developers have confirmed that my workaround of removing the
database by hand was OK -in fact, that is the intended procedure.

"the effect you are observing, is actually intended. In order not to
accidentially loosing data, we required to explicitly remove the database
file in the gdbm version and to drop the db in the postgres version.

So the way you fixed it, is the way it is meant to work."
    Feb 8, 2010 08:08 PM UTC by Ted Hesselroth 
    I found one problem when I tried it, that the "mdb remove" command does
not totally remove the database. Seems OK to use dropdb by hand before
recreating it. I will write to dcache.org to see if there is some other
problem here.

One other difference in my procedure: according to
http://www.dcache.org/archive/user-forum/1635.html it is better to
remove the postgres file before disabling the database.

I created databases by following
http://www.dcache.org/manuals/Book/config/cf-pnfs-db.shtml, and your
tags situation with the typo and "rm"

[root@camp5 data2]# grep "" $(cat ".(tags)()")
.(tag)(OSMTemplate):StoreName USAtlas
.(tag)(sGroup):IllinoisHEP
grep: .(tag)(AccessLatencey): No such file or directory
.(tag)(RetentionPolicy):
.(tag)(AccessLatency):

Then did this:

rm -f /opt/pnfsdb/pnfs/databases/data2
mdb disable data2
mdb update
mdb remove data2
mdb update
mdb show

mdb show
ID   Name         Type    Status       Path
----------------------------------------------
0    admin         r     enabled (r)   /opt/pnfsdb/pnfs/databases/admin
1    data1         r     enabled (r)   /opt/pnfsdb/pnfs/databases/data1
3    data3         r     enabled (r)   /opt/pnfsdb/pnfs/databases/data3

    
cd /pnfs/fs
cat '.(id)(usr)'
000000000000000000001080
sclient rmdirentry 1122 000000000000000000001080 data2

At this point it should be possible to follow the dCache book procedure,
but I got

mdb create data2 /opt/pnfsdb/pnfs/databases/data2
error message, try problematic call
mdcreate root 2 /opt/pnfsdb/pnfs/databases/data2
Creating database with ID 2 on file /opt/pnfsdb/pnfs/databases/data2
CREATE DATABASE command failed: ERROR:  database "data2" already exists

So I dropped that database by hand.

dropdb -U postgres data2

and followed the dCache book procedure without error after that.

mdb create data2 /opt/pnfsdb/pnfs/databases/data2
touch /opt/pnfsdb/pnfs/databases/data2
cd /pnfs/fnal.gov
mkdir '.(2)(data2)'
sclient getroot 1122 2 0000000000000000000010E0
cd data2
echo 'StoreName USAtlas' > '.(tag)(OSMTemplate)'
echo 'IllinoisHEP' > '.(tag)(sGroup)'
grep "" $(cat ".(tags)()")
.(tag)(OSMTemplate):StoreName USAtlas
.(tag)(sGroup):IllinoisHEP

Hi Neha,

the simples solution is to leave tags empty:

echo "" > '.(tag)(xxx)'

The empty tags are treated as non existing tags. But of course they are still
visible in pnfs. The other solution is to remove the directory. This will work
for
pirmaty tags only, e.g. for tags which was created in this directory not for
inherited
