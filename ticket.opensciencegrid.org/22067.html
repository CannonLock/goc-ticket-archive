Just talked to the GFAL developer. He confirms indeed that in GFAL 1, when you don't use the "-b" option, the BDII is contacted as declared in LCG_GFAL_INFOSYS.     Aug 13, 2014 02:00 PM UTC 
    The observation that no connections from Wisconsin were happening around Aug 13 12:26 UTC 2014 could have been due to a network outage affecting T2_US_Wisconsin between 12:10-12:50 UTC.

We have made a change just now that we think should avoid any further jobs of this type causing the same problem.  What may have made Wisconsin different from some other OSG sites is that we had made a custom environment setting requested by a user.  Apparently this user and some colleagues have written their own custom job scripts that rely on it.  We were setting

LCG_GFAL_INFOSYS = exp-bdii.cern.ch:2170

We no longer do this.  That returns our site to the default behavior in OSG in which the following command fails:

$ lcg-cp srm://... file://...
[GFAL][get_se_types_and_endpoints][] [BDII][g1_sd_get_se_types_and_endpoints]: Cannot connect to BDII

Instead, one needs to include additional information on the command-line:

$ lcg-cp -bD srmv2 srm://... file:...

If there is a different recommendation for the setting of this environment variable, we can try that.  However, we are not aware of anything that relies on it, other than the scripts of the group in Germany that triggered this issue.

--Dan
