    Hi Arvind,

:-), yes, the ticket got long.

Let me try to help you:

1) each entry which starts with 'dn: GlueVOInfoLocalID=...' should have
in the dn (after the SALocalID) a SEUniqueID=...  .The SALocalID is not
sufficient to identify the VOInfo globally, since it is not unique. Only
with the SE - which is globally unique- the entry can be assigned
correctly.
SE         global
/
SA    SA     local
/
VOInfo  VOInfo      local

the openldap server cannot add the entry since it cannot find the
parent.

ldapsearch -t -x -h is.grid.iu.edu:2170 -b o=grid objectClass=GlueVOInfo
dn | perl -pe 'BEGIN { $/ = "" } s/\n //g' | grep -v "GlueSEUniqueID" |
grep dn
