Hi Tim,

We had similar problems when we used the CE's alias in the host
certificate.  When we changed the certificate to use the DNS A record
for the system, authentication problems went away.  Users can still
contact and authenticate to our site using the host alias, and never
notice that the host certificate itself does not match the alias.  I
think the trick was making sure that reverse DNS maps to the same name
in the host certificate.

CE:  198.32.44.169
DNS A record:  cithep169.ultralight.org
DNS CNAME record:  cit-gatekeeper.ultralight.org
/etc/grid-security/hostcert.pem subject:
/DC=org/DC=doegrids/OU=Services/CN=cithep169.ultralight.org
Name registered with the GOC:  cit-gatekeeper.ultralight.org

--Mike

Hi Tim and Mike,

you're probably right that the error comes from the mismatch between
the real hostname (cms.cse.ucdavis.edu) and the host certificate
(cms.physics.ucdavis.edu).

    
One thing you can try is to add the line

-globus-gatekeeper-host cms.physics.ucdavis.edu

in $VDT_LOCATION/globus/etc/globus-job-manager.conf (or change the one
that's already there), and see if that helps.

It may not, though, which case it would probably be the easiest to
simply request a host certificate for cms.cse.ucdavis.edu as you suggest.

If it does help, then you should also add

export GLOBUS_HOSTNAME=cms.physics.ucdavis.edu   (or setenv for .csh)

to $VDT_LOCATION/globus/etc/globus-user-env.[c]sh, so that globus client
commands used on that gatekeeper will also report the alias as their
originating hostname.

Hope this helps,

Horst
