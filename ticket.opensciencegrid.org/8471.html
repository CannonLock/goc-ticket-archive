    The Fermi/Remedy ticket has been updated.

Work Info Type* : Resolution Communications
Date+           :
Source          :
Summary*        : Resolved as quirk of "show query" function: not a problem.
Notes           : Investigation showed that the apparent problem was the result of a quirk of SHOW QUERY:

1.  Most of the query used in the graphical reports is generic and selects more columns than are generally used by report.  A tailored, "GROUP BY" clause is added by the specific report to get the information needed.

2. Hitting, "SHOW QUERY" will produce a table of results, but the query will only contain the GROUP BY entries required to make the columns required by the specific report meaningful: therefore some columns shown on the text query result may not be meaningful (when using GROUP BY, selected quantities not either GROUPed BY or aggregated are not meaningful).

While isolating the cause of the anomalous results, it was observed that there was a period where two entries describe each of several hundred jobs due to the reprocessing by the probe of old accounting data. The reason these were not flagged as duplicates is that the reverse grid-map lookup produced a different valid DN for the user and therefore these entries were classified as being for different events. As an issue unrelated to this Ops ticket, Shreyas is investigating how best to deal with the duplication issue.
