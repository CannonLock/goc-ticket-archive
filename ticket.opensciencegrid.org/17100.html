    The MWT2_CE_UC probe ( condor:uct2-gk.mwt2.org) has been reporting
System CPU of 2147483648 seconds on several jobs each day. This has
been identified as a Condor bug of some sort but resolution has been difficult.
See https://htcondor-wiki.cs.wisc.edu/index.cgi/tktview?tn=3814 for more information.

The OSG gratia condor probe v1.13.15 has been modified to stop this from being
forwarded to Gratia based on this criteria:
It writes message to the Gratia probe log when either RemoteUserCpu or RemoteSysCpu
in the condor logs for a job exceeds 2000000000 and something called slot_ratio is > 1000.

if classad[field] < 2000000000:
return False
slot_ratio = classad[field] / (float(classad['CumulativeSlotTime']) + 1)
if slot_ratio > 1000:
return True

It writes the log message and sets the offending CPU value to zero.  This is only affects the
System CPU that Gratia gets.  User CPU and Wall Duration in Gratia is still valid and sent.
This algorithm was suggested by the Condor folks.

What we would like you to do, is upgrade your condor probe to v1.13.15 as soon as possible.

To correct this in Gratia, we are zeroing out this CPU on a daily basis in order for Gratia reporting
to look reasonable.

Once the probe is installed, you can grep the gratia condor logs for "Record has invalid" to see
how many jobs are being affected daily.  It is not that many, we hope.

Does this sound reasonable?  Please advise.

John Weigand
