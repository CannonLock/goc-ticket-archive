    Jan 22, 2014 04:08 AM UTC by Marco Mambelli 
    Hi Terrence,
I don't think you need GUMS.

Gratia's HTCondor probe is used for:
1. CEs where x509 authentication is used and GUMS or edg-mkgridmap generate the users-vi-mapfile used by the probe to map users to the correct VO
2. submit hosts without grid certificates (e.g. condor schedds flocking to a VO frontend) where the VO mapping uses a manual override

For the 2nd case your probe configuration needs to have the following 3 lines (replace VO_NAME with your VO):
<pre>
MapUnknownToGroup="1"
MapGroupToRole="1"
VOOverride="VO_NAME"
</pre>

This is documented here: https://twiki.grid.iu.edu/bin/view/Accounting/ProbeConfigGlideinWMS#Users_without_Certificates

You may be interested also in:
- This document mentioning the RPM to install a submit host to OSG-XD and other OSG front ends (I think it was already mentioned below): https://twiki.grid.iu.edu/bin/view/CampusGrids/InstallCondorFlockSubmit
- This document describing the "manual" configuration of a similar setup (flocking to an arbitrary VO frontend ): https://twiki.grid.iu.edu/bin/view/Documentation/Release3/GlideinWMSCampusGrid

The ticket is long and I'm not sure I understood correctly your installation
If you prefer we can have a brief call/IM to work on the issue, drop me an email (marcom@....)

    Hi Terrence,

My guess is the following:

1. The probe uses the file /var/lib/osg/user-vo-map to "know" to which VO map certain transactions

2. When QuarantineUnknownVORecords is set to 1 which is the default and strongly recommended. The probe code quarantines the records for which it cannot map to a VO.

So I recommend you two things, ( I guess you are using GUMS authentication),

Check here (Although the twiki is about the grid ftp probe the gums authorization config should be similar if not the same):

https://twiki.grid.iu.edu/bin/view/Documentation/Release3/GratiaTransferProbe#Using_GUMS_authorization_mode

Also in the ProbeConfig grep for UserVOMapFile and double check that it is pointing to /var/lib/osg/user-vo-map then check the gums part of the twiki to make sure the user-vo-map gets created. You can also test if setting QuarantineUnknownVORecords to 0 "fixes" temporarily the reporting.
