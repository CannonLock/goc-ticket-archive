Hello,

We have managed to mix up ownerships of files and directories in a way that is preventing us from staging persistent data and applications to pg.ihepa.ufl.edu.  Peter (CC'ed) and I are in two VOs: SBGrid and NEBioGrid.  Last year we attempted to shift all our computing to the NEBioGrid VO, however only about half the sites that supported SBGrid enabled NEBioGrid, forcing us to continue with the SBGrid VO.

This has caused endless headaches, since depending on how you map our DN and whether we have properly used voms-proxy-init to get a VO-specific proxy cert, it is possible that our jobs will run as the NEBioGrid VO and get mapped to the "nebiogrid" user.  Something like this appears to have happened at some point in the past, and now these directories:

/raid/osgpg/pg/app/sbgrid
/lustre/pg/data/sbgrid

drwxr-xr-x 13 g14669  guser    4096 Mar 19  2010 sbgrid/

cannot be written by my current DN user (see below for voms-proxy-info output).

These directories are owned by user g14669:guser, however the DN I am now using is mapped to:

uid=16205(g16205) gid=20000(guser) groups=20000(guser)

I am happy to re-try these operations as a different user if you can tell me the DN I should use, or perhaps you can map the DN below to the same UID, or perhaps UFL supports "Roles" and I can then use the "software" Role for these operations and you can key on that to map to the g16205 UID.

Because there are multiple people here with independent DNs who should be able to equivalently write to $OSG_DATA/sbgrid and $OSG_APP/sbgrid, an option that uses VOMS Roles would be the best for us, and second best would be a system where you manually map the DNs for our admins (about 3 or 4 DNs total) all to the same UID so we can all write to those shared areas.

If none of those can be done in the short term, then a simple

# sudo chown -R g16205:guser $OSG_APP/sbgrid $OSG_DATA/sbgrid

would be great.

Please advise how we can proceed to correct this, and when this has been done and we'll re-try our data and application staging to prairefire.  Until then we won't be able to run any jobs there.

Regards,

Ian
